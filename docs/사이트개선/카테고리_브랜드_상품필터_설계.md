# 카테고리·브랜드 상품 필터 설계

> 목적: 카테고리 목록·브랜드 목록에서 **카테고리 성격에 맞는 필터(facet)** 를 제공한다. 의류는 성별·사이즈·색상, 가전은 에너지등급·용량, 도서는 저자·출판사처럼 **1뎁스 카테고리마다 필터 셋이 다르다**. 이 문서는 1뎁스 12개 전부에 대해 필터 종류를 정의하고, 이를 담을 데이터 모델·URL 규약·구현 순서를 정한다.

## 0. 결론 요약

0. **출발점: 현재 고객 목록에는 가격대 필터조차 없다.** 있는 것은 카테고리·브랜드·뱃지 4종·유통뱃지 1종뿐이다(§1.3). 속성 필터를 논하기 전에 **가격대부터** 넣어야 한다.
1. **필터는 3계층으로 나눈다.** `공통(Tier 0, 전 카테고리 동일)` + `그룹(Tier 1, 여러 카테고리 공유)` + `전용(Tier 2, 카테고리 고유)`. 카테고리마다 필터를 통째로 새로 만들지 않는다.
2. **필터 정의는 코드가 아니라 DB 에 둔다.** `facet_definition` / `facet_value_definition` / `category_facet` 3개 테이블 신설. 몰 빌더 제품 성격상 "몰마다 다른 필터 셋"이 필연이므로 하드코딩하면 안 된다. 카테고리 상속 로직은 `category_option` 의 기존 구현(`services/catalog/categoryOptionService.js:53-84`)을 재사용한다.
3. **필터 값의 저장소는 이미 있다 — `product_attribute`(EAV).** 테이블과 `is_searchable` 컬럼까지 설계돼 있으나 **0행이고 참조 코드도 0건**인 빈 슬롯이다. 이 문서의 최대 과제는 필터 UI 가 아니라 **속성값 적재**이며, 적재는 전부 **관리자 화면**에서 이뤄져야 한다(§9).
4. **필터 후보는 상상이 아니라 네이버 상품정보제공고시 스키마(`naver_notice_schema`, 36종)에서 도출했다.** DB 에 적재돼 있고 카테고리 3뎁스에 `notice_type` 이 매핑돼 있어(1,524개 리프) **카테고리 → 고시타입 → 속성 후보**가 데이터로 이어진다. (단 현재 이 테이블을 읽는 코드는 없다 — D-7)
5. **브랜드 페이지의 1급 필터는 "카테고리"다.** 브랜드 하나가 최대 11개 1뎁스에 걸쳐 있어(오너클랜), 카테고리를 먼저 좁히지 않으면 속성 필터를 부여할 수 없다.
6. **구현 제약**: `productController.getList` 는 카운트를 `SELECT *` 문자열 치환으로 만들어 **JOIN 을 추가할 수 없다.** 모든 신규 필터는 `EXISTS` 서브쿼리로 붙인다(§1.3, §6.3).

#### ⚠️ 구현상 최대 제약 — `getList` 의 SQL 조립 방식

`controllers/productController.js` 는 SQL 을 **문자열 누적**으로 만들고, 카운트를 `query.replace('SELECT *', 'SELECT COUNT(*) as total')` 로 파생시킨다(`:270`). 그래서:

- **`FROM` 절에 JOIN 을 추가할 수 없다.** 추가하면 카운트 쿼리가 깨진다. 판매량·리뷰 정렬조차 상관 서브쿼리로 넣고 있다(`:22-31` 주석).
- → **새 필터는 반드시 `EXISTS` / `IN (SELECT …)` 서브쿼리로 붙여야 한다.** §6.3 이 EXISTS 형태를 택한 이유가 이것이다.

반면 `services/brand/brandService.js:383-388` 은 WHERE 배열 push + **별도 COUNT 쿼리**라 확장이 쉽다. 장기적으로 `getList` 도 같은 형태로 정리하는 것이 바람직하다.

#### 선행 리팩터링 과제

`views/user/products/list.ejs:64-99` 에 URL 빌더가 **3벌 중복**(distPrefix / sortPrefix / badgePrefix + `_buildPageUrl`) 정의돼 있다. 필터 축이 5개에서 15개로 늘면 조합 폭발 지점이 된다. 브랜드 상세(`views/user/brands/detail.ejs:14-19`)의 `URLSearchParams` 기반 단일 헬퍼로 통일하는 것이 Phase 3 의 선결 조건이다.

#### D-2 근본 원인 — 도매꾹 수량별 단가표 오파싱 (2026-07-22 규명·수정)

가격이 깨진 진짜 이유는 "누가 이상한 값을 넣었다"가 아니라 **파서 버그 + 비엄격 sql_mode** 의 조합이었다.

1. 도매꾹 `price.dome` 은 단일가(`"9900"`)뿐 아니라 **수량별 단가표**(`"1+17000|2+16800"` = 1개↑ 17,000원 / 2개↑ 16,800원)로도 온다.
2. `normalize.js` 의 `toNum()` 이 숫자가 아닌 문자를 전부 지워 `"117000216800"` → **117,000,216,800** 을 만들었다.
3. **`sql_mode` 가 빈 값(비엄격)** 이라 MySQL 이 에러 없이 조용히 잘랐다.
   `117,000,216,800` → `supply_price decimal(12,2)` → **9,999,999,999.99** → `products.price INT` → **2,147,483,647**
4. 그 결과 21억짜리 상품 2건이 `ON`·`PUBLIC` 으로 고객 화면에 노출돼 있었다.

|---|---|---|
| `products.price` · `purchase_price` · `original_price` · `product_sku.price` | `INT signed` | **2,147,483,647원 (약 21억 4,748만원)** |
| `supplier_product.supply_price` | `decimal(12,2)` | 9,999,999,999.99원 (약 100억) |

실사용 최고가는 **24,539,130원**(24K 포나인 골드바 100g)이다. 천만원 초과 상품은 5건 있으나 **전부 금거래소 골드바로 정상 상품**이다(예상과 달리 존재한다). INT 한계 대비 87배 여유가 있어 **타입 변경은 불필요**하다.

**조치**

- `parsePriceField()` 신설 — 수량별 단가표를 파싱해 최소 수량 구간 단가를 공급가로 쓴다 (`services/sourcing/supplier/domeggook/normalize.js`)
- `MAX_PRICE = 1억` 가드 — 비정상 가격이면 발행을 막고 사용자에게 원본 확인을 안내한다 (`services/sourcing/publishService.js`)
- 오염된 2건은 `raw_json` 에서 **결정적으로 재계산**해 복원 — 11074 → 17,500원 / 11075 → 17,000원 (Phase 0 마이그레이션). 매직 넘버를 박지 않아 재실행해도 같은 결과가 나온다.

> ⚠️ **`sql_mode` 가 비어 있다는 사실이 이 사고의 본질이다.** 범위를 넘는 값이 예외 없이 잘려 들어가므로, 앞으로도 **금액·수량은 DB 에 넣기 전에 애플리케이션이 상한을 검사**해야 한다.

## 2. 필터 계층 구조

```
Tier 0  공통 필터      — 12개 1뎁스 전부 동일. 카테고리 무관.
Tier 1  그룹 필터      — 성격이 같은 카테고리군이 공유. (색상·사이즈·소재·성별 등)
Tier 2  카테고리 전용   — 그 카테고리에서만 의미 있는 필터. (에너지등급·저자·SPF·사용연령 등)
```

부여 규칙:
- 1뎁스 진입 시 → **Tier 0 + 그 1뎁스에 정의된 Tier 1/2 중 `is_primary` 인 것**만 노출(최대 6~7개).
- 2뎁스 이하로 좁히면 → 그 카테고리에 상속·추가된 Tier 1/2 를 **전부** 노출.
- 값이 0건인 필터·값은 **자동 숨김**(파셋 카운트 0 은 표시하지 않음).

## 4. 1뎁스 카테고리별 필터 정의

표기: **P** = 1뎁스 진입 시 바로 노출(primary), 그 외는 2뎁스 이하에서 노출. `출처` 는 값을 어디서 얻는지.

> ⚠️ **구현 시 정리한 점 — "품목/종류" 축은 facet 으로 만들지 않는다.**
> 아래 표의 `ITEM_TYPE` · `COSMETIC_TYPE` · `DEVICE_TYPE` · `FURNITURE_TYPE` · `BABY_ITEM` · `FOOD_TYPE` · `SPORT_TYPE` · `BOOK_GENRE` · `SERVICE_TYPE` · `CLOTH_TYPE` 은 전부 **2뎁스(또는 3뎁스) 카테고리 파생**이라 Tier 0 `CATEGORY` 필터와 값이 같다. 같은 것을 두 벌 관리하면 반드시 어긋나므로 **`CATEGORY` 하나로 통합**했다. 표에는 설계 의도를 남겨 두되, 실제 적재된 facet 에는 없다. → 필터 수 약 90 → **100개(Tier 0 10 · Tier 1 15 · Tier 2 75)**, 중복 축 10개 제거.

## 6. 데이터 모델 (신설)

### 6.1 테이블

```sql
-- 필터(facet) 정의: 전 몰 공통 카탈로그
CREATE TABLE facet_definition (
  id            INT AUTO_INCREMENT PRIMARY KEY,
  facet_code    VARCHAR(50)  NOT NULL,           -- COLOR, SIZE_SHOE_MM, ENERGY_GRADE ...
  facet_name    VARCHAR(50)  NOT NULL,           -- 화면 표시명 (색상, 발 길이 ...)
  tier          TINYINT      NOT NULL DEFAULT 2, -- 0=공통 1=그룹 2=전용
  ui_type       ENUM('CHECKBOX','CHIP','COLOR_SWATCH','SIZE_GRID','RANGE','TOGGLE','SELECT')
                             NOT NULL DEFAULT 'CHECKBOX',
  value_source  ENUM('DEFINITION','ATTRIBUTE','OPTION','COLUMN','CATEGORY','DERIVED')
                             NOT NULL DEFAULT 'ATTRIBUTE',
  source_key    VARCHAR(100) NULL,               -- ATTRIBUTE: product_attribute.attr_name / COLUMN: 컬럼명
  data_type     ENUM('STRING','NUMBER','BOOL','RANGE') NOT NULL DEFAULT 'STRING',
  unit          VARCHAR(20)  NULL,               -- mm, ml, g, cm, 인치 ...
  is_multi      TINYINT(1)   NOT NULL DEFAULT 1, -- 다중 선택 허용
  meta_json     JSON         NULL,               -- 구간 프리셋, 별칭 사전 등
  is_active     TINYINT(1)   NOT NULL DEFAULT 1,
  is_primary_default TINYINT(1) NOT NULL DEFAULT 0, -- 카테고리 매핑이 없을 때의 기본 펼침
  display_order INT          NOT NULL DEFAULT 0,
  UNIQUE KEY uk_facet_code (facet_code)
);

-- 필터 값 정의 (닫힌 집합인 facet 만)
CREATE TABLE facet_value_definition (
  id            INT AUTO_INCREMENT PRIMARY KEY,
  facet_id      INT          NOT NULL,
  value_code    VARCHAR(50)  NOT NULL,
  display_name  VARCHAR(100) NOT NULL,
  meta_json     JSON         NULL,               -- {"hex":"#111","aliases":["차콜","먹색"],"min":0,"max":30000}
  display_order INT          NOT NULL DEFAULT 0,
  UNIQUE KEY uk_facet_value (facet_id, value_code),
  FOREIGN KEY (facet_id) REFERENCES facet_definition(id) ON DELETE CASCADE
);

-- 카테고리 ↔ 필터 부여 (계층 상속)
CREATE TABLE category_facet (
  id                  INT AUTO_INCREMENT PRIMARY KEY,
  category_id         INT        NOT NULL,       -- 1·2·3뎁스 모두 가능
  facet_id            INT        NOT NULL,
  is_primary          TINYINT(1) NOT NULL DEFAULT 0,  -- 접힘 없이 바로 노출
  is_visible          TINYINT(1) NOT NULL DEFAULT 1,  -- 상위에서 상속받은 것을 여기서 끄기
  inherit_to_children TINYINT(1) NOT NULL DEFAULT 1,
  display_order       INT        NOT NULL DEFAULT 0,
  meta_json           JSON       NULL,           -- 이 카테고리만의 오버라이드(가격 구간 프리셋 등)
  UNIQUE KEY uk_cat_facet (category_id, facet_id),
  KEY idx_cat (category_id)
);
```

> `is_primary_default` 가 없으면 Tier 0 10개가 전부 펼쳐져 1뎁스에서 필터가 12~15개씩 열린다(실측). `CATEGORY`·`PRICE`·`BRAND`·`DISCOUNT` 넷만 기본 펼침으로 두면 카테고리별 4~10개로 떨어진다.
>
> `category_facet.meta_json` 은 §3.3 의 **카테고리별 가격 구간**을 담는다. 같은 `PRICE` 필터인데 식품은 `1만/3만/5만/10만`, 디지털·가전은 `10만/50만/100만/200만` 이어야 하기 때문이다.

**값 저장은 신설하지 않는다.** 이미 있는 `product_attribute` 를 쓴다.

```sql
-- 기존 테이블. 현재 0행. 필터 조회용 인덱스만 추가한다.
ALTER TABLE product_attribute
  ADD INDEX idx_attr_filter (attr_name, attr_value, product_id),
  ADD INDEX idx_attr_product (product_id, attr_name);
```

규약: `product_attribute.attr_name` = `facet_definition.facet_code`, `attr_value` = `facet_value_definition.value_code`(닫힌 집합) 또는 숫자 문자열(RANGE 타입). `is_searchable=1` 인 행만 필터 대상.

### 6.2 필터 해석 순서 (런타임)

```text
1. 현재 카테고리의 조상 체인을 구한다.               (1뎁스 → 2뎁스 → 3뎁스)
2. 체인 위쪽부터 category_facet 을 모아 머지한다.
   - inherit_to_children=1 이면 하위로 전파
   - 하위에 같은 facet 이 있으면 하위 설정이 이김 (is_visible=0 이면 숨김)
3. Tier 0 facet 을 항상 앞에 붙인다.
   - 단, 카테고리에 is_visible=0 오버라이드가 있으면 제외
     (예: 여가/생활편의 → DELIVERY·STOCK 제외 / 도서 → COLOR·SIZE 제외)
4. 각 facet 의 값 목록 + 파셋 카운트를 구한다. 카운트 0 인 값은 숨김.
5. 1뎁스에서는 is_primary=1 만, 2뎁스 이하에서는 전부 노출.
```

### 6.3 조회 쿼리 형태

facet 간 **AND**, facet 내 값 간 **OR**. `EXISTS` 서브쿼리를 facet 수만큼 붙인다.

```sql
SELECT p.id, p.name, p.price, p.main_image
FROM products p
WHERE p.mall_id = ?
  AND p.status = 'ON' AND p.visibility = 'PUBLIC'
  AND p.category_id IN (:descendantCategoryIds)
  AND p.price BETWEEN :priceMin AND :priceMax
  -- facet 1: 색상 = 블랙 OR 화이트
  AND EXISTS (SELECT 1 FROM product_attribute a
              WHERE a.product_id = p.id AND a.attr_name = 'COLOR'
                AND a.attr_value IN ('BLACK','WHITE') AND a.is_searchable = 1)
  -- facet 2: 소재 = 면
  AND EXISTS (SELECT 1 FROM product_attribute a
              WHERE a.product_id = p.id AND a.attr_name = 'MATERIAL'
                AND a.attr_value IN ('COTTON') AND a.is_searchable = 1)
ORDER BY ... LIMIT ? OFFSET ?;
```

파셋 카운트는 **자기 자신을 제외한 나머지 조건**을 적용해 facet 별로 따로 집계한다(선택한 값 때문에 다른 값이 0으로 보이는 문제 방지).

성능: 현재 9,714행 규모라 위 형태로 충분하다. 5만행을 넘거나 facet 이 6개 이상 동시에 걸리면 `product_facet_flat`(product_id, facet_code, value_code 비정규화 + 커버링 인덱스) 캐시 테이블을 추가한다.

> ⚠️ `controllers/productController.js` 의 카운트 쿼리가 `SELECT *` 문자열 치환으로 만들어지므로(§1.3) **JOIN 을 쓰면 안 된다.** 위 `EXISTS` 형태는 그 제약을 만족한다.

### 6.4 네이버 상품속성 API 와의 정합

`docs/사이트개선/네이버_마스터데이터_수신_설계.md` §3·§4 에 **카테고리별 속성 스키마 수신**이 이미 설계돼 있다(테이블·코드는 미구현).

```
GET /v1/product-attributes/attributes?categoryId=       → attributeSeq, attributeName,
                                                           attributeClassificationType(SINGLE|MULTI_SELECT),
                                                           attributeType, unitUsable, maxMatchingCount
GET /v1/product-attributes/attribute-values?categoryId=  → attributeValueSeq, minAttributeValue(라벨), exposureOrder
```

네이버 스펙이 본 설계의 `facet_definition` 과 거의 1:1 로 대응한다:

| 네이버 필드 | 본 설계 필드 |
|---|---|
| `attributeName` | `facet_definition.facet_name` |
| `attributeClassificationType` (SINGLE / MULTI_SELECT) | `facet_definition.is_multi` |
| `unitUsable` | `facet_definition.unit` |
| `exposureOrder` | `facet_value_definition.display_order` |
| `minAttributeValue` | `facet_value_definition.display_name` |

따라서 네이버 속성 수신이 구현되면 **`facet_definition`/`facet_value_definition` 을 카테고리 단위로 자동 생성**할 수 있다. 다만 **키 축이 다르다** — 네이버는 `naver_category_id`, 우리는 `category_id`. 브리지는 `categories.naver_category_id`(2·3뎁스 전건 보유)를 경유하되, D-1 collation 때문에 `COLLATE` 명시가 필요하다.

이 경로가 열리기 전까지는 §4 의 정의를 마이그레이션 SQL 로 적재한다(§10 Phase 2).

## 7. URL 규약

```
/categories/:slug?color=BLACK,WHITE&size_alpha=M,L&price=30000-50000&brand=585,591&sort=popular&page=2
/brands/:id?cat=3309&color=BLACK&sort=new
```

- 쿼리 키 = `facet_code` **소문자**. 값은 `value_code` 콤마 구분.
- 구간형은 `min-max`(`price=30000-50000`, 열린 구간 `price=100000-`).
- 예약어(`sort`, `page`, `q`, `view`, `mall`)와 충돌하는 `facet_code` 는 금지 — `facet_definition` 저장 시 검증한다.
- 필터 상태가 URL 에 온전히 담기므로 공유·뒤로가기·SEO 가 자연스럽게 동작한다.
- 필터가 걸린 목록 페이지는 `noindex, follow` 로 색인 폭증을 막는다(현재 전역 `noindex` 상태라 당장은 무관).

## 8. UI 규칙

- **PC**: 좌측 사이드바 고정. `is_primary` facet 은 펼침, 나머지는 접힘. 값 8개 초과 시 "더보기".

## 9. 속성값 적재 전략 (최대 난관)

`product_attribute` 가 0행이고 코드 참조도 0건이므로, 필터 UI 를 만들어도 값이 없으면 아무것도 걸리지 않는다.

> **전제 (CLAUDE.md「사용자 전제」)**: 최종 사용자는 터미널을 쓸 수 없다. **적재 경로는 전부 관리자 화면의 버튼으로 제공해야 하며, 일회성 스크립트로 이 인스턴스에만 데이터를 채워 두면 안 된다.** 납품되는 다른 몰에서는 그 상태가 존재하지 않기 때문이다. 아래 표의 "실행 위치"가 전부 관리자 화면인 이유다.

| # | 경로 | 대상 | 실행 위치 | 상태 |
|---|---|---|---|---|
| ~~A~~ | ~~고시 정보 입력값 재활용~~ | — | — | ❌ **현재 불가** — 아래 참고 |
| B | **관리자 수기 입력** | 신규·핵심 상품 | 상품 등록·수정 화면 [상품 속성] | ✅ 구현 (Phase 6) |
| C | **옵션값·상품명·공급사 원본 추출** | 기존 상품 전량 | 관리자 [상품 속성 추출·검수] | ✅ 구현 (Phase 8) |
| D | **상품명·상세 AI 추출** | 사전으로 못 잡는 나머지 | 관리자 버튼(예정) | ⬜ C 의 채움률을 보고 판단 |

> ⚠️ **경로 A 는 전제가 틀렸다(2026-07-22 확인).** "고시 입력값을 재활용한다"는 계획은 **상품별 고시 값이 어딘가에 저장돼 있다**는 가정 위에 있었는데, 실제로는 그런 저장소가 없다.
> 고시 값은 **몰 단위 기본값**(`naver_publish_profile.notice_defaults_json`, `mall_notice_default`)과 채널 전송 시점의 override 로만 다뤄지고(`services/sourcing/channel/naverMapper.js:113-127`), `products` 에도 별도 테이블에도 상품별 고시 입력값 컬럼이 없다.
> 따라서 A 를 살리려면 **상품별 고시 입력 화면과 저장소를 먼저 만들어야** 한다. 그건 필터가 아니라 상품 등록 쪽 과제다. §12 부록의 고시타입 ↔ 필터 매핑표는 **어떤 속성을 필터로 삼을지 고르는 근거**로는 여전히 유효하다(그래서 §4 가 그 위에 서 있다).

권장 순서: **B 로 신규분을 막고, C 로 기존 9,714건을 소급**한다.

D·E 는 자동 추출이라 오류가 섞인다. 반드시 다음 2단 구조를 지킨다.

```
자동 추출 → product_attribute(is_searchable = 0)   ← 필터에 안 걸림
                    ↓ 관리자 검수 화면에서 승인
             product_attribute(is_searchable = 1)   ← 필터에 걸림
```

### 2026-07-22 (2) — Phase 5~9 완료

**Phase 5 — 관리자 필터 부여** (`/admin/products/facets`)
- `facetAdminService.getFacetMatrix()` — 전체 정의 + 자기 지정 + 상속을 한 화면에.
- 필터마다 3상태(**기본/상속 · 사용 · 숨김**) + 즉시노출 + 하위상속. `is_visible=0` 이 "상속받았지만 여기서는 끈다".
- 속성 필터는 **값 보유 현황**(몇 개 상품에 값이 있는지)을 같이 보여 준다 — 값이 없으면 고객 화면에 안 뜨기 때문이다.
- `meta_json`(가격 구간 프리셋)은 저장 시 보존한다. 화면에서 안 다루는 값을 재구성으로 날리면 안 된다.

**Phase 6 — 상품 속성 입력** (상품 등록·수정 화면)
- 카테고리를 고르면 `GET /admin/products/attribute-form` 으로 그 카테고리의 입력 항목을 받아 그린다.
- 제출은 평범한 폼 필드(`name="attr_COLOR"`). **JSON textarea 를 만들지 않는다**(CLAUDE.md).
- 정의에 없는 값은 저장하지 않는다. 관리자 입력이므로 `is_searchable=1`.

**Phase 7 — 속성 파셋 카운트**
- `getAttributeCounts()` 한 방으로 값별 건수를 낸다. **속성 조건은 전부 빼고** 센다 — 자기 자신을 넣으면 같은 그룹의 다른 값이 전부 0 으로 접힌다.
- 카운트를 아는데 0 인 값은 렌더하지 않는다.
- `product_attribute` 가 비어 있으면 이 쿼리 자체를 건너뛴다(지금이 그렇다).

**Phase 8 — 자동 추출·검수** (`/admin/products/facet-extract`)
- `facetExtractor` 가 **상품명 + 옵션값 + 공급사 원본**(manufacturer·model_name·size_text·country)에서 사전으로 값을 뽑는다. 사전은 `facet_value_definition.meta_json.aliases`(§3.1 별칭)를 쓰고 **긴 토큰부터** 맞춘다("로즈골드"가 "골드"보다 먼저).
- 뽑은 값은 전부 **`is_searchable=0`(검수 대기)**. 승인해야 고객 필터에 걸린다. 이 2단 구조가 이 기능의 핵심이다.
- `uk_product_attr_value (product_id, attr_name, attr_value)` 유니크로 재실행 중복을 막는다.
- 실측: 패션의류 1,116건 중 **912건에서 1,853개** 추출(GENDER 456·MATERIAL 364·SIZE_WAIST 237·SLEEVE 227·COLOR 212…). 재실행 적재 0개.

**Phase 9 — 브랜드관 필터**
- `buildPredicates` 에 `alias` 옵션 추가 — 브랜드 목록 쿼리는 `products` 를 `p` 로 별칭하므로 그대로 쓰면 `EXISTS ... products.id` 가 깨진다.
- 카테고리를 고르기 전에는 공통 필터만, 고르면 그 카테고리의 속성 필터가 붙는다(§5.1).
- 브랜드 필터는 자기 페이지에서 제외(파셜이 `currentBrand` 로 판단).
- 필터 CSS 를 `_facet_styles.ejs` 로 분리해 목록·브랜드관이 함께 쓴다.
