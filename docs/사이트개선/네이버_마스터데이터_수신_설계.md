# 네이버 마스터 데이터 수신 설계 (몰 빌더 상품 구성 정보)

> 목적: 상품 등록 시 쓸 **상품 구성 마스터 정보**(브랜드·제조사·원산지·속성·카테고리별 표준옵션·사이즈타입·상품정보제공고시 등)를 네이버 커머스 API에서 **수신**해, 우리 사이트 상품 등록에 활용한다. 스마트스토어 외 타 오픈마켓 연동까지 대비한 마스터 테이블 구조.
>
> 짝 문서: 카테고리 계층·매핑 원칙은 [`쇼핑몰_상품_옵션_세트_묶음_관리구조_정리.md`](./쇼핑몰_상품_옵션_세트_묶음_관리구조_정리.md) §32.

---

## 1. 핵심 원칙 (§32 연장)

1. **우리 것은 우리 것, 네이버는 별도.** 네이버 마스터는 **`naver_*` 참조 테이블**(raw 수신, 전역·mall 무관 — `naver_category` 와 동일)로 저장한다. 우리 정본은 그대로 둔다:
   - 우리 브랜드 = `categories(type='BRAND')`
   - 우리 옵션 사전 = `option_definition`/`option_value_definition`
   - 우리 상품 속성 = `product_attribute`, 카테고리 추천옵션 = `category_option`
2. **네이버 데이터는 등록 화면의 검색·프리필 소스.** 이미 만든 **카테고리 위젯 패턴**(네이버 리프 검색 → 우리 값 프리필 + `products.naver_category_id` 매핑)을 브랜드·제조사·속성에 그대로 확장한다.
3. **상품은 네이버 매핑 id 를 보유**(전송용). `products.naver_category_id`·`naver_brand_id` 이미 존재. 필요 시 `naver_manufacturer_id` 등 추가.
4. **키 축 주의(중요).** 네이버 "카테고리별 표준옵션"·"속성"은 **`naver_category_id` 축**으로 키잉된다. 우리 `category_id` 축의 `category_option` 과 **다른 참조 테이블**이다. 두 홉 브리지: `products.naver_category_id` → 네이버 표준옵션/속성 → 우리 `product_option`/`product_attribute` 로 프리필.

## 2. 현황

| 항목 | 상태 |
|---|---|
| `naver_category` | **5,815건 수신 완료**(`/v1/categories`, 24h 스케줄) |

## 3. 마스터 데이터 유형 ↔ 엔드포인트

> **프로브로 확정 완료**(2026-07-17, 개발서버). 실제 존재 API는 아래 4개뿐(제조사·원산지·사이즈·고시 등은 **별도 API 없음** — 사용자 확인).

| 마스터 | 확정 엔드포인트 | 응답 구조 |
|---|---|---|
| 카테고리 | `GET /v1/categories` | 전체 트리 배열(현행 벌크) |
| **브랜드** | `GET /v1/product-brands?name={검색어}` | **배열** `[{id, name}]` |
| **모델(카탈로그)** | `GET /v1/product-models?name={검색어}` | **페이징** `{contents:[{id, name, categoryId, wholeCategoryName, brandCode, brandName, manufacturerCode, manufacturerName}], page, size, totalElements, totalPages, first, last}` |
| **카테고리별 속성(이름)** | `GET /v1/product-attributes/attributes?categoryId={네이버카테고리}` | **배열** `[{attributeSeq, attributeName, attributeClassificationType(SINGLE/MULTI_SELECT…), attributeType, unitUsable, attributeValueMaxMatchingCount}]` |
| **카테고리별 속성값** | `GET /v1/product-attributes/attribute-values?categoryId={네이버카테고리}` | **배열** `[{attributeSeq, attributeValueSeq, minAttributeValue(값 라벨), exposureOrder}]` |

**확정 포인트**
- 검색 파라미터는 **`name`**(brandName/modelName 은 400). 브랜드=단순 배열, 모델=페이징(page/size).
- 속성은 두 콜: **`/product-attributes/attributes`**(속성명 메타) + **`/product-attributes/attribute-values`**(값). **`attributeSeq` 로 조인**. (사용자 문서의 `/v1/product-attributes` 단일 경로는 404 — 실제는 하위 경로 2개.)
- 속성은 **`categoryId`(네이버 카테고리) 종속** — 상품의 네이버 카테고리로 on-demand 조회. (예: 미들부츠 50004190 → 부츠종류·소재·힐높이·기장 등 9속성)
- 제조사명/브랜드명은 **모델 응답에 이미 포함**(brandCode/brandName/manufacturerCode/manufacturerName) — 별도 제조사 API 불필요.

## 4. 테이블 설계 (naver_* 참조)

> 방침: `raw_json` 보존(필드 편차 흡수). 전역(mall_id 없음). 성격에 따라 **① 벌크 mirror**(카테고리), **② 검색 캐시**(브랜드·모델·제조사), **③ 카테고리 종속 캐시**(속성)로 나뉜다.

**② 검색형(브랜드·모델)** — 전량 덤프 없음 → 벌크 테이블 X. **선택분만 캐시**(등록 시 검색→선택 항목 upsert). 확정 응답 필드에 맞춤:
```sql
naver_brand(naver_brand_id PK,        -- = 응답 id
            name, raw_json, fetched_at)
naver_model(naver_model_id PK,        -- = 응답 id (bigint)
            name, naver_category_id, whole_category_name,
            brand_code, brand_name, manufacturer_code, manufacturer_name,
            raw_json, fetched_at)      -- 모델 응답에 브랜드·제조사·카테고리 포함
```

**③ 카테고리 종속(속성)** — `categoryId`(네이버 카테고리) 축. 등록 시 on-demand 조회 후 캐시. **두 콜을 `attribute_seq` 로 조인**:
```sql
naver_category_attribute(id PK, naver_category_id, attribute_seq,
    attribute_name, classification_type,   -- SINGLE_SELECT / MULTI_SELECT ...
    attribute_type, unit_usable, max_matching_count,
    raw_json, fetched_at,
    UNIQUE(naver_category_id, attribute_seq))          -- /attributes 응답
naver_attribute_value(id PK, naver_category_id, attribute_seq,
    attribute_value_seq, value_label,                  -- minAttributeValue
    exposure_order, raw_json,
    UNIQUE(naver_category_id, attribute_value_seq),
    KEY(naver_category_id, attribute_seq))             -- /attribute-values 응답
```

> 제조사·원산지·사이즈·상품정보제공고시는 **별도 API 없음**(확인) → 마스터 테이블 미생성. 제조사명/브랜드명은 필요 시 모델 응답에서 취득.
> `raw_json` 보존으로 스키마 편차 흡수.

## 5. 정합 — naver_* ↔ 우리 테이블

| 네이버 마스터 | 우리 정본 | 브리지 |
|---|---|---|
| naver_brand | `categories(type=BRAND)` | 등록화면 검색→우리 브랜드 프리필/생성 + `products.naver_brand_id` 매핑 |
| naver_category_attribute + naver_attribute_value | `product_attribute` | `products.naver_category_id`→속성/값 조회→선택분을 우리 속성으로 저장 |

**절대 병합 금지**: 네이버 속성을 우리 `product_attribute` 정본에 "참조 캐시"와 섞지 않는다. 캐시(`naver_*`)는 별도, 저장은 우리 `product_attribute` 로만(축이 다름 — §1-4).

## 6. 수집 파이프라인

성격에 따라 두 방식:

**A. 검색형(브랜드·모델·제조사·속성)** — 스케줄 sync 없음. `naverClient` 에 `searchBrands(q)`·`searchModels(q)`·`getCategoryAttributes(naverCategoryId)` 등 추가. 관리자에 **검색 프록시 라우트**(`/admin/sourcing/naver/brands?q=`)를 두고, **개발서버에서 네이버 실시간 검색** → 결과 반환. 등록화면 위젯(카테고리 위젯 패턴)이 이걸 호출. **선택한 항목만** naver_brand/manufacturer/model 에 캐시 upsert.
> IP 화이트리스트 때문에 이 프록시는 **개발서버에서만** 실제 응답을 준다. 로컬은 403.

**B. 벌크형(카테고리, 그리고 원산지·사이즈·고시가 전량 조회로 확정되면)** — `naverTaxonomySync` 를 **제네릭 `syncResource(resource, fetcher, mapper, upserter)`** 로 일반화. `naver_taxonomy_sync_log.resource`·스케줄 재사용.

- 관리자에 **마스터 브라우저/검색 콘솔** — `/admin/sourcing/` 하위(캐시 조회 + 검색 프록시 테스트).

## 7. 선결 — 개발서버 read-only 프로브

로컬은 IP 차단이라, **개발서버에서** 각 엔드포인트를 GET 만 호출해 실제 payload(필드명·페이징·카테고리 종속 여부)를 찍는다. 이걸로 §3 ❓·§4 컬럼을 확정한 뒤 sync 를 구현한다. (쓰기 없음)

## 8. 단계별 계획

| Phase | 범위 | 상태 |
|---|---|---|
| **0 프로브** | 개발서버에서 4개 엔드포인트 실응답 확정(필드·`name` 파라미터·페이징·속성 2콜) | ✅ 완료 |
| **1 브랜드 검색 위젯** | `naverClient.searchBrands(name)` + 관리자 프록시 라우트 + 등록화면 브랜드 검색 위젯(카테고리 위젯 복제) + 선택 캐시(`naver_brand`) + `products.naver_brand_id` 저장 | ⬜ 다음 |
| **2 카테고리별 속성 프리필** | `getCategoryAttributes`+`getAttributeValues`(attributeSeq 조인) → 등록화면에서 상품의 네이버 카테고리로 속성·값 노출 → 선택분 우리 `product_attribute` 로 저장 + `naver_category_attribute`/`naver_attribute_value` 캐시 | ⬜ |
| **3 모델(선택)** | 필요 시 모델 검색 위젯(브랜드/카테고리 자동 채움) | 선택 |

> 제조사·원산지·사이즈·상품정보제공고시는 **별도 API 부재로 범위 제외**.

> 원칙(§32-4 재확인): "우리 카테고리→네이버 자동선택 안내"(상품 전송 시)와 이 마스터 수신은 상호 보완. 전송 기능은 별도.

## 9. 운영 규칙 (필수)

- 공용 개발 DB. **정리 DELETE 는 실제 `insertId` 로만**(문자열 파싱 금지), `mall_id` + `[TEST]` 마커 가드.
- 네이버 호출은 **개발서버(허용 IP)** 에서만. 로컬은 프로브도 403.
