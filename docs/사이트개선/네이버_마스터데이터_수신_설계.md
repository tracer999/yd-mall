# 네이버 마스터 데이터 수신 설계 (몰 빌더 상품 구성 정보)

> 목적: 상품 등록 시 쓸 **상품 구성 마스터 정보**(브랜드·제조사·원산지·속성·카테고리별 표준옵션·사이즈타입·상품정보제공고시 등)를 네이버 커머스 API에서 **수신**해, 우리 사이트 상품 등록에 활용한다. 스마트스토어 외 타 오픈마켓 연동까지 대비한 마스터 테이블 구조.
>
> 짝 문서: 카테고리 계층·매핑 원칙은 [`쇼핑몰_상품_옵션_세트_묶음_관리구조_정리.md`](./쇼핑몰_상품_옵션_세트_묶음_관리구조_정리.md) §32.

---

## 1. 핵심 원칙 (§32 연장)

1. **우리 것은 우리 것, 네이버는 별도.** 네이버 마스터는 **`naver_*` 참조 테이블**(raw 수신, 전역·mall 무관 — `naver_category` 와 동일)로 저장한다. 우리 정본은 그대로 둔다:
   - 우리 브랜드 = `categories(type='BRAND')`
   - 우리 옵션 사전 = `option_definition`/`option_value_definition`
   - 우리 상품 속성 = `product_attribute`, 카테고리 추천옵션 = `category_option`
2. **네이버 데이터는 등록 화면의 검색·프리필 소스.** 이미 만든 **카테고리 위젯 패턴**(네이버 리프 검색 → 우리 값 프리필 + `products.naver_category_id` 매핑)을 브랜드·제조사·속성에 그대로 확장한다.
3. **상품은 네이버 매핑 id 를 보유**(전송용). `products.naver_category_id`·`naver_brand_id` 이미 존재. 필요 시 `naver_manufacturer_id` 등 추가.
4. **키 축 주의(중요).** 네이버 "카테고리별 표준옵션"·"속성"은 **`naver_category_id` 축**으로 키잉된다. 우리 `category_id` 축의 `category_option` 과 **다른 참조 테이블**이다. 두 홉 브리지: `products.naver_category_id` → 네이버 표준옵션/속성 → 우리 `product_option`/`product_attribute` 로 프리필.

## 3. 마스터 데이터 유형 ↔ 엔드포인트

| 마스터 | 엔드포인트 | 확정도 | 성격 |
|---|---|---|---|
| 카테고리 | `GET /v1/categories` | ✅ 사용 중 | **전체 트리 일괄** → 벌크 수신(현행) |
| 브랜드 | `GET /v1/product-brands` | ✅ 확정 | **브랜드명 검색형** → 벌크 불가, 검색 프록시 |
| 모델(카탈로그) | `GET /v1/product-models`·`/{id}` | ✅ 확정 | **모델명 검색형** → 검색 프록시 |
| 카테고리별 속성 | `GET /v1/product-attributes` | ✅ 확정 | **카테고리에 입력 가능한 속성**(naver_category_id 축) |
| 카테고리별 속성값 | `GET /v1/product-attribute-values`(추정) | 🔶 경로 확인필요 | 속성별 값 목록 |
| 제조사 | `GET /v1/product-manufacturers` | ✅ 확정(#527) | 검색형 추정 |
| 원산지 | (미확정) | ❓ | 소수 enum 성 |
| 사이즈 타입 | (미확정) | ❓ | 소수 enum 성 |
| 상품정보제공고시 | (미확정) | ❓ | **카테고리별 필수 필드셋 — 무거움** |

> **결정적 구분**: 카테고리만 전체 일괄이고, **브랜드·모델·제조사는 검색형**이다(검색어 없이 전량 덤프 불가). → 이들은 **벌크 sync 테이블이 아니라 "실시간 검색 프록시 + 선택분 캐시"** 로 설계한다(§4·§6). **속성은 카테고리(naver_category_id) 종속**이라 상품 등록 시 그 상품의 네이버 카테고리로 on-demand 조회한다.
> ❓/🔶 항목의 정확한 경로·파라미터·응답 필드는 **개발서버 read-only 프로브**(§7)로 확정한다.

## 4. 테이블 설계 (naver_* 참조)

> 방침: `raw_json` 보존(필드 편차 흡수). 전역(mall_id 없음). 성격에 따라 **① 벌크 mirror**(카테고리), **② 검색 캐시**(브랜드·모델·제조사), **③ 카테고리 종속 캐시**(속성)로 나뉜다.

**② 검색형(브랜드·모델·제조사)** — 전량 덤프가 없으므로 벌크 테이블을 만들지 않는다. **선택분만 캐시**한다(등록 시 검색→선택한 항목만 upsert). 표시명 캐시가 있어야 목록·재조회가 빠르다.
```sql
naver_brand(naver_brand_id PK, name, name_en, raw_json, fetched_at)          -- 기존 골격 = 검색 선택 캐시로 재정의
naver_manufacturer(naver_manufacturer_id PK, name, name_en, raw_json, fetched_at)
naver_model(naver_model_id PK, name, naver_category_id?, brand_name?, raw_json, fetched_at)
```

**③ 카테고리 종속(속성)** — `naver_category_id` 축. 상품 등록 시 그 상품의 네이버 카테고리로 on-demand 조회 후 캐시.
```sql
naver_category_attribute(id PK, naver_category_id, naver_attribute_id, name, input_type, unit?, required?, raw_json, fetched_at, KEY(naver_category_id))
naver_attribute_value(id PK, naver_attribute_id, naver_value_id, name, raw_json, KEY(naver_attribute_id))  -- 속성값 조회 결과
```

**소수 enum·미확정(원산지·사이즈·고시)** — 프로브로 성격 확정 후. 상품정보제공고시는 카테고리별 필수 필드셋이라 별도(Phase 4):
```sql
naver_origin_area(naver_origin_id PK, name, parent_id?, raw_json, fetched_at)        -- 성격 미확정
naver_notice_type(naver_notice_type_id PK, name, raw_json, fetched_at)
naver_notice_field(id PK, naver_notice_type_id, field_name, required, raw_json, KEY(naver_notice_type_id))
```

> ⚠️ 컬럼 세부는 **프로브 payload 확정 후** 맞춘다. `raw_json` 이 있어 초기 스키마가 어긋나도 재적재로 복구.

## 5. 정합 — naver_* ↔ 우리 테이블

| 네이버 마스터 | 우리 정본 | 브리지 |
|---|---|---|
| naver_brand | `categories(type=BRAND)` | 등록화면 검색→우리 브랜드 프리필/생성 + `products.naver_brand_id` 매핑 |
| naver_manufacturer | (우리 `provider`/신설) | `products.naver_manufacturer_id`(신설) 매핑 |
| naver_category_option | `product_option`(옵션상품) | `products.naver_category_id`→표준옵션→우리 옵션 프리필 |
| naver_category_attribute | `product_attribute` | 〃 → 우리 속성 프리필 |
| naver_origin_area | `product_attribute`(원산지 속성) | 선택 → 속성값 |

**절대 병합 금지**: 네이버 표준옵션/속성을 우리 `option_definition`/`category_option` 에 직접 섞지 않는다(축이 다름 — §1-4).

## 6. 수집 파이프라인

성격에 따라 두 방식:

**A. 검색형(브랜드·모델·제조사·속성)** — 스케줄 sync 없음. `naverClient` 에 `searchBrands(q)`·`searchModels(q)`·`getCategoryAttributes(naverCategoryId)` 등 추가. 관리자에 **검색 프록시 라우트**(`/admin/sourcing/naver/brands?q=`)를 두고, **개발서버에서 네이버 실시간 검색** → 결과 반환. 등록화면 위젯(카테고리 위젯 패턴)이 이걸 호출. **선택한 항목만** naver_brand/manufacturer/model 에 캐시 upsert.
> IP 화이트리스트 때문에 이 프록시는 **개발서버에서만** 실제 응답을 준다. 로컬은 403.

**B. 벌크형(카테고리, 그리고 원산지·사이즈·고시가 전량 조회로 확정되면)** — `naverTaxonomySync` 를 **제네릭 `syncResource(resource, fetcher, mapper, upserter)`** 로 일반화. `naver_taxonomy_sync_log.resource`·스케줄 재사용.

- 관리자에 **마스터 브라우저/검색 콘솔** — `/admin/sourcing/` 하위(캐시 조회 + 검색 프록시 테스트).

---

## 7. 선결 — 개발서버 read-only 프로브

로컬은 IP 차단이라, **개발서버에서** 각 엔드포인트를 GET 만 호출해 실제 payload(필드명·페이징·카테고리 종속 여부)를 찍는다. 이걸로 §3 ❓·§4 컬럼을 확정한 뒤 sync 를 구현한다. (쓰기 없음)

## 8. 단계별 계획

| Phase | 범위 | 비고 |
|---|---|---|
| **0 프로브** | 개발서버에서 `/v1/product-brands`·`/v1/product-models`·`/v1/product-attributes`(+속성값)·제조사 응답 확인 | 필드·검색 파라미터·페이징 확정 |
| **1 브랜드(+제조사) 검색 위젯** | `searchBrands` 프록시 + 등록화면 브랜드 검색 위젯(카테고리 위젯 복제) + 선택 캐시(naver_brand) + `products.naver_brand_id`(기존) 저장 | **검색-프록시 패턴 증명** |
| **2 카테고리별 속성** | `getCategoryAttributes(naverCategoryId)`+속성값 → 등록화면에서 상품의 네이버 카테고리로 속성 프리필 → 우리 `product_attribute` | naver_category_id 축(§1-4) |
| **3 원산지·사이즈·모델** | 프로브 결과에 따라 검색형/벌크형 결정 | |
| **4 상품정보제공고시(필드셋)** | 카테고리별 필수 필드셋 | **가장 무거움** — 별도 |

> 원칙(§32-4 재확인): "우리 카테고리→네이버 자동선택 안내"(상품 전송 시)와 이 마스터 수신은 상호 보완. 전송 기능은 별도.

## 9. 운영 규칙 (필수)

- 공용 개발 DB. **정리 DELETE 는 실제 `insertId` 로만**(문자열 파싱 금지), `mall_id` + `[TEST]` 마커 가드.
- 네이버 호출은 **개발서버(허용 IP)** 에서만. 로컬은 프로브도 403.
