# 아울렛(Outlet) 설계 — 몰 안의 몰 (shop-in-shop)

> 작성 2026-07-11. **상태: ⬜ 설계 초안 · 미착수.** 현재 `/outlet` 은 준비중 랜딩(`comingSoon`)이다.
>
> 이 문서는 **착수 전에 확정된 결정만** 기록한다. 스키마·화면은 착수 시점에 확정한다.
> 지금 기록하는 이유: GNB 메뉴 정리 작업(2026-07-11) 중 아울렛의 성격이 재정의됐고,
> 그 결정을 잃어버리면 같은 논의를 다시 하게 되기 때문이다.

---

## 0. 한 줄 정의

**아울렛은 필터로 뽑는 상품 목록이 아니라, 몰 안에 있는 또 하나의 몰이다.**

```text
❌ 아울렛 = discount_rate > 0 인 상품 목록          (폐기)
❌ 아울렛 = 상품그룹(product_group) 하나            (폐기 — 표현력 부족)
✅ 아울렛 = 아울렛 전용 상품 + 자체 카테고리 + 전용 관리자 메뉴로 구성된 독립 매장
```

---

## 1. 확정 결정 (2026-07-11, 사용자)

| # | 항목 | 결정 | 함의 |
|---|---|---|---|
| 1 | **상품 귀속** | **아울렛 전용 상품이 따로 있다.** 아무 상품이나 조건으로 끌어오는 게 아니다 | 아울렛 상품 지정(매핑 또는 플래그)이 필요 |
| 2 | **가격** | **같은 상품이 아울렛에서 다른 가격을 갖지는 않는다.** 단 **할인율·혜택은 아울렛에서 다를 수 있다** | **이중 가격 없음 → 장바구니·결제 경로를 건드리지 않는다.** 공동구매식 `group_buy_price` 패턴 불필요 |
| 3 | **구조** | **자체 카테고리/네비게이션이 필요하다** | 진짜 '몰내 몰'. 아울렛 전용 카테고리 관리가 따라온다 |
| 4 | **관리자** | **아울렛 관리 메뉴를 따로 둔다** | 상품그룹 관리로 대신하지 않는다("단순 상품그룹으로만 관리하면 약함") |
| 5 | **우선순위** | 랭킹·브랜드(경량 메뉴)를 먼저 끝낸 뒤 **별도 모듈 프로젝트로 착수** | MVP 1차 메뉴 정리 범위 밖 |

### 1-1. 결정 2가 중요한 이유

가격이 상품마다 이중화되면 **가격 표시 → 장바구니 → 주문 생성 → 결제 검증**이 전부 영향을 받는다.
이 저장소는 결제 트랜잭션 변경(`order_items` ALTER 등)을 프로젝트 차원에서 미루고 있다.

> 사용자 확답: *"같은 상품이 다른 가격을 가지거나 그런 것은 아니지만 할인율이나 혜택 등은 다를 수 있음"*

→ **아울렛은 진열(merchandising) 모듈이다.** 결제 경로 리스크가 없다. 이게 설계 난이도를 크게 낮춘다.
→ 다만 "할인율·혜택이 다를 수 있다"는 부분은 **착수 시 정확히 무엇을 뜻하는지 다시 확인**해야 한다(§4-1).

## 2. 폐기된 접근 (되풀이 금지)

### 2-1. `discount_rate` 필터 방식 — 폐기

`gnb_menu_design.md` §2-2 는 아울렛을 "상시 재고 소진, 할인율 큰 순"으로 정의하고
`discount_rate >= minDiscount` 필터 + 할인율 구간칩(30/50/70%↑)으로 설계했다.

**2026-07-11 구현했다가 되돌렸다.** 이유:
- 아울렛 상품이 "할인율이 높은 상품"과 같지 않다. 아울렛은 **어떤 상품이 아울렛에 들어가는가**의 문제다.
- 이 방식은 mall1 에서 항상 0건이었고(할인 상품 없음), mall2 에서는 4,499건이 무차별로 딸려 왔다.

제거한 것: `routes/feature.js` 의 `/outlet` 필터 라우트, `productController` 의 `isOutlet`·`minDiscount`·`maxDiscount`,
`views/partials/list_scaffold/hero_outlet.ejs`.

### 2-2. 상품그룹(`product_group`) 방식 — 부적합

오늘특가·베스트는 `product_group`(manual) + 관리자 상품그룹 관리로 해결했다(2026-07-11).
**아울렛에는 이 패턴을 늘려 쓰지 않는다.**

| 아울렛이 요구하는 것 | `product_group` 이 주지 못하는 것 |
|---|---|
| 자체 카테고리 분류 | 그룹은 평면 상품 목록일 뿐, 계층이 없다 |
| 아울렛 전용 배너·전시 | 그룹에 전시 요소가 없다 |
| 전용 관리자 메뉴 | 상품그룹 관리 화면 하나에 얹으면 아울렛의 운영 맥락이 사라진다 |
| 할인율·혜택 차등 | 그룹에 상품별 부가 속성이 없다 |

사용자 표현: **"단순 상품 그룹으로만 관리하면 약함."**

## 3. 참고할 기존 패턴

아울렛은 **기획전(exhibition) 패턴에 자체 카테고리를 더한 것**에 가깝다. 새로 발명하지 말고 클론한다.

| 필요 | 이미 있는 것 | 위치 |
|---|---|---|
| 상품 전시 매핑 | `exhibition_product` (상품 원래 가격 재사용, 순수 진열) | [`exhibition_design_and_development.md`](./exhibition_design_and_development.md) |
| 내부 섹션/탭 | `exhibition_section` | 〃 |
| 관리자 CRUD 관례 | 폼 POST + EJS + redirect | `controllers/admin/exhibitionController.js` |
| 상품 검색 모달 | `getProductSearch` JSON | `controllers/admin/productGroupController.js` |
| 카테고리 계층 | `categories` + `depthGuard`(max 3뎁스) | `services/tree/depthGuard.js` |
| 0건 폴백 랜딩 | `COMING_SOON.outlet` | `routes/feature.js` |
| 몰 스코프 | `req.mallId` / `mall_id` 첫 인덱스 컬럼 | `middleware/mallContext.js` |

**가격을 건드리지 않으므로 공동구매(`group_buy_product.group_buy_price`) 패턴은 쓰지 않는다.**

## 4. 착수 시 결정할 것 (지금 정하지 않는다)

### 4-1. "할인율·혜택은 다를 수 있다" 의 정확한 의미 ★

가장 먼저 사용자에게 확인해야 한다. 해석에 따라 스키마가 달라진다.

```text
해석 A) 아울렛 상품은 그냥 할인율이 높게 설정돼 있을 뿐이다 (products.discount_rate 그대로)
        → 추가 컬럼 불필요. 가장 가볍다.

해석 B) 같은 상품이라도 아울렛에서는 다른 할인율/혜택을 적용한다
        → outlet_product 에 할인율·혜택 컬럼 필요.
        → 단 '가격은 같다'고 했으므로, 할인율이 최종가를 바꾸면 결정 2와 모순된다.
           "혜택"이 배송비·적립·쿠폰 같은 부가 혜택을 뜻하는지 확인 필요.
```

### 4-2. 아울렛 카테고리를 어떻게 표현할 것인가

```text
안 1) categories.type 에 'OUTLET' 추가       — 기존 카테고리 트리·depthGuard 재사용
안 2) outlet_category 별도 테이블            — 아울렛 안에서만 쓰는 독립 분류
```
기존 `categories.type` 은 `NORMAL`/`THEME`/`BRAND` 3종이다. 안 1이 자산 재사용 면에서 유리하나,
아울렛 카테고리가 일반 카테고리와 섞여 보이지 않도록 스코프 처리가 필요하다.

### 4-3. 상품 귀속 방식

```text
안 1) outlet_product 매핑 테이블   — 관리자가 상품을 골라 담는다 (기획전 방식)
안 2) products.is_outlet 플래그    — 상품 자체에 아울렛 여부를 표시
```
"아울렛 전용 상품이 있다"는 결정을 보면 안 2 도 후보다. 단 한 상품이 일반몰과 아울렛에
동시에 있을 수 있는지에 따라 갈린다 → **착수 시 확인**.

### 4-4. 그 밖

- 아울렛 랜딩 구조(배너·기획전·베스트 등 아울렛 내부 전시를 어디까지)
- 아울렛 전용 GNB/네비를 아울렛 진입 후 헤더에 띄울지
- `/outlet` 하위 URL 체계 (`/outlet/category/{id}`, `/outlet/products/{slug}` ?)
- 관리자 메뉴 위치(`admin_menus` parent 및 `visible_roles`)

## 5. 현재 상태

```text
/outlet                     → comingSoon('outlet') 준비중 랜딩 (routes/feature.js)
feature_menu.OUTLET         → module_ready = 1 (GNB 에 노출 중)
아울렛 관련 테이블           → 없음
아울렛 관리자 메뉴           → 없음
```

착수 순서는 기획전과 같다: **스키마 → 관리자 CRUD → 콘텐츠 1건 등록 → 고객 라우트 교체**.
`dev = prod` DB 이므로 고객 라우트를 먼저 열면 운영에 빈 화면이 뜬다. **0건이면 준비중 랜딩 폴백을 유지**한다.
