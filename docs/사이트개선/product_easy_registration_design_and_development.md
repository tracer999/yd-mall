# 상품 등록 시 카테고리·브랜드 자동 생성 — 설계·개발 계획서

> **핵심 목적**: **상품을 등록하면, 그 상품의 카테고리와 브랜드가 자동으로 생성(또는 기존 항목에 자동 매칭)된다.**
> 운영자가 분류체계를 미리 만들어 두지 않아도, **상품 정보만 넣으면 분류가 상품을 따라 생겨난다.**
> 이것이 자동화의 본체이며, 그 외 등록 편의 방안은 §3에 **제안(의견)** 으로 덧붙인다.
> 작성: 2026-07-15 · 대상: 관리자 상품 등록 흐름(`controllers/admin/productController.js`)

---

## 1. 배경 — 왜 필요한가

이 프로젝트는 **몰 빌더 솔루션**이다. 목표는 "몰을 쉽게 찍어내는 것"이고, 납품받은 고객은 이커머스·분류체계 지식이 없는 경우가 많다. 그런데 현재는 **분류체계를 먼저 만들고, 그다음에 상품을 그 안에 넣는** 순서다:

- 카테고리·브랜드가 **모두 셀렉트박스(단일 선택)** 이고, 선택지는 운영자가 **미리 만들어 둔 것만** 뜬다.
- 아직 카테고리/브랜드를 하나도 안 만든 신규 몰이라면 **셀렉트가 비어 있어** 상품을 분류할 수 없다.
- 결국 운영자는 "상품을 등록하기 전에 카테고리·브랜드부터 손수 설계"해야 한다. 이 선행 작업이 초보 운영자에게 가장 큰 진입장벽이다.

**이 계획서가 뒤집으려는 순서**: 분류를 먼저 만드는 게 아니라, **상품을 등록하는 행위 자체가 카테고리·브랜드를 만들어 낸다.** 상품에 적힌 브랜드명·상품 성격을 근거로, 같은 것이 이미 있으면 거기에 붙이고(중복 방지), 없으면 새 카테고리·브랜드를 **자동 생성**해 상품에 연결한다. 운영자는 상품만 계속 넣으면 분류체계가 저절로 자라난다.

### 현재 구조 (조사 결과 요약)

| 항목 | 현황 | 근거 |

## 2. 핵심 — 상품 등록 = 카테고리·브랜드 자동 생성

이것이 **자동화의 본체**다. 상품 등록 시 브랜드명·카테고리명을 **자유 텍스트로 적기만 하면**(또는 URL/AI가 뽑아 오면), 시스템이 그 텍스트를 근거로 **기존 항목에 매칭하거나 없으면 새로 생성**해 상품에 연결한다. 셀렉트박스에서 미리 만든 것을 고르는 방식은 "선택지 중 하나"로 남고, 기본 경로는 **텍스트 → 자동 분류**가 된다.

### 2-1. 동작 흐름

상품에 담긴 브랜드/카테고리 텍스트(운영자 입력 · URL 임포트 · AI 추정 어디서 왔든)를 입력으로:

```
입력 텍스트(브랜드명/카테고리명 후보)
      │
      ▼
① 정규화 (공백·대소문자·특수문자·초성 처리)
      │
      ▼
② 유사 항목 검색 (categories 내 동일 type)
      │
   ┌──┴──────────────┐
   │ 임계값 이상 후보 있음 │  → ③-a 자동 매핑(단독 확실) 또는 후보 제시(모호)
   │ 후보 없음           │  → ③-b 신규 생성(가드 경유) 후 매핑
   └─────────────────┘
      │
      ▼
④ 상품에 category_id / brand_category_id 세팅 후 저장
```

### 2-2. 유사도 매칭 알고리즘

`categories` 의 `name`, `slug`(+ 브랜드는 `brand_profile.name_en`, `alias`, `initial_chosung`)를 대상으로 **다단계 매칭**. 무거운 임베딩 없이 규칙 기반으로 충분하다.

1. **정규화**: 소문자화, 앞뒤 공백·중복 공백 제거, `()[]-·/` 등 기호 제거, 한글 초성 추출(`shared/hangul.js` 의 `toChosung` 재사용).
2. **완전일치**: 정규화 문자열 `=` → **신뢰도 1.0**. 즉시 매핑.
3. **정규화 후 포함/역포함**: `LIKE '%키워드%'` 양방향(예: "코스알엑스" ↔ "COSRX") → 후보.
4. **별칭/영문/초성 일치**(브랜드 한정): `brand_profile.alias`·`name_en`·`initial_chosung` 일치 → 후보. `brandController.searchJson` 로직 재사용.
5. **편집거리(선택)**: 오타 흡수용 Levenshtein/Jaro-Winkler. 문자열 길이 대비 유사도 ≥ 0.85 만 후보. (경량 구현 또는 `fast-levenshtein` npm)
6. **스코어링 & 임계값**:
   - `score ≥ 0.95` **단일** 후보 → **자동 매핑**(운영자 확인 불필요, 로그만 남김)
   - `0.6 ≤ score < 0.95` 또는 **복수** 후보 → **후보 제시**(운영자 1클릭 선택, "새로 만들기"도 병기)
   - `score < 0.6` → **후보 없음**으로 간주 → 신규 생성 경로

> 임계값은 상수로 하드코딩하지 말고 `system_settings` 키(예: `auto_taxonomy_confidence_auto`, `_suggest`)로 두어 몰별 조정 가능하게 한다. (프로젝트 컨벤션: 설정은 `system_settings` 우선)

### 2-3. 없을 때 자동 생성 — 무결성이 핵심

신규 카테고리/브랜드 생성은 **반드시 정본 로직(`categoryController.postAdd`)과 동일한 가드**를 경유한다. 컨트롤러에서 직접 `INSERT` 하지 말고 **공용 서비스로 추출**한다.

- 신설 서비스 `services/catalog/taxonomyResolver.js` (제안):
  - `resolveOrCreateCategory({ mallId, name, type, parentId=null })`
  - `resolveOrCreateBrand({ mallId, name, aliasHints })`
- 생성 시 필수 통과 가드 (`services/tree/depthGuard.js`):
  - `assertDepthAllowed` — 뎁스 초과 차단
  - `wouldCreateCycle` — (부모 지정 시) 순환 차단
  - `assertSameType` — 동일 type 내에서만 부모 지정
- 신규 생성 카테고리 기본값:
  - `depth = assertDepthAllowed 반환값`, `parent_id = NULL`(최상위) 기본, `is_active=1`, `pc_visible/mobile_visible=1`, `display_order = MAX+1`
  - **`slug` 자동 생성**(현재 categoryController가 안 채우는 항목) — 한글→로마자/난수 슬러그. 이참에 보완.
- 신규 BRAND 생성 시 **`brand_profile` 스텁도 함께 생성**(현재는 편집 최초 진입 때만 생김) → 초성/영문명 채워 두면 다음 유사매칭이 더 정확해지는 선순환.
- **Shopify 동기화**: 현재 비활성(`shopify_sync_enabled=0`)이므로 자동생성 시 컬렉션 동기화는 **호출하지 않는다**. 단 훗날 켜질 때를 위해 resolver가 "동기화 대상 신규 id"를 반환하도록 설계.

### 2-4. 폭주 방지 — 자동 생성의 리스크

자동 생성은 편하지만 **오타 하나가 유령 카테고리를 양산**할 수 있다("코스알엑스", "코스알엑스 ", "COSRX", "cosrx" 4개 생성). 방지책:

- **생성 전 유사매칭을 반드시 선행**(§2-2). 임계값 이하일 때만 생성.
- **대량 등록 시 세션 내 신규 생성 캐시** — 같은 배치에서 방금 만든 항목을 우선 재매칭.
- **"자동 생성됨" 플래그/뱃지** — 자동 생성된 분류는 관리자 목록에서 표시(예: `description='[auto]'` 또는 신규 `is_auto_generated` 컬럼)하여 운영자가 나중에 **병합·정리**할 수 있게. (선택: 카테고리 병합 도구는 별도 과제)
- **자동 생성 총량 가드** — 한 번의 등록/배치에서 신규 생성이 N개를 넘으면 자동 생성을 멈추고 운영자에게 확인 요청.

### 2-5. UX — "확인 후 저장" 기본, "완전 자동" 옵션

두 가지 모드를 `system_settings.auto_taxonomy_mode` 로 제공:

| 모드 | 동작 | 대상 |
|---|---|---|
| `assist`(기본) | 유사후보/신규생성안을 **제안**, 운영자가 1클릭 확정 | 신중한 운영자 |
| `auto` | 임계값 넘으면 매핑, 없으면 생성까지 **무확인 진행**, 결과만 통지 | 대량·빠른 등록 |

폼(`form.ejs:71-96`)에는 셀렉트박스 옆에 **"직접 입력/자동" 토글**을 추가:
- 셀렉트에 없는 브랜드는 **자유 텍스트 입력** 가능 → 제출 시 resolver가 처리
- `postImportUrl`(URL 임포트) 결과의 브랜드명도 완전일치가 아니라 **유사매칭 후보로** 넘김(현재는 완전일치만)

### 2-6. AI 사용 여부에 따른 등급 분기 (Standard / Premium) — ★설계 전제

납품 고객마다 AI(OpenAI 등) 사용 조건이 다르다(비용·키 발급·정책). 따라서 **AI 사용 여부를 납품 등급으로 정하고, AI를 쓰든 안 쓰든 각각 완결되는 두 갈래를 처음부터 설계**한다. 이것이 가능한 이유는 **자동화 본체(§2-2 매칭 · §2-3 생성 · 무결성 가드 · 폭주 방지)가 AI를 전혀 요구하지 않기** 때문이다. AI는 "분류할 근거 텍스트를 만들어 내는 입력층"에만 얹힌다.

#### 아키텍처 이음새 — 입력층 ↔ 처리층 분리 (핵심)

```
[입력층: 브랜드/카테고리 텍스트 확보]          [처리층: §2 공용 엔진]
  Standard → 운영자 타이핑 · URL 임포트 · 키워드 사전   ┐
  Premium  → + AI 추정(상품명·이미지·설명)             ├─→  taxonomyResolver
                                                      ┘    (매칭→생성→가드)
                                                           ※ 등급 무관 동일 코드
```

- **처리층(§2)은 등급과 무관하게 한 벌**이다. AI on/off는 **입력층만 교체**한다 → 두 등급이 로직을 공유하므로 유지보수 부담이 없다.
- 이는 이미 코드베이스가 쓰는 패턴(`system_settings` 플래그 + `middleware/shopifyFlag.js` 식 UI 게이팅)과 동일한 방식으로 구현한다.

#### 등급별 동작

| 구분 | **Standard (AI 미사용)** | **Premium (AI 사용)** |
|---|---|---|
| 브랜드/카테고리 근거 확보 | 운영자 자유텍스트 입력 · URL 임포트 · **키워드 사전** | 좌측 전부 **+ AI가 상품명·이미지·설명에서 추정** |
| 무입력 상태 등록 | "미분류" 폴백(§B-1) | AI가 추정해 **무입력도 실제 분류로** |
| 매칭·자동 생성(§2-2·2-3) | ✅ 동일 | ✅ 동일 |
| 무결성 가드·폭주 방지 | ✅ 동일 | ✅ 동일 |
| 추천문구·SEO 메타·태그 | 수동/템플릿 | AI 일괄 생성(§B-2) |
| 이미지 기반 분류 | ✗ | ✅ 비전 모델 |

> Standard도 **자동 생성 자체는 100% 동작**한다. 단지 "근거 텍스트"를 운영자가 주거나(타이핑·URL) 키워드 사전이 대는 것이고, Premium은 그 근거까지 AI가 만들어 "무입력 완전 자동"에 가까워진다.

#### 설정 · 폴백 · 전환

- **설정 키**(`system_settings`): `product_taxonomy_ai_enabled`(0/1) — 납품 시 등급에 맞춰 세팅. 프로젝트 규칙상 `system_settings`가 `.env`를 오버라이드하므로 여기 두면 관리자 화면에서 등급 전환 가능.
- **Fail-safe**: Premium인데 `OPENAI_API_KEY`가 없거나 호출 실패 → **자동으로 Standard 경로로 폴백**(경고 로그만, 등록은 계속). AI는 어디까지나 선택 계층이라 장애가 등록을 막지 않는다.
- **UI 게이팅**: Premium일 때만 폼에 "AI 자동 채움" 버튼 노출(`shopifyFlag`와 동일한 미들웨어 플래그 방식). Standard에선 숨김.
- **무중단 업그레이드/다운그레이드**: Standard→Premium 은 **코드 재배포 없이** 플래그 on + 키 입력만으로 전환. 반대도 동일. 이미 등록된 상품·분류는 그대로 유지된다.

---

## 3. 등록 자동화를 더 밀어붙이는 추가 제안 (의견)

§2의 자동 생성이 본체라면, 아래는 **"등록 행위 자체를 최대한 쉽고 자동화"** 하기 위해 함께 제안하는 방안들이다. 모두 §2의 자동 분류 엔진을 재사용하며, 운영자가 손댈 것을 줄이는 방향으로 우선순위를 매겼다.

### B-1. 최소 필드 등록(Quick Add) — ★최우선
- **운영자가 직접 입력하는 건 딱 3가지: 상품명 · 가격 · 대표이미지.** 브랜드·카테고리를 포함한 나머지는 자동 생성/스마트 기본값이 채운다.
- 브랜드·카테고리 처리 경로는 등급(§2-6)에 따라 다르다: **Standard** = 운영자가 적은 텍스트/URL을 §2 엔진이 매칭·생성, **Premium** = 상품명·이미지·설명에서 AI가 추정한 근거까지 §2 엔진에 투입 → "3필드만 입력" 경험은 특히 Premium에서 완성된다.
- 근거가 전혀 없는 극단적 경우(Standard에서 텍스트 미입력 등)에만 **"미분류" 기본 카테고리**로 폴백(몰 생성 시 1개 시드)해 상품이 유실되지 않게 한다. 재고 `999`·몰 기본 배송·`판매중`을 기본값으로.
- 저장 후 "상세정보 더 채우기" CTA로 점진적 완성 유도(진행률 배지).

### B-2. AI 자동 채움(One-shot Enrichment) — ★상 · **Premium 등급 전용**(§2-6)
> 이 항목은 §2-6의 **Premium 트랙에서만 활성**된다. Standard 등급은 아래 대신 키워드 사전·URL 임포트로 근거를 확보한다.

이미 붙어 있는 OpenAI 클라이언트(`productController.js:8-19`)를 **분류·메타 일괄 생성**으로 확장:
- 입력(상품명 + 대표이미지 + 짧은 설명) → **카테고리 후보 · 브랜드 추정 · 추천문구 · SEO 메타 · 태그**를 한 번에 생성.
- 카테고리/브랜드 결과는 §2-2 유사매칭에 그대로 투입(생성 남발 방지).
- 기존 `generateAIRecommendation`(:112), `generateMetaDescription`(:141)과 같은 계층에 `suggestTaxonomy()` 추가.
- 키 없으면(openai=null) **규칙 기반으로 폴백**(키워드 사전 매칭) → AI 미설정 몰도 동작.

### B-3. URL 임포트 강화 — ★상
`postImportUrl`(:544)은 이미 외부 URL에서 상품 정보를 긁어온다. 여기에:
- 브랜드 완전일치(`=`) → **유사매칭(§2-2)** 로 교체
- 카테고리도 임포트 데이터에서 추정해 자동 매핑/생성
- "붙여넣기 한 번으로 90% 완성" 경험.

### B-4. 대량 등록(엑셀/CSV) — ★중
- 스프레드시트 업로드 → 행마다 방안 A resolver 적용(세션 캐시로 중복 생성 방지).
- 컬럼 최소화(상품명·가격·이미지URL·브랜드텍스트·카테고리텍스트). 브랜드/카테고리는 **텍스트로만 적어도** 자동 매핑/생성.
- 결과 리포트: 매핑 N · 신규생성 M · 실패 K + 되돌리기.

### B-5. 스마트 기본값 & 템플릿 — ★중
- **상품 템플릿**: 자주 쓰는 배송/반품/옵션 세트를 템플릿화해 신규 등록에 원클릭 적용.
- **최근값 기억**: 직전 등록의 브랜드·배송·카테고리를 기본 프리필.

### B-6. 등록 마법사(Wizard) — ★하(UX 리뉴얼 시)
- 3스텝(기본정보 → 이미지 → 확인). 각 스텝에서 방안 A/B가 백그라운드로 채움.
- 초보 운영자용. 기존 단일 폼은 "고급 모드"로 유지.

## 4. 자동 생성 결과가 "실제로 팔리는 상태"인지 확인 (부수 조건)

자동화의 목적은 쉬운 등록이지만, 자동으로 만들어진 분류에 상품이 붙었을 때 **그 상품이 고객 화면에 실제로 노출되는지**까지 확인해야 자동화가 완결된다(안 그러면 "자동으로 분류됐는데 안 보이는" 상황). 등록 완료 상품이 만족해야 할 최소 조건:

- [ ] `category_id`가 자동 매핑/생성됨(최악의 경우 "미분류") → 목록/검색 노출
- [ ] `price` > 0, `status = 판매중`, 재고 > 0(또는 재고관리 off)
- [ ] 대표이미지 1장 이상
- [ ] (권장) `brand_category_id` 자동 매핑/생성 → 브랜드 허브 노출. 없으면 provider 텍스트라도 채움

검증기 `services/catalog/sellabilityCheck.js`(제안)가 저장 시 위를 점검하고, 미충족 항목을 **경고로만** 표시(저장은 막지 않음 — 마찰 최소화 원칙). "판매 가능" / "노출 제한" 배지로 상태를 시각화.

## 5. 개발 로드맵 (단계별)

> ⚠️ **정책 전환 — 이 로드맵을 그대로 읽지 말 것.**
> [`네이버_기반_글로벌_카테고리_재구성_설계.md`](./네이버_기반_글로벌_카테고리_재구성_설계.md) §0 이 이 문서의 대전제("분류를 먼저 만들지 않는다 · 상품 등록 행위가 카테고리를 만든다")를 **의도적으로 뒤집었다.**
> 지금 방침은 **네이버 L1~L3 표준 뼈대를 선(先)-시드**하고, 상품 등록은 "생성"보다 **"기존 표준 노드에 매칭"** 이 기본이다.
> 아래 §2 의 유사매칭 엔진·무결성 가드·폭주 방지는 그 전환 뒤에도 그대로 쓰이지만, "자동 생성이 본체"라는 서술은 더 이상 현행 정책이 아니다.

> **1~4단계 완료** — `taxonomyResolver`(유사매칭·자동생성·브랜드 스텁), `postAdd`/`postEdit`/`postImportUrl` 훅, 임포트 카테고리 추출, 폼 "직접 입력" 칸, 몰별 "미분류" 폴백 시드까지 반영됨. 상세는 git 이력 참조.
>
> **잔여(5~8단계)**: `assist`(후보 확인 UI) 모드 · Quick Add(B-1) · AI 채움(B-2·Premium) · CSV 대량등록(B-4) · 정리 도구(자동생성 뱃지·카테고리 병합).


| 단계 | 산출물 | DDL |
|---|---|---|
| **1. 무결성 리팩터** | `services/catalog/taxonomyResolver.js` 로 카테고리/브랜드 생성 로직 추출(가드 경유). `categoryController.postAdd`도 이 서비스 사용하도록 통합 | 0 |
| **2. 유사매칭 엔진** | `resolveOrCreate*` + 정규화·초성·LIKE·(선택)편집거리. `brandController.searchJson` 재사용 | 0 |
| **3. 등록 훅 + 등급 게이트** | `postAdd`/`postEdit`/`postImportUrl` 에 "빈 값/텍스트 → resolve" 연결. `assist`/`auto` 모드 + **`product_taxonomy_ai_enabled` 등급 플래그**(§2-6) 분기 | 0 |
| **4. UX** | `form.ejs` 자유텍스트 토글 + 후보 제시 UI + (Premium만) "AI 자동 채움" 버튼 게이팅. "미분류" 기본 카테고리 시드 | 미분류 시드 1행 |
| **5. Quick Add(B-1)** | 최소 3필드 등록 + 스마트 기본값 (Standard/Premium 공통 동작) | 0 |
| **6. AI 채움(B-2) · Premium** | `suggestTaxonomy()` + 키 없음/실패 시 Standard 폴백 | 0 |
| **7. 대량/CSV(B-4)** | 업로드 파서 + 배치 resolver + 리포트 | 0 |
| **8. 정리 도구** | 자동생성 항목 뱃지 + 카테고리 병합 | (선택) `is_auto_generated` 컬럼 |

## 6. 리스크 · 결정 필요 사항

- **자동 생성 남발**: §2-4 방지책 필수. `auto` 모드는 신뢰 운영자에게만 권장.
- **한글 유사도 정확도**: 초성·별칭 사전이 빈약한 초기 몰에선 오매핑 가능 → 초기엔 `assist` 모드 기본.
- **"미분류" 카테고리 정책**: 이 카테고리를 고객 GNB에 노출할지, 백오피스 전용으로 숨길지 **결정 필요**. (권장: 고객에겐 숨기고 검색·전체상품에만 포함)
- **자동생성 플래그 컬럼**: `is_auto_generated` 추가는 스키마 변경 → 개발 DB 기준으로만 진행(상용 없음). `alter-table` 스킬로 schema.sql 동기화.
- **provider 중복 저장**: 브랜드 자동매핑 시 `resolveBrandName`(:71)과 정합 유지 필요.
- **납품 등급 기본값(§2-6)**: 신규 몰의 `product_taxonomy_ai_enabled` 기본을 Standard(0)로 둘지 Premium(1)로 둘지, 그리고 Premium 등급의 과금·키 발급을 납품 계약 어느 단계에서 정할지 **결정 필요**. (권장: 기본 Standard, Premium은 옵션 상품화 — 키 미입력 시 자동 Standard 동작이라 안전)

## 7. 요약

- **최대 목적은 "상품을 등록하면 카테고리·브랜드가 자동으로 생성된다"** — 분류를 먼저 만들고 상품을 넣는 순서를 뒤집어, **상품이 분류를 만들어 내게** 한다. 이것이 자동화의 본체다(§2).
- 이 자동 생성은 **DDL 0으로 구현 가능**하다. 브랜드·카테고리가 같은 `categories` 테이블이라 로직을 통일할 수 있고, 무결성 가드(`depthGuard`)·유사검색(`searchJson`)·AI 클라이언트가 이미 있어 확장 지점이 명확하다.
- 단, 자동 생성은 **유사매칭 선행 + 임계값 + 폭주 방지**가 없으면 유령 분류를 양산한다. 그래서 "같은 게 있으면 붙이고, 없을 때만 생성"이 알고리즘의 핵심이고, 초기엔 `assist`(확인 후) 모드를 기본으로 둔다.
- **AI는 필수가 아니라 등급 옵션이다(§2-6)**. 자동 생성 본체(§2)가 AI를 안 쓰므로, 납품 시 **Standard(AI 미사용) / Premium(AI 사용)** 으로 갈라도 처리층은 한 벌을 공유한다. AI on/off는 "근거 텍스트를 만드는 입력층"만 바꿀 뿐이고, `system_settings` 플래그로 **코드 재배포 없이 전환**된다. Premium인데 키가 없으면 자동으로 Standard로 폴백돼 장애가 등록을 막지 않는다.
- 나머지(§3)는 **의견**이다 — 등록 자동화를 더 밀어붙이는 방향으로 **최소필드 Quick Add · (Premium)AI 일괄 채움 · URL 임포트 강화 · CSV 대량등록**을 제안한다. 모두 §2 엔진을 재사용해 "운영자가 손댈 것을 최소화"하는 데 초점을 둔다.
