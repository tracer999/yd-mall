# 도매꾹·온채널 → 빌더 → 스마트스토어 연동 상세 설계 (통합본)

- 문서 버전: 3.0 (상세설계 v1.0 + 검토·보강 v2 통합, 자기완결형)
- 작성일: 2026-07-16
- 대상 스택: **Express 5 / EJS SSR / MySQL(yd_mall) / PM2 fork / Redis** (yd-mall 몰 빌더)
- 함께 볼 것: [`개발계획서`](./도매꾹_온채널_스마트스토어_연동_개발계획서.md)
- **네이버 스마트스토어(인증·상품등록·호출제한·고시·매핑)는 [`네이버_스마트스토어_연동.md`](./네이버_스마트스토어_연동.md) 가 최신이며 이 문서보다 우선한다.** 네이버 관련 신규 내용은 그 파일에만 추가할 것.
- 범위: 무재고 위탁판매 상품의 소싱, 상품 편집, 스마트스토어 등록, 재고·가격 동기화, 주문 수집, 공급처 발주, 송장 반영, 취소·반품, 그리고 스마트스토어 직접등록 상품의 역방향 수집.

> 이 문서의 외부 API 세부(엔드포인트·파라미터·버전)는 작성 시점 기준 정리이며, **개발 착수 전 각 사업자 최신 공식 문서로 재검증**을 전제로 한다.

## 1. 목적

몰(판매자)이 도매꾹·도매매 또는 온채널에서 위탁판매 가능한 상품을 선택해, 빌더에서 상품정보를 편집하고 네이버 스마트스토어에 등록·판매하는 통합 시스템을 설계한다. 아울러 스마트스토어에 **직접 등록된 상품을 우리 몰로 가져오는 역방향 흐름**도 지원한다.

목표 사용자 흐름:

```text
공급처 상품 검색 → 가져오기(중간 테이블 적재) → 선택 후 상품 등록
   → 상품명·가격·이미지·옵션·고시정보 편집 → 스마트스토어 등록(검수)
   → 재고·가격 동기화 → 주문 수집 → 공급처 발주 → 송장 수집 → 배송 처리
   → 취소·교환·반품 및 정산 관리
역방향:  스마트스토어 직접등록 상품 → 우리 몰로 수집 → 통합 관리
```

핵심 원칙: **상품정보는 빌더가 표준화하고, 공급 재고는 공급처가 기준이며, 판매 주문은 스마트스토어가 발생원**이다.

## 2. 제품·운영 전제 (yd-mall 반영)

- **몰 빌더 성격**: yd-mall 은 특정 몰을 운영하는 앱이 아니라 몰을 찍어내는 빌더다. 이 연동은 **개별 몰(스토어)에 붙는 기능**이다.
- **몰당 1:1 연동**: "위탁판매 판매자 = 몰 1개"로 본다. 한 몰이 자신의 도매꾹/온채널/스마트스토어 계정을 각 1세트 연결한다. (한 몰에 다계정 요건이 생기면 `channel_account` 1:N 승격.)
- **사용 여부 = 몰별 설정 + 유료 기능**: 관리자에서 몰마다 이 연동의 사용 여부(on/off)를 켠다. **납품 시 이 기능 사용은 추가 계약 대상**이므로, 플래그가 꺼진 몰에서는 대메뉴·기능·워커 처리가 전부 비활성이어야 한다.
- **릴리스 차수**:
  | 차수 | 범위 |
  |---|---|
  | **1차** | 상품 가져오기 → 중간 테이블 → 선택 등록 → 스마트스토어 등록 + 역방향 수집 + **재고·주문 "가져오기" 버튼(수동)**. 배치 없음 |
  | **2차** | 재고·주문 자동(배치) 동기화, (반)자동 발주, 송장·클레임 |
  1차의 수동 "가져오기" 로직을 **2차 배치가 그대로 재사용**(자동 트리거만 추가).
- **개발 환경 전제**: 이 프로젝트는 전 과정 개발 단계다. `main` 푸시 = 개발 서버 즉시 반영. DB 는 로컬·서버 공용 1벌.

## 3. 공식 연동 범위와 전제조건

### 3.1 네이버 스마트스토어 (커머스API)

네이버 커머스API로 상품·주문·정산·판매자정보를 다룬다. 빌더가 처리할 주요 기능: 상품 등록, 원상품·채널상품 조회·수정, 판매상태 변경, 옵션별 재고 변경, 주문 조회, 발주확인, 배송 처리, 취소·반품·교환 처리, 판매자 인증·권한.

필수 전제: 스마트스토어 판매자 계정, 커머스API센터 애플리케이션, 판매자별 인증·접근 권한, 스토어·채널 정보, 호출 제한·오류 정책 대응. **인증 방식의 실제 구현은 §8.2 참고.**

공식 문서: https://apicenter.commerce.naver.com/docs/introduction , https://apicenter.commerce.naver.com/docs/commerce-api/current

### 3.2 도매꾹·도매매

- **Open API**: 상품 목록/상세/카테고리 조회 중심. REST 파라미터(`ver`, `mode`(getItemList·getItemView 등), `aid`(API Key)) 형태로 JSON/XML 응답 — 정확한 필드는 최신 문서로 매핑.
- **Private API(2026-03 승인제)**: 로그인·주문·구매·상품등록 등 회원권한 기능은 승인 필요. 승인 전엔 주문 생성 불가일 수 있음.
- 전제: 사용자별 계정 연결, Open API Key, (필요 시) Private 승인, 위탁배송 가능 여부 확인.

공식 문서: https://openapi.domeggook.com/ , https://domemedb.domeggook.com/index

### 3.3 온채널

공식 안내는 주로 **온채널 자체 서비스가 판매자 스마트스토어와 연결**되는 구조다. 제3자 빌더가 온채널 상품·재고·발주 API를 직접 쓸 수 있는지는 공개 문서로 확정 불가. 따라서 세 수준을 지원 설계한다.

| 수준 | 방식 | 빌더 역할 |
|---|---|---|
| **L1** | 상품정보 수동·CSV 가져오기 | 상품 편집 및 스마트스토어 등록 (**MVP 시작점**) |
| L2 | 온채널 제휴 API | 상품·재고·주문·송장 자동화 |
| L3 | 온채널 자체 스마트스토어 연동 병행 | 빌더는 상품 관리 중심 |

공식 안내: https://onchannelnotice.oopy.io/guide/api , https://www.onch3.co.kr/seller.html

## 4. 아키텍처

### 4.1 빌더가 통합 허브 (공급처/판매채널 직접 결합 금지)

공급처 상품을 그대로 채널 API로 보내지 않고, 반드시 빌더 내부 표준 모델을 거친다.

```text
                    ┌────────────────────┐
                    │ 판매자 관리자 화면 │
                    └─────────┬──────────┘
                    ┌─────────▼──────────┐
                    │   쇼핑몰 빌더 Core │
                    │ 상품·SKU·가격·주문 │
                    └──────┬───────┬─────┘
              ┌────────────┘       └─────────────┐
    ┌─────────▼──────────┐             ┌─────────▼──────────┐
    │ Supplier Gateway   │             │ Sales Channel GW   │
    └───────┬───────┬────┘             └─────────┬──────────┘
    ┌───────▼───┐ ┌─▼─────────┐          ┌────────▼─────────┐
    │ 도매꾹/매 │ │ 온채널    │          │ 네이버 커머스API │
    └───────────┘ └───────────┘          └──────────────────┘
```

```text
잘못된 구조:  도매꾹/온채널 상품 객체 → 그대로 네이버 API 전송
권장 구조:    도매꾹/온채널 상품 → 정규화 → BuilderProduct/Variant → 채널 변환 → NaverOrigin/ChannelProduct
```

### 4.2 기준 데이터 범위

| 데이터 | 기준 시스템 |
|---|---|
| 공급 상품 원본 | 도매꾹·도매매 / 온채널 |
| 판매용 상품명·설명·마진 | 빌더 |

| 실제 공급 가능 재고 | 공급처 |
| 스마트스토어 노출 재고 | 빌더(안전재고 정책 적용 계산) |
| 고객 주문 | 스마트스토어 |
| 내부 통합 주문 | 빌더 |
| 공급처 발주 | 도매꾹·도매매 / 온채널 |
| 송장 원본 | 공급처 |
| CS·반품 책임 | 스마트스토어 판매자 |

### 4.3 실제 스택 배치 — 모듈러 모놀리스 + 아웃박스 + 워커

메시지버스(Kafka/SQS)·PostgreSQL·마이크로서비스가 아니라, **기존 Express 앱 + MySQL 아웃박스 + PM2 워커 1개**로 구현한다.

```text
PM2 app 1: yd-mall (기존 웹, fork/1)   — 관리자 화면. 사용자 명령을 DB 트랜잭션(상태+outbox)으로만 커밋
        │ (같은 MySQL: yd_mall)
MySQL   — 도메인 테이블 + integration_outbox
        │ poll
PM2 app 2: yd-mall-worker (신규, fork/1) — worker.js: outbox 소비 + (2차)폴링 스케줄러 + 재시도/DLQ
        ├─ services/sourcing/supplier/*  (Domeggook, OnChannel)
        └─ services/sourcing/channel/*   (NaverCommerce)
```

**왜 이 구성인가**
- 큐 인프라 신규 도입 최소화: 저장소에 BullMQ/bull/agenda/node-cron 없음, Redis 는 세션 전용. 위탁 연동의 신뢰성 요건은 **MySQL 아웃박스 + 폴링 워커**로 충분.
- PM2 워커 분리는 `ecosystem.config.cjs` 의 `apps` 배열에 두 번째 앱 추가로 가능. 웹 프로세스가 외부 API I/O로 막히지 않도록 **반드시 분리**.
- 모듈 경계: `services/sourcing/` 아래 `supplier/`·`channel/`·`core/`·`outbox/` 로 나눠 후일 분리 가능.

**BullMQ 대안(선택, 결정 게이트 D2)**: Redis 재사용해 아웃박스 소비를 BullMQ로. 지연 짧음·지수백오프 내장이 장점이나 의존성 추가·이중 정합 부담. **MVP는 MySQL 아웃박스 채택**, 처리량 급증 시 후속 도입.

### 4.4 어댑터 계층 (services/shopify 패턴 재사용)

기존 `services/shopify/adminClient.js`(토큰 캐시 + fetch 래퍼 + 에러 정규화)와 `webhookHandler.js`(TOPIC_HANDLERS 맵)를 **참고 템플릿**으로 쓴다. 단 전역 `process.env`·`_cachedToken` 싱글톤을 **자격증명 주입형(`cred` 인자 + `Map<credId, token>` 캐시)** 으로 바꾼다(§8).

## 5. 멀티몰 스코핑 & 자격증명 저장

- 이 연동의 모든 테넌시 개념은 이 저장소에서 **`mall_id`** 로 구현한다. 신규 테이블은 `mall_id BIGINT NOT NULL DEFAULT 1` 스코프 컬럼을 가진다.
- 관리자 컨트롤러는 **`req.adminMallId`**(편집 중인 몰)로 스코프한다. 손님 몰(`req.mallId`)과 혼동 금지.
- **몰별 사용 여부 플래그**(§2)는 몰 설정 테이블/컬럼(`outlet_setting` 선례처럼 `mall_id` PK)에 저장. 워커도 처리 전 이 플래그를 확인.
- **외부 계정 자격증명**은 전역 `system_settings`(단일 계정)가 아니라 **몰 스코프 테이블 + `shared/crypto` 암호화**로 저장:

```sql
mall_channel_credential
- id BIGINT PK
- mall_id BIGINT NOT NULL
- channel ENUM('DOMEGGOOK','DOMEME','ONCHANNEL','NAVER_SMARTSTORE')
- account_label VARCHAR(100)
- client_id VARCHAR(255)          -- 평문 식별자
- secret_enc TEXT                 -- 'ENC:' + AES-256-GCM (shared/crypto)
- extra_json JSON                 -- 스토어채널ID, 반품지코드 등
- status ENUM('ACTIVE','INVALID','EXPIRED','DISABLED')
- last_verified_at DATETIME
- created_at / updated_at
- UNIQUE (mall_id, channel, account_label)
```

- 저장: `secret_enc = 'ENC:' + crypto.encrypt(secret, ENCRYPTION_KEY)`. 읽을 때 얇은 헬퍼로 복호화해 **어댑터에 인자 주입**.
- 액세스 토큰은 DB 영속화하지 않고 프로세스 메모리 캐시(`adminClient.js` 의 `_cachedToken` 패턴), 만료 1분 전 갱신.

### 5.1 몰별 독립 자격증명 (서브몰 대비)

- **모든 API 키·설정은 몰별(`mall_id`)로 독립 설정한다.** 각 몰이 자기 채널 자격증명을 따로 갖는다. `mall_channel_credential` 는 `UNIQUE(mall_id, channel, account_label)` 로 몰마다 독립 행이다.
- **"1:N"(하나의 자격증명을 여러 몰에 공유하는 참조) 개념은 두지 않는다.** 여러 서브몰이 같은 실제 계정을 쓰더라도 **각 몰에 동일 키를 각각 저장**한다(공유 참조 엔티티 없음). → 스키마가 단순하고, 몰 삭제·권한 변경이 서로 간섭하지 않으며, 특정 몰만 키를 교체하기도 쉽다.
- **대부분 단일 몰**이지만 한 운영자가 여러 (서브)몰을 운영할 수 있으므로, 관리자 연결 화면은 다음을 제공한다:
  - **현재 편집 중인 몰(`req.adminMallId`)을 명확히 표시 + 몰 전환** UI. (자격증명·사용여부·정책은 전부 이 몰 기준으로 저장)
  - 편의 기능 **"다른 몰의 자격증명 복사"** — 같은 계정을 여러 서브몰이 쓸 때 재입력 없이 **값을 복제**(참조 공유가 아니라 복사). 복사 후 각 몰에서 독립 수정 가능.
- 사용여부(`mall_channel_setting`)·기본 마진 정책도 동일하게 **몰별 독립**. 신규 몰(서브몰)이 생기면 `sourcing_enabled=0` 으로 시딩되어, 계약·설정 전에는 꺼져 있다.
- 서브몰은 `mall` 테이블상 별도 `mall_id` 행일 뿐(현재 스키마는 평면 구조, 부모-자식 컬럼 없음)이라, 이 연동은 **몰 계층을 전제하지 않고 각 `mall_id` 를 독립 단위로** 다룬다.

## 6. 사용자 역할

- **판매자**: 공급처·스마트스토어 계정 연결, 상품 검색·선택, 판매 승인 신청, 가격·마진 설정, 콘텐츠 수정, 스마트스토어 등록, 주문·품절·클레임 확인, 공급처 결제수단 관리.
- **빌더 운영자**: 커넥터 관리, API 상태 모니터링, 카테고리 매핑, 금지·인증상품 규칙, 실패 재처리, 판매자 권한·사용량, 개인정보 접근 감사.

- **네이버 스마트스토어**: 상품 노출, 고객 주문·결제, 주문·클레임 상태, 구매자 정보 제공, 배송·정산 상태.

## 8. 어댑터 설계

### 8.1 공통 인터페이스 (자격증명 주입형) + Capability

기능을 필수로 강제하지 않는다. 공급처·계약 등급에 따라 지원이 다르므로 **Capability 기반**으로 화면에서 미지원 기능을 숨기거나 수동 처리로 전환한다.

```typescript
interface SupplierAdapter {
  validateConnection(cred: Credential): Promise<ConnectionResult>;
  searchProducts(cred: Credential, req: SearchRequest): Promise<Page<SupplierProductSummary>>;
  getProduct(cred: Credential, supplierProductNo: string): Promise<SupplierProductDetail>;
  getInventory(cred: Credential, nos: string[]): Promise<SupplierInventory[]>;
  getPrice?(cred, nos): Promise<SupplierPrice[]>;
  requestSaleApproval?(cred, no): Promise<SaleApprovalResult>;
  createProcurementOrder?(cred, req): Promise<ProcurementResult>;
  getProcurementOrder?(cred, no): Promise<ProcurementOrderStatus>;
  getShipment?(cred, no): Promise<SupplierShipment[]>;
  requestCancellation?(cred, req): Promise<SupplierClaimResult>;
  requestReturn?(cred, req): Promise<SupplierClaimResult>;
}

interface SalesChannelAdapter {
  validateConnection(cred): Promise<ConnectionResult>;
  createProduct(cred, product): Promise<ChannelProductCreateResult>;
  updateProduct(cred, id, product): Promise<void>;
  updateInventory(cred, requests): Promise<BatchResult>;
  updateSaleStatus(cred, id, status): Promise<void>;
  fetchOrders(cred, cursor): Promise<OrderFetchResult>;
  confirmOrders(cred, nos): Promise<BatchResult>;         // 발주확인
  dispatchOrders(cred, requests): Promise<BatchResult>;   // 배송처리
  fetchClaims(cred, cursor): Promise<ClaimFetchResult>;
  fetchChannelProducts(cred, cursor): Promise<Page<ChannelProductSummary>>; // 역방향 §10
  getChannelProduct(cred, no): Promise<ChannelProductDetail>;               // 역방향 §10
}
```

Capability 코드: `PRODUCT_SEARCH`, `PRODUCT_DETAIL`, `INVENTORY_READ`, `PRICE_READ`, `SALE_APPROVAL`, `ORDER_CREATE`, `ORDER_STATUS_READ`, `SHIPMENT_READ`, `CANCEL_REQUEST`, `RETURN_REQUEST`.

### 8.2 네이버 커머스API 인증 — 전자서명 OAuth2 (필수)

네이버 커머스API는 **OAuth2 client_credentials + 전자서명** 방식이다.

```text
POST {COMMERCE_API_BASE}/external/v1/oauth2/token
form:
  client_id          = 애플리케이션 ID
  timestamp          = 현재시각(ms)
  grant_type         = client_credentials
  client_secret_sign = Base64( bcrypt( `${client_id}_${timestamp}`, salt = client_secret ) )
  type               = SELF     # 자사 스토어. 솔루션형(다수 판매자)은 SELLER + account_id
→ access_token (유효 약 3시간) → 메모리 캐시, 만료 전 갱신
```

- `client_secret` 은 bcrypt 해시 형식 문자열이며 **서명 salt로 그대로 사용**. 저장소에 `bcrypt`(^6) 존재 → 재사용.
- **다수 판매자 처리**(자사형 앱 vs 커머스솔루션형 `type=SELLER`)는 계약 확인 항목(§32 D1)이며 `extra_json` 스키마에 영향.
- 주요 오퍼레이션(버전·경로 재검증): 원상품 등록 `POST /external/v2/products`, 채널상품 조회/수정, 옵션 재고 변경, 주문 조회 `.../pay-order/seller/product-orders`, 발주확인/발송처리, 클레임 조회.

### 8.3 도매꾹·도매매

- Open API(상품 목록/상세/카테고리)로 시작. **재고 수량은 정확히 안 줄 수 있음**(상태형) → §12.2 가상재고 정책을 기본 경로로.
- Private 승인 전엔 주문 생성 불가 → **반자동/수동 발주만 정식 경로**(§13.4).
- MVP Capability: `PRODUCT_SEARCH`·`PRODUCT_DETAIL` 확정, 나머지는 승인에 따라 on/off.

### 8.4 온채널

- 제3자 API 불확실 → **L1(수동·CSV)로 시작**. 어댑터는 CSV 임포트 + 수동 발주 기록만, 나머지 Capability 는 `unsupported`. 제휴 확인 후 L2 승격.

## 9. 상품 소싱·등록 흐름

### 9.1 검색

```text
검색어 입력 → 빌더 Search → 공급처 Adapter 호출 → 공통 상품요약 변환
   → 중복·금지·인증필요 표시 → 판매자에게 결과 제공
```
필터: 위탁배송 가능·국내배송·재고 있음·공급가 범위·배송비 유형·반품지 제공·판매 승인 필요·KC 인증 필요·브랜드 제한·이미지 사용 허용·최소 마진·규제 카테고리(성인/의료/식품 등).

### 9.2 가져오기 → 중간 테이블 적재 (1차)

```text
공급처 상품 선택 → 상세 재조회 → 원본 스냅샷 저장 → supplier_product/variant 적재(중간 테이블)
```
**여기까지가 "가져오기".** `builder_product` 승격은 하지 않는다. 대량으로 긁어와 중간 테이블에 쌓아둔다.

### 9.3 선택 후 상품 등록 (빌더 승격)

```text
중간 테이블 목록에서 선택 → "상품 등록" → builder_product/variant 생성(source_type=SUPPLIER, DRAFT)
   → 옵션 정규화 → 이미지·상세 검증 → 카테고리 추천 → 판매가 계산
```

### 9.4 카테고리 매핑 (네이버 leaf 강제)

공급처 카테고리 ≠ 네이버. **네이버는 최하위(leaf) 카테고리에만 등록 가능** → leaf 아닌 매핑은 등록 차단.

```text
도매꾹/온채널 카테고리 → 빌더 표준 카테고리 → 네이버 leaf 카테고리
```
```text
category_mapping
- supplier_type / supplier_category_code
- internal_category_id / naver_category_id
- confidence / mapping_status / reviewed_by / reviewed_at
```
추천 우선순위: ① 검수 완료 매핑 ② 동일 상품 기존 매핑 ③ 공급처 카테고리 룰 ④ 상품명·속성 자동추천 ⑤ 수동 선택. 초기엔 자동추천만으로 즉시 등록하지 말고 사용자 확인.

### 9.5 상품정보제공고시 — 최상위 병목

**실무에서 등록이 가장 많이 막히는 지점.** 고시항목은 카테고리(상품군)별 필수값이 다르고, 누락 시 등록 실패한다.

```text
naver_category_notice_schema     # 상품군별 필수 고시 정의
- naver_category_id / notice_type / required_fields_json / fetched_at
builder_product_notice           # 상품별 고시값
- builder_product_id / notice_type / values_json / completed
```
등록 전 검증(§22)에 **`고시 필수항목 완료` + `네이버 leaf 매핑 완료`** 를 하드 게이트로 둔다. MVP는 취급 카테고리를 소수 비규제 상품군으로 제한해 고시 관리 부담을 낮춘다.

### 9.6 상품명·콘텐츠 정책

공급처 상품명 복제는 중복상품·검색품질 저하·표현 오류를 부른다. 빌더 편집 항목: 판매용 상품명, 검색 키워드, 브랜드·제조사, 모델명, 원산지, 상세설명, 대표/추가 이미지, 상품정보고시, 인증정보, 배송·반품 정보.

```text
content_usage_policy
- image_download_allowed / image_edit_allowed / detail_edit_allowed
- watermark_required / prohibited_channels
```

### 9.7 판매 승인

일부 상품(온채널 등)은 판매 전 승인 필요. 상태: `NOT_REQUIRED / NOT_REQUESTED / REQUESTED / APPROVED / REJECTED / EXPIRED`. `APPROVED` 아니면 스마트스토어 게시 차단.

### 9.8 스마트스토어 등록 → 검수

등록은 아웃박스 topic `channel.product.publish` 로 비동기 처리. **등록 API 성공 = 즉시 판매가 아니다** — 네이버 검수 후 노출된다. 상태머신(§17.1)의 `PENDING_REVIEW`/`REVIEW_REJECTED` 로 추적하고, 반려 사유는 장애센터에 노출.

## 11. 가격 정책

```text
판매가격 = (공급가 + 공급처 배송비 + 예상 추가비용) ÷ (1 - 목표마진율 - 판매채널수수료율 - 결제수수료율)
```
예: 공급가 10,000 + 배송비 3,000, 채널비용률 8%, 마진율 20% → 13,000 ÷ (1 - 0.28) ≈ 18,056 → 권장 18,900.

```text
pricing_policy
- mall_id / supplier_type / category_id
- margin_type(RATE/FIXED/TIERED) / margin_value / channel_fee_rate / extra_cost
- rounding_unit / minimum_profit / maximum_sale_price
```
공급가 변경 대응: 자동 판매가 변경 / 사용자 승인 후 변경 / 일정 비율 초과 시 판매중지 / 마진이 최소 기준 미만이면 품절 처리.
```text
if expected_profit < minimum_profit: suspend product
else: update sale price
```
> 네이버 가격 수정 허용 범위·검수 정책은 별도 확인.

## 12. 재고 동기화

### 12.1 기준·계산

실제 재고는 공급처 기준. 스마트스토어 전송 재고:
```text
판매가능재고 = max(0, 공급처재고 - 안전재고 - 예약재고)
```
예: 공급처 10, 안전 2, 예약 1 → 노출 7.

### 12.2 재고를 모르는 경우 (가상재고)

공급처가 상태만 줄 때: `IN_STOCK → 가상재고 N개`, `LOW_STOCK → 1개`, `OUT_OF_STOCK → 0 및 판매중지`, `UNKNOWN → 0 또는 검토`. 가상재고를 크게 잡으면 초과판매 위험.

### 12.3 1차 수동 / 2차 배치

- **1차**: `/admin/sourcing/sync` 의 **"재고 가져오기" 버튼**으로 그 시점 조회·계산·(선택 시)네이버 전송. 자동 스케줄 없음.
- **2차**: 워커 폴링 스케줄러가 **1차의 수동 로직(`core/inventory.js`)을 그대로** 주기 호출.

| 상태 | 2차 주기(권장) |
|---|---|
| 판매 중 상품 | 5~15분(적응형) |
| 저재고 상품 | 1~5분 |
| 주문 발생 직후 | 즉시 재조회 |
| 장기 미판매 | 30~60분 |
| 전체 정합성 검증 | 일 1회(cron 셸+one-shot) |

### 12.4 품절 전파

```text
공급처 품절 감지 → builder_variant.available_quantity=0 → 네이버 옵션 재고=0
   → 모든 옵션 품절이면 상품 판매중지 검토
```
부분 옵션 품절과 전체 상품 품절을 분리한다.

```text
inventory_sync_job
- mall_id / supplier_product_id / builder_variant_id
- source_quantity / calculated_quantity / previous_channel_quantity
- status / retry_count / next_retry_at
```

## 13. 주문 수집·발주 (2차)

### 13.1 흐름

```text
스마트스토어 주문 → 네이버 주문 수집 → channel_product_order_no 중복검사 → 빌더 주문 생성
   → [발주확인(SLA) 최우선] → 공급처 재고·공급가 재확인 → 발주 가능성 판단
   → 공급처 주문 생성 → 공급처 결제 → 발주 완료
```

### 13.2 발주확인 SLA

**결제완료(신규주문) 후 발주확인(상품준비중)을 시한 내 안 하면 네이버 자동취소/페널티**가 발생할 수 있다. 따라서 `PAYMENT_CONFIRMED → DISPATCH_ACK`(발주확인) 전이를 **최우선 워커 처리**로 두고, 실제 공급처 발주 성공 여부와 분리해 먼저 발주확인한다.

### 13.3 자동발주 사전검사

다음 중 하나라도 실패하면 자동발주하지 않고 운영자 검토 큐로 보낸다: 상품 매핑 없음, 옵션 매핑 없음, 판매 승인 만료, 공급처 품절, 공급가 급등, 고객 주소 형식 오류, 도서산간/제주 추가배송비 확인 필요, 주문자·수령자 정보 부족, 공급처 계정 잔액 부족, Private API 권한 없음, 금지된 개인정보 전달 조건, 묶음배송 불가.

예약재고: 주문 수집 시 발주 전이라도 `reserved_quantity += order_quantity`. 발주 실패 시 즉시 해제하지 않고 재시도·수동처리 유예 동안 유지.

### 13.4 자동/반자동/수동 발주

- **자동**: 공급처 주문 API + 자동결제 가능 + 매핑 확정 + 주소·배송비 검증 통과.
- **반자동**: 빌더가 발주 데이터 구성 → 판매자가 공급처 화면에서 최종 확인 → 주문번호 회수.
- **수동**: 공급처 API 없음 → 판매자 직접 주문, 빌더는 상태·송장만 기록.
> 온채널 제3자 권한 확인 전, 도매꾹 Private 승인 전에는 반자동/수동을 정식 경로로 유지.

### 13.5 발주 자금 흐름

```text
발주 준비(READY) → 결제수단 확인
  ├─ 예치금/포인트 잔액 ≥ 공급가+배송비 → 자동결제 → PAID → 공급처 접수(ACCEPTED)
  │      └─ 실패(잔액부족/한도) → HELD_FUNDS → 운영자 알림, 예약재고 유지
  └─ 자동결제 미지원(수동) → 판매자 결제 후 주문번호 회수(반자동)
```
- `HELD_FUNDS` 는 즉시 취소하지 않고 충전·재시도 유예(예: 30분) 후에도 미해결이면 운영자 큐 + 고객 대응 판단.
- **정산 대사**: `procurement_order`(공급처 결제) ↔ `sales_order`(채널 정산) 대사 배치로 역마진·누락 탐지(§20.4 리포트 편입).
- 자동결제/예치금 API 지원 여부는 계약 확인(§32 D1). 미확인 구간은 전부 반자동으로 강등.

## 14. 개인정보·법적 지위

- **지위**: 빌더 운영자는 통상 통신판매중개자, 몰(판매자)은 통신판매업자. 위탁배송을 위해 구매자 배송정보를 공급사에 처리위탁/제3자 제공한다 — 계약서·개인정보처리방침에 명시(산출물).
- **전달 항목**: 수령인, 연락처, 우편번호, 주소, 배송메시지, 주문상품·수량.
- **구현 원칙**:
  - 저장: `sales_order` 의 수령인·연락처·주소는 `*_encrypted`(`shared/crypto`). 로그 원문 금지.
  - 전달 시점: **발주 아웃박스 처리 시점에만** 복호화해 공급처로 최소 전달. 그 외 화면·API 는 마스킹.
  - 접근: 관리자 조회는 RBAC(`requireMenuAccess`) + 감사로그(`operator_audit`).
  - 보존: 배송 완료/클레임 종료 + 보존기간 경과 후 비식별화/삭제 배치.

## 18. 비동기 처리·아웃박스·멱등성

외부 API는 사용자 HTTP 요청 안에서 직접 완료하지 않는다. 상태 변경과 **같은 트랜잭션**으로 아웃박스에 적재하고 워커가 소비한다.

```sql
integration_outbox
- id BIGINT PK, mall_id BIGINT
- topic VARCHAR(60)              -- 예: channel.product.publish
- idempotency_key VARCHAR(200)  -- UNIQUE
- payload_json JSON
- status ENUM('PENDING','PROCESSING','DONE','FAILED','DLQ')
- attempt INT / next_retry_at DATETIME / last_error TEXT
- locked_by / locked_at
- created_at / updated_at
- UNIQUE(idempotency_key), INDEX(status, next_retry_at)
```

**워커 루프(worker.js)**:
```text
loop(2~5s):
  rows = SELECT ... WHERE status IN (PENDING,FAILED) AND next_retry_at<=now
         ORDER BY id LIMIT N FOR UPDATE SKIP LOCKED
  for row: PROCESSING → TOPIC_HANDLERS[topic](payload)
      성공 → DONE
      실패 → attempt++; (>=MAX → DLQ+알림) else FAILED, next_retry_at=backoff
```

이벤트(topic) 예: `supplier.product.import.requested`, `channel.product.publish.requested`, `supplier.inventory.changed`, `channel.inventory.update.requested`, `channel.order.collected`, `supplier.procurement.create.requested`, `supplier.shipment.detected`, `channel.dispatch.requested`, `channel.claim.detected`.

**멱등키 규칙**:
```text
상품등록:  mall_id + channel_account + builder_product_id + content_version
재고변경:  mall_id + channel_variant_mapping_id + inventory_version
주문수집:  channel + channel_product_order_no
발주:      supplier + sales_order_item_id
```

폴링(2차): 웹훅보다 **변경분 폴링** 중심(네이버 주문/변경, 공급처 재고). 주문·송장은 낮은 지연 우선순위 topic 그룹. `FOR UPDATE SKIP LOCKED` 로 향후 워커 다중화 대비.

## 20. 정합성 검증

API 성공 응답만으로 동기화 완료로 보지 않는다.

- **상품**: 빌더 상태 vs 네이버 상태, 옵션 수 비교, 매핑 누락, 이미지·카테고리·고시 변경.

- 마이그레이션: `scripts/migrate_sourcing_*.sql`(멱등 + `admin_menus` INSERT). 개발 DB 공용 1벌이므로 신중히.
- 시크릿: 자격증명 `ENC:`+`shared/crypto`. `ENCRYPTION_KEY`(/etc/environment) 없으면 앱 기동 불가.
- 초기엔 모듈러 모놀리스 + 단일 워커, 성장 시 워커 분리.

## 26. 개발 순서

1. **API 계약 확인** — 도매꾹 Open 발급, Private 범위, 온채널 제휴 여부, 네이버 앱 유형, 개인정보 처리위탁 검토.
2. **내부 표준 모델** — 상품/옵션/공급처 매핑/스마트스토어 매핑/주문/발주/송장/클레임.
3. **Mock Adapter** — 도매꾹/온채널/네이버 Mock(품절·가격변경·토큰만료·주문중복·발주실패·송장지연).
4. **상품 가져오기·등록** — 중간 테이블 적재 → 선택 등록 → 검증 → 카테고리 매핑 → 가격 → 네이버 등록 → 옵션 매핑 → 검수.
5. **역방향 수집 + 재고·주문 수동 가져오기(1차 완결)**.
6. **주문·발주(2차)** — 주문 수집 → 발주확인 → 예약재고 → 발주 사전검사 → 공급처 주문 → 송장 반영.
7. **운영 안정화** — 재시도·DLQ·정합성 배치·재처리·감사·개인정보 보존.

## 29. 권장 구조

- **권장안 A (완전 통합)**: 도매꾹 Private + 온채널 제휴 + 네이버 권한 확보 시. 빌더가 상품·재고·주문·발주·송장 통합.

- **권장안 C (공급처 자체 연동 병행)**: 빠르나 주문 단일 기준이 깨질 수 있어 장기 구조로는 비권장.

## 31. 최종 제안

```text
1. 공급처와 판매채널을 Adapter로 완전히 분리(자격증명 주입형)
2. 빌더 내부 표준 상품·옵션·주문 모델 구축(mall_id 스코프)
3. 도매꾹 Open API로 상품 조회, 온채널 L1(CSV)로 시작
4. 네이버 커머스API로 상품 등록·재고·주문·배송, 전자서명 인증
5. 2단계 적재(가져오기 ≠ 등록) + 역방향 수집 지원
6. 공급처 재고 기준 + 안전재고, 주문 직전 재검증
7. 모든 외부 호출을 MySQL 아웃박스 + PM2 워커로 비동기
8. 검수·발주확인·고시·자금흐름을 상태·게이트로 명시
9. 1차=수동 "가져오기" 버튼, 2차=동일 로직 배치 자동화
10. 멱등성·정합성 검증을 주문·발주·송장에 적용
```
이 구조면 향후 쿠팡·11번가·오너클랜 등을 핵심 도메인 변경 없이 Adapter 확장만으로 붙일 수 있다.
