## 1. 목적

몰(판매자)이 도매꾹·도매매 또는 온채널에서 위탁판매 가능한 상품을 선택해, 빌더에서 상품정보를 편집하고 네이버 스마트스토어에 등록·판매하는 통합 시스템을 설계한다. 아울러 스마트스토어에 **직접 등록된 상품을 우리 몰로 가져오는 역방향 흐름**도 지원한다.

목표 사용자 흐름:

```text
공급처 상품 검색 → 가져오기(중간 테이블 적재) → 선택 후 상품 등록
   → 상품명·가격·이미지·옵션·고시정보 편집 → 스마트스토어 등록(검수)
   → 재고·가격 동기화 → 주문 수집 → 공급처 발주 → 송장 수집 → 배송 처리
   → 취소·교환·반품 및 정산 관리
역방향:  스마트스토어 직접등록 상품 → 우리 몰로 수집 → 통합 관리
```

핵심 원칙: **상품정보는 빌더가 표준화하고, 공급 재고는 공급처가 기준이며, 판매 주문은 스마트스토어가 발생원**이다.

## 2. 제품·운영 전제 (yd-mall 반영)

- **몰 빌더 성격**: yd-mall 은 특정 몰을 운영하는 앱이 아니라 몰을 찍어내는 빌더다. 이 연동은 **개별 몰(스토어)에 붙는 기능**이다.
- **몰당 1:1 연동**: "위탁판매 판매자 = 몰 1개"로 본다. 한 몰이 자신의 도매꾹/온채널/스마트스토어 계정을 각 1세트 연결한다. (한 몰에 다계정 요건이 생기면 `channel_account` 1:N 승격.)
- **사용 여부 = 몰별 설정 + 유료 기능**: 관리자에서 몰마다 이 연동의 사용 여부(on/off)를 켠다. **납품 시 이 기능 사용은 추가 계약 대상**이므로, 플래그가 꺼진 몰에서는 대메뉴·기능·워커 처리가 전부 비활성이어야 한다.
- **릴리스 차수**:
  | 차수 | 범위 |
  |---|---|
  | **1차** | 상품 가져오기 → 중간 테이블 → 선택 등록 → 스마트스토어 등록 + 역방향 수집 + **재고·주문 "가져오기" 버튼(수동)**. 배치 없음 |
  | **2차** | 재고·주문 자동(배치) 동기화, (반)자동 발주, 송장·클레임 |
  1차의 수동 "가져오기" 로직을 **2차 배치가 그대로 재사용**(자동 트리거만 추가).
- **개발 환경 전제**: 이 프로젝트는 전 과정 개발 단계다. `main` 푸시 = 개발 서버 즉시 반영. DB 는 로컬·서버 공용 1벌.

## 3. 공식 연동 범위와 전제조건

### 3.1 네이버 스마트스토어 (커머스API)

네이버 커머스API로 상품·주문·정산·판매자정보를 다룬다. 빌더가 처리할 주요 기능: 상품 등록, 원상품·채널상품 조회·수정, 판매상태 변경, 옵션별 재고 변경, 주문 조회, 발주확인, 배송 처리, 취소·반품·교환 처리, 판매자 인증·권한.

필수 전제: 스마트스토어 판매자 계정, 커머스API센터 애플리케이션, 판매자별 인증·접근 권한, 스토어·채널 정보, 호출 제한·오류 정책 대응. **인증 방식(전자서명 OAuth)의 실제 구현은 [`네이버_스마트스토어_연동.md`](./네이버_스마트스토어_연동.md) §2 참고.**

공식 문서: https://apicenter.commerce.naver.com/docs/introduction , https://apicenter.commerce.naver.com/docs/commerce-api/current

### 3.2 도매꾹·도매매

- **Open API**: 상품 목록/상세/카테고리 조회 중심. REST 파라미터(`ver`, `mode`(getItemList·getItemView 등), `aid`(API Key)) 형태로 JSON/XML 응답 — 정확한 필드는 최신 문서로 매핑.
- **Private API(2026-03 승인제)**: 로그인·주문·구매·상품등록 등 회원권한 기능은 승인 필요. 승인 전엔 주문 생성 불가일 수 있음.
- 전제: 사용자별 계정 연결, Open API Key, (필요 시) Private 승인, 위탁배송 가능 여부 확인.

공식 문서: https://openapi.domeggook.com/ , https://domemedb.domeggook.com/index

### 3.3 온채널

공식 안내는 주로 **온채널 자체 서비스가 판매자 스마트스토어와 연결**되는 구조다. 제3자 빌더가 온채널 상품·재고·발주 API를 직접 쓸 수 있는지는 공개 문서로 확정 불가. 따라서 세 수준을 지원 설계한다.

| 수준 | 방식 | 빌더 역할 |
|---|---|---|
| **L1** | 상품정보 수동·CSV 가져오기 | 상품 편집 및 스마트스토어 등록 (**MVP 시작점**) |
| L2 | 온채널 제휴 API | 상품·재고·주문·송장 자동화 |
| L3 | 온채널 자체 스마트스토어 연동 병행 | 빌더는 상품 관리 중심 |

공식 안내: https://onchannelnotice.oopy.io/guide/api , https://www.onch3.co.kr/seller.html

## 10. 역방향 수집 — 스마트스토어 상품 → 우리 몰

빌더를 거치지 않고 판매자가 스마트스토어에 **직접 등록한 상품**도 우리 몰로 가져와 통합 관리한다.

```text
네이버 채널상품 조회(fetchChannelProducts/getChannelProduct)
   → builder_product(source_type=CHANNEL_IMPORT) 생성 → 옵션 정규화
   → channel_product_mapping(naver_channel_product_no) 생성
   → 이후 재고·주문 동기화 대상에 포함(2차)
```
- **중복 방지**: `naver_channel_product_no` 로 기존 매핑 있으면 갱신.
- **공급처 없음**: `source_supplier_product_id` 가 비므로 **자동발주 대상 아님**(재고·주문 동기화만, 발주는 판매자 자체 처리).

## 11. 가격 정책

```text
판매가격 = (공급가 + 공급처 배송비 + 예상 추가비용) ÷ (1 - 목표마진율 - 판매채널수수료율 - 결제수수료율)
```
예: 공급가 10,000 + 배송비 3,000, 채널비용률 8%, 마진율 20% → 13,000 ÷ (1 - 0.28) ≈ 18,056 → 권장 18,900.

```text
pricing_policy
- mall_id / supplier_type / category_id
- margin_type(RATE/FIXED/TIERED) / margin_value / channel_fee_rate / extra_cost
- rounding_unit / minimum_profit / maximum_sale_price
```
공급가 변경 대응: 자동 판매가 변경 / 사용자 승인 후 변경 / 일정 비율 초과 시 판매중지 / 마진이 최소 기준 미만이면 품절 처리.
```text
if expected_profit < minimum_profit: suspend product
else: update sale price
```
> 네이버 가격 수정 허용 범위·검수 정책은 별도 확인.

## 12. 재고 동기화

### 12.1 기준·계산

실제 재고는 공급처 기준. 스마트스토어 전송 재고:
```text
판매가능재고 = max(0, 공급처재고 - 안전재고 - 예약재고)
```
예: 공급처 10, 안전 2, 예약 1 → 노출 7.

### 12.2 재고를 모르는 경우 (가상재고)

공급처가 상태만 줄 때: `IN_STOCK → 가상재고 N개`, `LOW_STOCK → 1개`, `OUT_OF_STOCK → 0 및 판매중지`, `UNKNOWN → 0 또는 검토`. 가상재고를 크게 잡으면 초과판매 위험.

### 12.3 1차 수동 / 2차 배치

- **1차**: `/admin/sourcing/sync` 의 **"재고 가져오기" 버튼**으로 그 시점 조회·계산·(선택 시)네이버 전송. 자동 스케줄 없음.
- **2차**: 워커 폴링 스케줄러가 **1차의 수동 로직(`core/inventory.js`)을 그대로** 주기 호출.

| 상태 | 2차 주기(권장) |
|---|---|
| 판매 중 상품 | 5~15분(적응형) |
| 저재고 상품 | 1~5분 |
| 주문 발생 직후 | 즉시 재조회 |
| 장기 미판매 | 30~60분 |
| 전체 정합성 검증 | 일 1회(cron 셸+one-shot) |

### 12.4 품절 전파

```text
공급처 품절 감지 → builder_variant.available_quantity=0 → 네이버 옵션 재고=0
   → 모든 옵션 품절이면 상품 판매중지 검토
```
부분 옵션 품절과 전체 상품 품절을 분리한다.

```text
inventory_sync_job
- mall_id / supplier_product_id / builder_variant_id
- source_quantity / calculated_quantity / previous_channel_quantity
- status / retry_count / next_retry_at
```

## 13. 주문 수집·발주 (2차)

### 13.1 흐름

```text
스마트스토어 주문 → 네이버 주문 수집 → channel_product_order_no 중복검사 → 빌더 주문 생성
   → [발주확인(SLA) 최우선] → 공급처 재고·공급가 재확인 → 발주 가능성 판단
   → 공급처 주문 생성 → 공급처 결제 → 발주 완료
```

### 13.2 발주확인 SLA

**결제완료(신규주문) 후 발주확인(상품준비중)을 시한 내 안 하면 네이버 자동취소/페널티**가 발생할 수 있다. 따라서 `PAYMENT_CONFIRMED → DISPATCH_ACK`(발주확인) 전이를 **최우선 워커 처리**로 두고, 실제 공급처 발주 성공 여부와 분리해 먼저 발주확인한다.

### 13.3 자동발주 사전검사

다음 중 하나라도 실패하면 자동발주하지 않고 운영자 검토 큐로 보낸다: 상품 매핑 없음, 옵션 매핑 없음, 판매 승인 만료, 공급처 품절, 공급가 급등, 고객 주소 형식 오류, 도서산간/제주 추가배송비 확인 필요, 주문자·수령자 정보 부족, 공급처 계정 잔액 부족, Private API 권한 없음, 금지된 개인정보 전달 조건, 묶음배송 불가.

예약재고: 주문 수집 시 발주 전이라도 `reserved_quantity += order_quantity`. 발주 실패 시 즉시 해제하지 않고 재시도·수동처리 유예 동안 유지.

### 13.4 자동/반자동/수동 발주

- **자동**: 공급처 주문 API + 자동결제 가능 + 매핑 확정 + 주소·배송비 검증 통과.
- **반자동**: 빌더가 발주 데이터 구성 → 판매자가 공급처 화면에서 최종 확인 → 주문번호 회수.
- **수동**: 공급처 API 없음 → 판매자 직접 주문, 빌더는 상태·송장만 기록.
> 온채널 제3자 권한 확인 전, 도매꾹 Private 승인 전에는 반자동/수동을 정식 경로로 유지.

### 13.5 발주 자금 흐름

```text
발주 준비(READY) → 결제수단 확인
  ├─ 예치금/포인트 잔액 ≥ 공급가+배송비 → 자동결제 → PAID → 공급처 접수(ACCEPTED)
  │      └─ 실패(잔액부족/한도) → HELD_FUNDS → 운영자 알림, 예약재고 유지
  └─ 자동결제 미지원(수동) → 판매자 결제 후 주문번호 회수(반자동)
```
- `HELD_FUNDS` 는 즉시 취소하지 않고 충전·재시도 유예(예: 30분) 후에도 미해결이면 운영자 큐 + 고객 대응 판단.
- **정산 대사**: `procurement_order`(공급처 결제) ↔ `sales_order`(채널 정산) 대사 배치로 역마진·누락 탐지(§20.4 리포트 편입).
- 자동결제/예치금 API 지원 여부는 **계약 확인 대상**이다(결정 게이트 D1 — [`개발계획서`](./도매꾹_온채널_스마트스토어_연동_개발계획서.md)). 미확인 구간은 전부 반자동으로 강등.

## 14. 개인정보·법적 지위

- **지위**: 빌더 운영자는 통상 통신판매중개자, 몰(판매자)은 통신판매업자. 위탁배송을 위해 구매자 배송정보를 공급사에 처리위탁/제3자 제공한다 — 계약서·개인정보처리방침에 명시(산출물).
- **전달 항목**: 수령인, 연락처, 우편번호, 주소, 배송메시지, 주문상품·수량.
- **구현 원칙**:
  - 저장: `sales_order` 의 수령인·연락처·주소는 `*_encrypted`(`shared/crypto`). 로그 원문 금지.
  - 전달 시점: **발주 아웃박스 처리 시점에만** 복호화해 공급처로 최소 전달. 그 외 화면·API 는 마스킹.
  - 접근: 관리자 조회는 RBAC(`requireMenuAccess`) + 감사로그(`operator_audit`).
  - 보존: 배송 완료/클레임 종료 + 보존기간 경과 후 비식별화/삭제 배치.

## 16. 취소·교환·반품

### 16.1 취소

### 16.3 클레임 상태
```text
REQUESTED → SUPPLIER_CONFIRMING → APPROVED/REJECTED → COLLECTING → RECEIVED → REFUND_READY → COMPLETED / FAILED
```
네이버 상태와 공급처 상태를 별도 저장하고 빌더 상태로 매핑.

## 19. 오류 처리

## 20. 정합성 검증

API 성공 응답만으로 동기화 완료로 보지 않는다.

## 부록 B. 주문 자동발주 판단 예시

```pseudo
function canAutoProcure(orderItem):
    mapping = findSupplierMapping(orderItem.builderVariantId)
    if mapping is null: return MANUAL_REVIEW("공급처 옵션 매핑 없음")
    product = supplier.getProduct(cred, mapping.supplierProductNo)
    if product.saleApprovalRequired and not product.approved: return MANUAL_REVIEW("판매 승인 없음")
    inventory = supplier.getInventory(cred, mapping.supplierVariantNo)
    if inventory.quantity < orderItem.quantity: return REJECT("공급처 품절")
    currentPrice = supplier.getPrice(cred, mapping.supplierVariantNo)
    if currentPrice > orderItem.supplyPriceSnapshot * 1.05: return MANUAL_REVIEW("공급가 5% 초과 상승")
    if not validateAddress(orderItem.order.shippingAddress): return MANUAL_REVIEW("배송지 검증 실패")
    return AUTO_PROCURE
```
