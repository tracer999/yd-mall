## 1. 방향 정의 (가장 먼저 확인할 것)

이 저장소에는 "등록"이라는 단어가 **반대 방향 두 곳**에 쓰인다. 혼동하면 엉뚱한 데이터를 만든다.

| 화면 | 경로 | 방향 | 서비스 |
|---|---|---|---|
| 가져온 상품(중간) 의 [등록] | `/admin/sourcing/staging/publish` | 공급처(도매꾹) → **우리 몰** | `services/sourcing/publishService.js` |
| **스마트스토어 등록** | `/admin/sourcing/publish` | **우리 몰 → 네이버** | `services/sourcing/channel/naverPublishService.js` |
| 스토어 상품 가져오기 | `/admin/sourcing/channel-import` | 네이버 → 우리 몰 (역방향) | `services/sourcing/channel/naverChannelImport.js` |
| 재고 연동 | `/admin/sourcing/sync` | **우리 몰 → 네이버**(전송) + 네이버 조회(대사) | `services/sourcing/channel/naverStockSync.js` |

**스마트스토어 등록 = 우리 상품을 네이버에 올리는 것**이다. 화면 문구에는 항상 방향을 함께 적는다.

### 연결 범위

네이버 커머스 API 는 **판매자 단위**로 묶인다. `client_id`/`client_secret` 은 그 판매자가 커머스API센터에서 직접 발급한 것이고, 토큰의 권한은 **발급 계정의 스토어 하나**로 한정된다. 따라서 **연결된 자기 스토어의 상품만** 주고받을 수 있으며, 다른 판매자의 스토어는 조회도 등록도 불가능하다.

## 2. 인증

```
POST https://api.commerce.naver.com/external/v1/oauth2/token
Content-Type: application/x-www-form-urlencoded

client_id, timestamp(ms 13자리), grant_type=client_credentials,
client_secret_sign, type=SELF
```

전자서명(특이 방식 — `client_secret` 을 **bcrypt salt 로 그대로 사용**):

```
password  = `${client_id}_${timestamp}`
signature = Base64( bcrypt(password, salt = client_secret) )
```

- 토큰 유효기간 **3시간(`expires_in` = 10800)**. 응답값을 그대로 읽어 쓰고 하드코딩하지 않는다.
- 만료 1분 전까지 메모리 캐시 재사용(`naverClient._tokenCache`). DB 에 저장하지 않는다.
- **401 `GW.AUTHN`** 수신 시 캐시를 버리고 **1회만** 재발급 후 원 요청 재시도.

구현: `services/sourcing/channel/naverClient.js`

### `type=SELF` vs `SELLER` — 미결정 항목

현재 코드는 `SELF`(자사 스토어) 고정이다. 몰 빌더로 여러 판매자에게 납품하면 **판매자마다 각자 커머스API센터에 앱을 등록**해야 한다. 커머스솔루션형(`type=SELLER` + `account_id`)으로 가면 우리가 앱 하나를 만들고 판매자가 권한을 위임하는 형태가 된다.

> 이 선택은 `mall_channel_credential.extra_json` 스키마와 모든 호출부에 영향을 준다. **계약 확인 후 확정할 것.** 나중에 바꾸면 자격증명 저장 구조부터 손봐야 한다.

## 3. 호출 제한과 배치 전략 ★

대량 등록 설계의 근거가 되는 절이다. **수치를 임의로 바꾸지 말 것.**

### 3.1 공식 제한

- Token Bucket 방식. **"내 스토어" 애플리케이션은 API 엔드포인트별 초당 2회(2 RPS) 고정**이며 사용자가 늘릴 수 없다.
- 엔드포인트별로 버킷이 **분리**된다(상품등록 2 RPS + 이미지업로드 2 RPS는 별개).
- Burst 로 순간 4 RPS 가 가능하나 **연속 사용 불가** → 가용 자원으로 계산하지 않는다.
- **분당·일일 한도는 문서에 없다.** 초당 제한만 존재.
- **`Retry-After` 헤더를 주지 않는다.** 공식 권고는 "3~5초 후 재호출"이라는 수동 딜레이뿐.
- 429 판별 헤더: `GNCP-GW-RateLimit-Remaining` / `-Replenish-Rate` / `-Burst-Capacity`
  - `Remaining > 0` 인데 429 → **서비스 혼잡**(2024-04-01부터 발생)
  - `Remaining = 0` → 쿼터 소진
  - ⚠ **헤더가 아예 누락되는 경우가 있다** → 부재 시 혼잡으로 간주하고 더 길게 대기.

### 3.2 벌크 등록 API 는 없다

네이버 공식 답변: 다수 상품을 한 번에 등록하는 **벌크 API 는 없으며 제공 계획도 없다.** 유상 호출량 증대도 미제공.

→ **"대량 등록"은 서버가 단건 POST 를 순차 반복하는 것**이다. 화면에도 그렇게 안내한다.

부분적 벌크가 존재하는 곳은 두 군데뿐:
- 이미지 업로드: 1요청 **최대 10장**
- 가격 수정: 상대적 변경(n원/n% 증감)만 가능. 절대값 지정 불가.

### 3.3 이미지 업로드의 동시성 1 제약 ★★

> "상품 이미지 업로드는 **스토어 계정 단위로 한 번에 1개 요청건만** 수용한다. 2건 이상 필요 시 반드시 이전 호출의 응답 반환 이후 다음 호출을 요청하도록 동기적으로 설계해야 한다." (공식)

위반 시 `"이전 요청이 진행중입니다"` 로 실패한다. 이는 rate limit 이 아니라 **락 성격의 제약**이라 재시도로 우회할 수 없다.

상품 등록 파이프라인은 이미지 업로드를 반드시 포함하므로 **전체 파이프라인이 직렬**이다. 동시성을 올리는 최적화는 불가능하다.

구현: `naverImages.js` 의 `withAccountLock()` (clientId 단위 Promise 체인)

### 3.4 채택 수치

| 항목 | 값 | 근거 |
|---|---|---|
| 스로틀 간격 | **600ms** 직렬 (≈1.67 RPS) | 공식 2 RPS(=500ms) + 20% 안전마진 |
| 동시성 | **1 (고정)** | 이미지 업로드 계정당 1요청 강제 |
| **1회 배치 상한** | **30건** | 건당 2~4초 → 최대 약 2분. 관리자 요청 타임아웃 내 |
| 이미지 묶음 | 10장 / 8MB | 공식 10장·10MB 미만에서 여유 |
| 429 재시도 | 최대 5회, **3→6→12→24→48초** + jitter ±20% | Retry-After 부재, 공식 권고 3~5초 |
| 적응형 백오프 | 429 연속 3회 → 간격 1.5배, 성공 10건 후 원복 | 자체 판단 필요 |
| 401 `GW.AUTHN` | 토큰 재발급 1회 + 원요청 1회 | |
| **4xx 검증 오류** | **재시도 0회**, 격리 후 사유 기록 | 고쳐지지 않는 요청 반복은 한도 낭비 |

> ⚠️ **도매꾹의 350ms 를 그대로 가져오면 안 된다.** 350ms ≈ 2.8 RPS 로 네이버 한도를 초과해 즉시 429 가 난다.

> ⚠️ **PM2 는 반드시 fork·`instances: 1` 을 유지해야 한다.** 스로틀이 프로세스 내 상태라, cluster 로 늘리면 프로세스마다 독립 스로틀이 되어 rate limit 이 즉시 깨진다. 확장하려면 Redis 기반 분산 토큰버킷이 필요하다.

## 4. 엔드포인트 — **버전이 섞여 있다** ★

이 연동에서 가장 자주 실수하는 지점이다.

| 기능 | 메서드 · 경로 | 버전 |
|---|---|---|
| 상품 등록 | `POST /v2/products` | **v2** |
| **판매자 상품 목록 조회** | `POST /v1/products/search` | **v1** |
| 원상품 조회·수정·삭제 | `GET|PUT|DELETE /v2/products/origin-products/{originProductNo}` | **v2** |
| 채널상품 조회·수정·삭제 | `GET|PUT|DELETE /v2/products/channel-products/{channelProductNo}` | **v2** |
| **이미지 업로드** | `POST /v1/product-images/upload` | **v1** |
| **옵션별 재고·가격 변경** | `PUT /v1/products/origin-products/{no}/option-stock` | **v1** |
| **판매상태 변경** | `PUT /v1/products/origin-products/{no}/change-status` | **v1** |
| 고시 상품군 목록·스키마 | `GET /v1/products-for-provided-notice[/{type}]` | **v1** |
| 카테고리 | `GET /v1/categories?last=true` | **v1** |
| 검수(수정요청) 목록 | `GET /v1/product-inspections/channel-products` | **v1** |

v1 으로 상품을 부르면 **308 리다이렉트** + "지원하지 않는 API Version" 이 뜬다.

구현: `services/sourcing/channel/naverProducts.js`

### 원상품번호 vs 채널상품번호

`originProductNo` 와 `smartstoreChannelProductNo` 는 **완전히 다른 번호**다.
- 스마트스토어 URL 에 노출되는 것은 **채널상품번호**
- 수정·재고·상태변경 API 는 **원상품번호**를 쓴다
- 어떤 상품의 원상품번호가 다른 상품의 채널상품번호와 **우연히 같을 수 있다** → 혼용하면 조용히 엉뚱한 상품을 건드린다.

둘 다 `channel_product_mapping` 에 각각 보관한다.

## 5. 이미지 업로드

```
POST /v1/product-images/upload
Content-Type: multipart/form-data
필드명: imageFiles   ← 다건이어도 전부 같은 이름. 다르면 400
```

| 제약 | 값 |
|---|---|
| 1회 최대 | **10장** |
| payload 합계 | **10MB 미만** (우리는 8MB 에서 끊는다) |
| 허용 포맷 | **JPG · GIF · PNG · BMP** |
| 동시 호출 | **계정당 1건** |
| 응답 | `{ "images": [ { "url": "http://shop1.phinf.naver.net/..." } ] }` |

### 우리 쪽 대응

- **외부 URL 직접 입력은 거부된다.** 반드시 업로드 API 가 돌려준 URL 만 상품 등록에 쓸 수 있다.
- **★ 우리 상품 이미지는 대부분 외부 URL 이다.** mall 2 기준 9,680건 중 **9,677건이 공급처 호스트의 `https://…` URL**이고 로컬 `/uploads/` 는 3건뿐이다(소싱으로 들어온 상품). 초기 구현은 로컬 파일만 읽어서 **사실상 전 상품이 등록 불가**였다(`ENOENT … /public/https:/…`). 지금은 `prepareFile()` 이 외부 URL 을 **내려받아** 같은 파이프라인에 태운다(타임아웃 15초, 원본 12MB 상한, UA 헤더 — 일부 CDN 이 UA 없으면 403).
- **우리 업로드는 webp 라 그대로는 못 올린다.** `sharp` 로 jpeg 재인코딩한다(알파 채널은 흰 배경으로 flatten — 안 하면 검게 뭉개진다). 원본 파일은 건드리지 않는다.
- Content-Type 은 확장자가 아니라 **실제 바이너리 포맷**과 일치해야 한다. 외부 URL 은 확장자가 없거나 실제 포맷과 다른 경우가 흔해 `sharp().metadata()` 로 **실제 포맷을 판정**한다(확장자·응답 헤더를 믿지 않는다).
- 캐시(`channel_image_cache.local_path`)에는 외부 URL 도 그대로 키로 들어간다.
- 같은 이미지를 상품마다 다시 올리지 않도록 `channel_image_cache` 에 로컬경로 → 네이버 URL 을 캐시한다.
- **상세 HTML(`detailContent`) 안의 `/uploads/...` 상대경로도 치환**한다. 그대로 두면 네이버에서 전부 깨진다.
- 응답 개수가 요청 개수와 다르면 **순서를 신뢰할 수 없으므로 중단**한다(대표이미지가 뒤바뀌는 사고 방지).

구현: `services/sourcing/channel/naverImages.js`

## 6. 상품정보제공고시 (productInfoProvidedNotice)

### 6.1 규칙

- `productInfoProvidedNoticeType` (UPPER_SNAKE, 40종) + **해당 하위 객체 1개만** 넣는다.
  문서의 모든 상품군 객체를 전부 넣으면 400 이 난다.
- 하위 키는 camelCase 로 이름이 다르다 (`DIET_FOOD` → `dietFood`).
- 카테고리를 먼저 확정한 뒤 `GET /v1/products-for-provided-notice?categoryId=` 로 맞는 상품군·필드 스키마를 조회하는 것이 정석.
- 판매자센터의 "정보고시 템플릿 번호"는 **API 미지원** — 매 상품 인라인으로 채워야 한다.

### 6.2 건강기능식품 ★

> **`HEALTH_FUNCTIONAL_FOOD` 라는 유형은 존재하지 않는다. 건강기능식품은 `DIET_FOOD` 다.**
> 이 저장소의 표준 예시몰이 건강식품몰이라 특히 자주 걸린다.

`DIET_FOOD` 필드 **19개** (2026-07-21 실호출 확인):

`productName, producer, location, expirationDate(date), expirationDateText, consumptionDate(date), consumptionDateText, storageMethod, weight, amount, ingredients, nutritionFacts, specification, cautionAndSideEffect, nonMedicinalUsesMessage, geneticallyModified(bool), importDeclarationCheck(bool), consumerSafetyCaution, customerServicePhoneNumber`

> ⚠ **정정** — 초기 문서에 적혀 있던 `foodType` 은 `DIET_FOOD` 에 **없다**(`GENERAL_FOOD` 에 있다). 반대로 유통기한·소비기한 4필드가 빠져 있었다. 스펙 추정으로 적은 목록이었고 실호출로 교정했다.
>
> ⚠ **네이버는 "필수 여부"를 알려주지 않는다.** 응답 필드는 `fieldType · fieldName · fieldDescription · fieldAddDescription · fieldMaxLength` 뿐이다. 어떤 항목이 진짜 필수인지는 **등록 400 응답으로만** 알 수 있다. 카탈로그의 `required` 는 전부 `false` 이며 **지어내지 말 것.**

### 6.3 공통 5필드

모든 상품군이 공유하며, 비우면 네이버가 "상품상세 참조"로 저장한다. 우리도 같은 문구를 기본값으로 채워 400 을 피한다.

`returnCostReason, noRefundReason, qualityAssuranceStandard, compensationProcedure, troubleShootingContents`

### 6.4 입력 UI — **관리자에게 JSON 을 적게 하지 않는다** ★

초기 구현은 프로필 화면에 `notice_defaults_json` textarea 를 두고 관리자가 JSON 을 손으로 적게 했다. **이건 운영자가 할 수 있는 입력이 아니다.** 상품군마다 필드가 40종으로 다르고, 오타 하나가 400 으로 돌아온다.

지금은 **상품군별 필드 카탈로그로 일반 입력폼을 생성**한다.

```
naver_notice_schema (상품군 → 필드 목록)
        ↓
[상품정보제공고시 유형] select 선택
        ↓
필드 목록을 순회해 input/textarea/예·아니오 select 를 그림 (name="notice[필드명]")
        ↓
서버(naverProfile.saveProfile)가 스키마에 있는 키만 남겨 JSON 으로 조립 → 저장
```

- **JSON 은 사람이 아니라 코드가 만든다.** 화면에 JSON 입력란은 없다.
- 스키마에 없는 키는 서버가 버린다(폼 조작으로 엉뚱한 필드가 섞이는 것 방지).
- boolean 필드는 네이버가 진짜 `true/false` 를 요구하므로 `'1'/'0'` 을 변환해 저장한다.
- 공통 5필드는 상품군과 무관하게 항상 같이 받고, 비면 매퍼가 "상품상세 참조"로 채운다.

**카탈로그는 시드로 배포된다.** `migrate_naver_notice_schema.sql` 에 **36종 411필드 전체**가 실응답 기준으로 들어 있다(`source='SEED'`). 자격증명이 없는 새 몰에서도 폼이 바로 동작해야 하기 때문이다. 최신화가 필요하면 화면의 **[네이버에서 상품군 수집]** 버튼이 같은 API 로 덮어쓴다(`source='NAVER'`, 수집분은 시드로 되돌아가지 않음).

**★ 수집은 API 호출 1회면 끝난다.** `GET /v1/products-for-provided-notice` 응답에 36종 전체의 필드 목록(`productInfoProvidedNoticeContents`)이 **이미 들어 있다.** 상품군마다 `/{type}` 을 다시 부를 필요가 없다(초기 구현은 37회를 불렀다). 호출이 1회뿐이라 §3.4 의 스로틀·백오프도 사실상 무의미하다.

실응답 shape:

```json
{ "productInfoProvidedNoticeType": "DIET_FOOD",
  "productInfoProvidedNoticeTypeName": "건강기능식품",
  "productInfoProvidedNoticeContents": [
    { "fieldType": "String", "fieldName": "productName",
      "fieldDescription": "제품명", "fieldMaxLength": 200,
      "fieldAddDescription": "…법령 근거 등(선택)" } ] }
```

`fieldType` 은 **String · Boolean · LocalDate · YearMonth · Integer · Long** 6종이며 `widgetOf()` 가 입력 위젯(text/textarea/boolean/date/month/number)으로 옮긴다. `fieldAddDescription` 은 법령 근거가 들어 있어 화면에 그대로 노출한다.

> "네이버 응답 → 우리 shape" 변환은 `naverNoticeSchema.normalizeNaverSchema()` **한 함수에만** 가둬 뒀다. 응답이 바뀌면 고칠 곳은 거기 하나다. 화면·매퍼는 우리 shape 만 본다.

> ⚠ 상품군은 **36종**이다(40종이 아니다). 매퍼의 `NOTICE_TYPE_TO_KEY` 에 `RENTAL_HA`(정수기·비데 대여)가 빠져 있어 추가했다 — 표에 없는 유형은 수집 시 조용히 건너뛰므로 목록과 대조할 것.

### 6.5 고시 유형은 **리프 카테고리**가 정한다 ★

초기 구현은 `naver_publish_profile.notice_type` = **몰당 1개**였다. 의류와 건강기능식품을 함께 파는 종합몰에서는 어느 한쪽이 반드시 틀린 고시로 나간다.

> ⚠ **네이버는 카테고리별 고시 유형을 알려주지 않는다.**
> `GET /v1/products-for-provided-notice?leafCategoryId=` 로 걸러 보려 했으나 **파라미터가 무시되고 36종 전체가 그대로 온다**(2026-07-21 실호출 확인 — 홍삼액·반팔티셔츠·무파라미터 세 응답이 모두 동일). `?categoryId=` 는 빈 배열을 준다. 문서에 "카테고리 확정 후 조회가 정석"이라고 적혀 있던 건 스펙 추정이었다.
> → **이 매핑은 우리가 만들어 보관해야 하는 자산이다.**

`naver_category.notice_type` 에 유형을 붙인다. 상품은 이미 `naver_category_id` 를 갖고 있으므로 등록 시 그대로 따라온다.

```
리프 4,999개  ← 사람이 하나씩 지정할 수 없다
   ↓ 경로 규칙(naverNoticeMapping.RULES) 일괄 배정
4,999건 배정 (미매칭 0)
   ↓ 틀린 것만 화면에서 개별 수정 → notice_source='MANUAL'
규칙 재적용해도 MANUAL 은 보존
```

규칙은 `whole_category_name` 경로를 **위에서부터** 평가하며, 구체적인 것이 위에 온다. 주의할 순서 두 가지:

- `식품>건강식품` → `DIET_FOOD` 를 `식품` → `GENERAL_FOOD` **앞에** 둔다.
- **"옷"은 대분류와 무관하게 `WEAR`** 다(`스포츠/레저>등산>등산의류`, `출산/육아>유아동의류`). 대분류 규칙보다 앞에 둔다.

**값도 유형별로 있어야 한다.** 유형이 달라지면 채울 항목이 달라지므로 값 한 벌로는 안 된다 → `mall_notice_default (mall_id, notice_type, values_json)`.

등록 시 결정 순서(`naverNoticeMapping.resolveForProduct`):

| 순위 | 유형 | 값 |
|---|---|---|
| 1 | 리프 카테고리의 `notice_type` | 그 유형의 `mall_notice_default` |
| 2 | `naver_publish_profile.notice_type` (폴백) | 프로필의 `notice_defaults_json` |
| 3 | 상품별 `override_json` 이 있으면 위를 덮는다 | |

화면: `/admin/sourcing/naver-taxonomy` 카테고리 탭 — 행마다 고시 유형 select + [규칙으로 일괄 배정] 버튼.

**카테고리 수집 시 자동 배정된다.** `syncCategories()` 가 끝나면 `applyRules()` 를 이어서 돌린다(`categoryReflect` 와 같은 자리). 이유:

- 카테고리 upsert 는 `notice_type` 을 건드리지 않으므로 **기존 값은 보존**되지만, **새로 들어온 리프는 NULL** 로 남는다.
- 미지정인 채 등록하면 몰 폴백 유형이 쓰여 **틀린 고시가 나간다.** 사용자가 버튼을 따로 눌러야만 채워지는 구조는 사고가 난다.
- `MANUAL` 은 보존되므로 자동 재적용이 안전하다. 실패해도 수집 자체는 SUCCESS 로 두고 화면 버튼으로 재시도할 수 있다.

> 같은 원칙을 다른 화면에도 적용한다 — **관리자 화면에 JSON 입력란을 새로 만들지 않는다.** 구조화된 값이 필요하면 선택·조회 UI 로 받고 조립은 서버가 한다. (2026-07-21 기준 관리자 뷰의 JSON textarea 는 전수 조사 결과 이 한 곳뿐이었고, 제거 완료.)

## 7. 데이터 매핑 (우리 몰 → 네이버)

### 7.0 카테고리는 두 개의 별개 축이다 ★

| | 우리 몰 카테고리 | 네이버 카테고리 |
|---|---|---|
| 저장 | `categories` (글로벌 `mall_id=0`, 계층) | `naver_category`(참조 리소스) + `products.naver_category_id`(매핑) |
| 최대 단계 | **3단계** (`navigation_config.category_max_depth`, 기본 3) | **4단계** (대>중>소>세) |
| 성격 | 우리가 자유롭게 만드는 전시·분류 트리 | 네이버가 정의한 고정 트리 — **임의 생성 불가** |

**우리 트리를 네이버에 만들 수 없다.** `leafCategoryId` 는 전체 카테고리 조회 API 가 반환하는 **유효한 네이버 ID만** 허용하고(임의값 400), 게다가 **등록 후 대분류 변경이 불가능**하다. 그래서 상품을 네이버의 **기존 리프에 매핑**하는 방식만 성립한다. (출처: 네이버 commerce-api Discussions #246, #1881)

경로 파싱 규칙 — 네이버 위젯에서 리프를 고르면 `whole_category_name`(4단계) 중 **마지막 3단계**만 우리 경로로 채운다(대분류를 자동 생략). 예: `패션잡화>여성신발>부츠/워커>미들부츠` → `여성신발>부츠/워커>미들부츠`. 몰 최대 뎁스를 넘으면 **앞(대분류)을 잘라** 가장 구체적인 리프를 보존한다. 생성은 `taxonomyResolver.resolveOrCreatePath` 가 부모-자식 체인을 단계별로 만들거나 재사용한다(depthGuard 경유).

> **잔여** — 상품에 네이버 카테고리가 매핑돼 있지 않을 때, **우리 카테고리 경로를 근거로 네이버 카테고리를 자동 추천**하고 확정 가능한 단계까지 좁혀 관리자가 최종 선택하게 하는 흐름은 미구현이다. 지금은 등록 폼에서 매번 직접 검색해 고른다.

### 7.1 필드 매핑표

| 네이버 필드 | 우리 원천 | 비고 |
|---|---|---|
| `statusType` | (고정) `SALE` | **등록 시 SALE 만 유효**. 재고 0이면 네이버가 자동 OUTOFSTOCK |
| `leafCategoryId` | `products.naver_category_id` | **리프**만 가능 |
| `name` | `products.name` | ≤100자 |
| `detailContent` | `products.description` | 새니타이즈 + 이미지 URL 치환 |
| `images.representativeImage` | `products.main_image` | 업로드 후 네이버 URL |
| `images.optionalImages` | `product_images` | **최대 9장** |
| `salePrice` | 옵션상품: **최저 SKU 가**<br>단일상품: `products.price` | ↓ 아래 설명 |
| `stockQuantity` | 옵션상품: 판매가능 SKU 재고 합<br>단일: `products.stock` | |
| `detailAttribute.optionInfo` | `product_option` / `product_sku` | **detailAttribute 하위**여야 함 |
| `deliveryInfo` | `naver_publish_profile` (+ `shipping_policy` 기본값) | |
| `afterServiceInfo` / `originAreaInfo` | `naver_publish_profile` | 우리 몰에 없는 개념 |
| `productInfoProvidedNotice` | `naver_publish_profile.notice_*` + 상품별 override | |
| `sellerCodeInfo.sellerManagementCode` | `products.product_code` (없으면 `YD-{id}`) | 주문 역추적 열쇠 |
| `smartstoreChannelProduct.channelProductDisplayStatusType` | `products.status`/`visibility` | **ON / SUSPENSION 만** |

### 옵션 가격 변환 ★

네이버의 옵션 `price` 는 **판매가 대비 추가금액**이고 우리 SKU 는 절대가다.

```
salePrice        = min(판매가능 SKU 가격)
조합.price       = SKU가격 - salePrice   (≥ 0)
```

이 방식이면 네이버 정책인 **"추가금액 0원 조합이 1개 이상"** 도 자동으로 만족한다. `status='OFF'` SKU 는 제외한다(팔 수 없는 옵션을 올리면 재고만 꼬인다).

**네이버 조합형 옵션 축은 최대 3개**다. 우리 축이 더 많으면 등록을 차단하고 안내한다.

### 페이로드 구조에서 틀리기 쉬운 5가지 (전부 실제 400 사례)

1. `smartstoreChannelProduct` 는 `originProduct` 의 **형제**다. 안에 넣으면 400.
2. `optionInfo` 는 **`detailAttribute` 하위**. 밖에 두면 조용히 "옵션 없음"으로 등록된다.
3. `optionCombinationGroupNames` 는 배열이 아니라 **객체** (`{optionGroupName1: "..."}`).
4. 고시는 **type 에 해당하는 하위 객체 1개만**.
5. `channelProductDisplayStatusType` 에 `"SELL"` 은 없다 (`ON`/`SUSPENSION`).

### ⚠ collation 드리프트 (실제로 쿼리가 죽는다)

| 컬럼 | collation |
|---|---|
| `products.naver_category_id` | `utf8mb4_general_ci` |
| `categories.naver_category_id` | `utf8mb4_general_ci` |
| `naver_category.naver_category_id` | `utf8mb4_unicode_ci` |

그냥 조인하면 `Illegal mix of collations` 로 **쿼리 자체가 실패**한다. 네이버 카테고리를 조인할 때는 반드시 명시한다:

```sql
LEFT JOIN naver_category nc
       ON nc.naver_category_id = p.naver_category_id COLLATE utf8mb4_unicode_ci
```

근본 해결은 `products`/`categories` 쪽 컬럼을 `utf8mb4_unicode_ci` 로 맞추는 것이나, `products` 가 9,680행이라 지금은 쿼리에서 처리하고 있다. 정리할 때 함께 바꿀 것.

### 9.1 등록 기본값(프로필)은 **몰당 한 번**만 입력한다

`naver_publish_profile` 은 `UNIQUE(mall_id)` — 몰당 1행이다. 한 번 저장하면 이후 모든 상품 등록에 자동 주입되며, 상품마다 다시 입력하지 않는다(상품별로 다르게 하려면 상품 override 로 덮는다).

입력 부담을 줄이려고 **비어 있는 항목은 몰의 다른 설정에서 자동으로 채운다**.

| 프로필 항목 | 자동 출처 |
|---|---|
| A/S 전화번호 | `site_settings.contact_phone` |
| A/S 안내 | `site_settings.cs_hours` (개행은 ` / ` 로 정리) |
| 기본 배송비 · 무료배송 기준금액 | `shipping_policy.base_fee` / `free_threshold` |

> ★ 자동 채움은 **"행이 없을 때만"이 아니라 빈 필드에 항상** 적용한다. 행 유무로만 판단하면 프로필을 한 번 저장한 순간 빈 칸이 NULL 로 굳어 자동 채움이 영영 멈춘다. 사용자가 직접 적은 값은 절대 덮지 않는다.
> 화면은 자동으로 채운 항목에 "사이트 설정에서 가져옴" 배지를 띄운다.

**자동화하지 않는 것 — 경계를 지킬 것.** `origin_area_code`(원산지 코드), 출고지·반품지 주소록 번호는 우리 몰에 대응 개념이 없고 조회 API 도 §12 #2·#6 에서 **미확인**이다. 근거 없는 값을 지어내면 등록이 조용히 틀린 채로 나간다. 확인 전까지는 수동 입력 + 안내로 둔다. 반품·교환 배송비도 우리 배송정책에 원천이 없어 수동이다.

### 9.2 "배송 정보"는 상품에 붙는 기본값이지 송장이 아니다

프로필의 택배사·배송비는 **상품에 표기되는 기본 배송 정보**다. 실제 출고 때 송장에 찍히는 택배사·운송장번호는 **주문마다 발송 시점에 따로 등록**하며 이 값과 달라도 된다(주문 수집·송장 등록은 2차 과제). 화면 문구에도 그렇게 적어 둔다 — 이 둘을 같은 것으로 오해하면 "왜 미리 택배사를 정하냐"는 혼선이 생긴다.

택배사는 자유 입력 대신 **코드 select** 로 받는다(`CJGLS`, `HANJIN`, `HYUNDAI`(롯데), `EPOST` 등). 코드를 오타 내면 등록이 실패한다.

## 10. 운영 흐름

```
[사전] 커머스API센터에서 앱 발급 → /admin/sourcing/connections 에 client_id/secret 등록 → [검증]
       → /admin/sourcing/naver-taxonomy 에서 카테고리 수집
       → /admin/sourcing/publish 의 [네이버 등록 기본값] 작성 (A/S·원산지·반품교환비·고시)

[등록] 상품에 네이버 카테고리(리프) 지정 → /admin/sourcing/publish 에서 선택 → [선택 상품을 네이버에 등록]
       → 건당 2~4초 순차 전송 → 성공 시 원상품번호·채널상품번호 저장

[검증] 등록 10~30분 뒤 [상태 확인] → 강제 카테고리 이동·판매금지 여부 확인
```

### "200 성공 ≠ 정상 등록" ★

네이버는 등록 요청에 200 을 준 **뒤에** 상품과 맞지 않는 카테고리를 강제 이동시키거나 판매금지(`PROHIBITION`)로 바꿀 수 있다. 그래서 등록 후 실제 상태를 되읽는 확인 단계가 반드시 필요하다.

`verifyPublished()` 가 `GET /v2/products/origin-products/{no}` 로 상태를 확인해, 아래 상태면 매핑을 `SUSPENDED` 로 내린다:
`PROHIBITION · REJECTION · UNADMISSION · SUSPENSION · CLOSE`

카테고리가 요청값과 다르면 "강제 이동" 으로 표시한다.

> 현재는 **수동 버튼**이다. 2차에서 등록 N분 후 자동 확인하는 배치로 승격할 것.

### 등록 후 대분류 변경 불가

등록 후에는 대분류를 바꿀 수 없다. 잘못 올렸으면 삭제 후 재등록해야 한다.

## 11. 실패 패턴과 처리

| 패턴 | 재시도 | 처리 |
|---|:--:|---|
| 429 쿼터 소진 (`Remaining: 0`) | ✅ | 3→6→12→24→48초 백오프 |
| 429 서비스 혼잡 (`Remaining > 0`) | ✅ | 더 긴 대기 + 적응형 스로틀 상향 |
| 429 헤더 누락 | ✅ | 혼잡으로 간주 |
| 401 `GW.AUTHN` | ✅ 1회 | 토큰 재발급 후 원요청 1회 |
| 5xx | ✅ | 백오프 |
| `"이전 요청이 진행중입니다"` | ⚠ | 이미지 동시 호출 위반 — **설계로 해결**(계정 락) |
| 필수값 누락·카테고리 오류·고시 누락 | ❌ | **재시도 금지**. 격리 + 사유 기록 |
| `GW.IP_NOT_ALLOWED` | ❌ | 등록 IP ≠ 실제 아웃바운드 IP |
| **"등록권한이 있어야만 판매가 가능합니다"** | ❌ | **카테고리 판매권한 미보유.** 코드 문제가 아니다 — 스마트스토어센터에서 해당 카테고리 권한을 신청해야 한다(건강기능식품 등 규제 카테고리). 화면에 "권한 신청 필요"로 안내할 것 |

**모든 실패 로그에 `GNCP-GW-Trace-ID` 를 남긴다.** 네이버 기술지원 문의 시 이 값이 없으면 원인 확인이 사실상 불가능하다. `naverClient` 가 응답 헤더에서 뽑아 `NaverApiError.traceId` 로 전달하고, `channel_publish_log.message` 에 기록한다.

오류 응답의 `invalidInputs[].name` 은 **실제 오류 필드와 정확히 일치하지 않을 수 있다**(공식 안내). `message` 내용으로 판단할 것.

### 보내기 전 자체 검증

네이버에 보내고 400 으로 알게 되면 호출 한도만 태우고 원인 추적도 어렵다. 그래서 `mapper.validateBeforePublish()` 와 `naverProfile.validateProfile()` 이 **전송 전에** 막는다:

상품 — 네이버 카테고리, 대표 이미지, 상품명, 판매가, (옵션상품이면) 판매가능 SKU
프로필 — A/S 전화·안내, 원산지 코드, 반품·교환 배송비
고시 — 선택한 상품군의 **필수 필드**(`naverNoticeSchema.missingRequired()`). 프로필 화면에 누락 항목명을 그대로 띄운다. `validateProfile()` 은 판매자 레벨만 보므로 고시는 이쪽이 담당한다.

### 카테고리 일괄 상속 — 리프 게이트 ★

상품에 네이버 리프 카테고리를 한 건씩 손으로 넣는 것은 9,714건 규모에서 현실성이 없다.
그런데 `categories.naver_category_id` 는 이미 채워져 있으므로 **상품이 속한 카테고리의 네이버 ID 를 상품으로 내려주면** 대량 등록이 열린다.

구현: `services/sourcing/channel/naverCategoryInherit.js` · 버튼 `/admin/sourcing/publish` → [카테고리에서 일괄 지정]

> ⚠ **`categories.naver_category_id` 를 그대로 복사하면 안 된다.** 우리 카테고리 트리는 네이버 **레벨 1~3** 까지만 반영돼 있고, 네이버 리프는 3,468개가 레벨 4 에 있다. 그래서 카테고리에 네이버 ID 가 있어도 대부분 **비리프**다. 비리프 ID 를 상품에 박으면 등록이 전량 400/반려가 되고 호출 한도만 태운다.
> → `naver_category.is_leaf = 1` 인 것만 내려보내고 나머지는 스킵으로 집계한다. **이미 값이 있는 상품은 덮어쓰지 않는다**(수동 지정 보호).

실측(2026-07-21, 트랜잭션 롤백으로 검증):

| 몰 | 전체 | 지정 가능(리프) | 비리프로 스킵 |
|---|---:|---:|---:|
| mall 2 | 9,680 | **2,069** | 7,611 |
| mall 28 | 34 | **24** | 10 |

즉 버튼 한 번으로 **2,069건이 등록 가능 상태가 된다.**

### 입력 지점은 **카테고리 관리 화면** 하나다 ★

상품 9,680건에 네이버 카테고리를 개별 입력할 수는 없다. 그렇다고 네이버 등록 시점에 받으면 등록 직전에 수천 건이 막혀 병목이 된다(실제로 "등록 가능 상품 0건"이 그 상태였다).

그래서 **사용자가 의식적으로 입력하는 지점은 카테고리 관리 한 곳**으로 모은다.

| 시점 | 하는 일 | 횟수 |
|---|---|---|
| **카테고리 상세**(`/admin/categories/:id`) | 우리 카테고리 ↔ 네이버 **리프** 연결 | 카테고리당 1회 |
| 상품 등록·수정 | 카테고리에서 자동 상속. 다르면 그때만 개별 지정 | 예외만 |
| 네이버 등록 화면 | 남은 것 선택 일괄 지정 + 등록 | 정리용 |

카테고리 상세의 [네이버 스마트스토어 연결] 카드:
- 리프만 검색되는 검색창(코드를 사람이 알 수 없으므로 **직접 입력 불가**)
- 연결된 리프가 정한 **고시 유형을 읽기 전용으로 표시**(§6.5 — 따로 고르지 않는다)
- [이 카테고리 상품에 바로 반영] 체크 시 소속 상품의 **빈 값만** 채움(이미 지정된 건 보존)
- 저장 시 **리프 검증**. 비리프·미수집 코드는 거부한다 — 실패를 상품 등록이 아니라 지정 단계에서 막는다.

구현: `naverCategoryInherit.setCategoryMapping()` / `categoryMappingInfo()` · `categoryController.postNaverCategory`

> ⚠ 기존 `categories.naver_category_id` 값의 **정확도는 검증되지 않았다.** 예: 성인 "티셔츠" 카테고리에 `50000535`(출산/육아>유아동의류>티셔츠)가 붙어 있다. `categoryReflect` 가 이름만 보고 붙인 결과로 보인다. 상속을 돌리기 전에 카테고리 화면에서 눈으로 확인할 것.

### 상속으로 못 채우는 상품 — 선택 일괄 지정

남는 7,611건은 우리 카테고리가 네이버 **대·중분류**에 걸려 있어 상속으로는 영영 채워지지 않는다. 이건 상품 수정 화면에서 한 건씩 넣을 규모가 아니므로 목록에서 **선택 → 리프 검색 → 일괄 지정** 경로를 둔다.

```
필터 [카테고리 미지정만] 로 대상을 좁힘
   → 전체선택(한 화면 30건)
   → 리프 카테고리 검색(GET /admin/sourcing/naver-taxonomy/search — is_leaf=1·is_active=1 만 반환)
   → [선택 상품에 지정]  (POST /admin/sourcing/publish/assign-category)
```

상속과 다른 점 두 가지:
- 사람이 명시적으로 고른 값이라 **이미 값이 있어도 덮어쓴다**(재지정이 목적).
- 그래도 **서버가 리프를 한 번 더 검증한다.** 화면 검색이 리프만 주더라도 폼 값은 조작될 수 있고, 비리프가 박히면 등록 때 전량 400 이 난다. `UPDATE` 는 `mall_id` 로 스코프를 걸어 다른 몰 상품에 닿지 않는다.

구현: `naverCategoryInherit.assignCategory()` · `sourcingController.postAssignNaverCategory`

읽기만 하고 저장하지 않는 것: **주소록**(`/v1/seller/addressbooks-for-page`), **판매채널**(`/v1/seller/channels`).

## 15. 역방향 수집 — 스마트스토어 → 우리 몰

화면 `/admin/sourcing/channel-import` · 구현 `services/sourcing/channel/naverChannelImport.js`

판매자가 스마트스토어에 **직접 등록해 둔 상품**을 우리 몰 상품(`products`)으로 만들어 한 곳에서 관리한다.

```
[조회] POST /v1/products/search           (읽기 — 우리 DB 를 바꾸지 않는다)
   ↓   목록에 "새 상품 / 우리가 등록함 / 이미 가져옴" 표시
[가져오기] GET /v2/products/origin-products/{no}  건당 1회
   ↓   이미지 다운로드 → products · product_sku · product_option(+값) · product_images
   ↓   channel_product_mapping(source_type='CHANNEL_IMPORT') + channel_sku_mapping
[확인] 상품 관리에서 내용 확인 후 판매·노출을 켠다 (기본은 판매중지·비노출)
```

### 15.1 목록 조회 — 실호출로 확정한 것 ★

**`POST /v1/products/search`** 는 존재하며 응답은 `contents[].channelProducts[]` 중첩이다
(원상품 1건 아래 채널상품 N건). 우리는 **채널상품 1건 = 목록 1행**으로 편다.

| 확인 항목 | 결과 |
|---|---|
| `orderType` | **보내면 400** (`허용하지 않는 정렬조건입니다`). 생략하면 `regDate DESC, productNo DESC` 로 온다 → **아예 보내지 않는다** |
| `productStatusTypes` | 정상 동작. `SALE`·`OUTOFSTOCK`·`SUSPENSION` 을 쓴다 |
| `searchKeywordType='SELLER_CODE'` + `sellerManagementCode` | **동작** |
| `searchKeywordType='CHANNEL_PRODUCT_NO'` + `channelProductNos`(숫자 배열) | **동작** (단수 `channelProductNo` 는 400) |
| **상품명 검색** | ❌ **제공되지 않는다.** `CHANNEL_PRODUCT_NAME` 은 enum 으로 통과하지만 짝 필드를 찾을 수 없었다 — `channelProductName`·`productName`·`searchKeyword`·`keyword`·`name`·`channelProductNames`·`productNames`·`searchKeywordValue`·`searchWord` **9종 전부 조용히 무시**(무매칭 키워드를 넣어도 전건 반환) |

> 상품명 파라미터를 지어내면 "검색했는데 엉뚱한 게 나오는" 화면이 된다. 그래서 화면 검색창은
> **판매자관리코드 / 채널상품번호**만 받고(형태로 자동 구분), 상품명은 **조회된 목록 안에서 걸러 보는** 입력으로 분리했다.

### 15.2 채널상품번호는 상세 응답에 없다 ★

`GET /v2/products/origin-products/{no}` 응답의 `smartstoreChannelProduct` 에는 **`channelProductNo` 가 없다**(실호출 확인).
목록 응답에는 있다. 그래서 체크박스 값을 `원상품번호:채널상품번호` 로 실어 보내 매핑에 함께 저장한다.
**두 번호는 별개**이므로 원상품번호를 채널상품번호 자리에 넣으면 안 된다(§4).

### 15.3 우리 몰 상품 생성 규칙

`publishService.publishToMall`(공급처 → 우리 몰)과 **같은 규칙**을 따른다. 규칙이 두 벌이 되면 재고·주문이 조용히 어긋난다.

| 항목 | 처리 |
|---|---|
| 옵션 가격 | **`sku 가격 = salePrice + combination.price`** — 매퍼(§7)의 정확한 역연산 |
| 옵션 축 | `optionCombinationGroupNames.optionGroupName1~3` → `product_option` |
| `usable=false` 조합 | SKU 를 만들되 `status='OFF'` (팔 수 없는 옵션을 켜 두면 재고가 꼬인다) |
| SKU 구조 | 옵션상품은 대표 SKU 없음 / 단일상품만 `is_default=1` 1건 |
| 이미지 | 네이버 CDN URL → `urlIngest` 로 내려받아 `/uploads/products/` 저장(핫링크 금지) |
| 카테고리 | `leafCategoryId` → `categories.naver_category_id` 역참조. **없으면 미분류(NULL)로 두고 가져오기를 막지 않는다** (화면에서 폴백 카테고리를 고를 수도 있다) |
| 상태 | **판매중지 · 비노출**이 기본. 배송·고시가 우리 정책과 다를 수 있어 검수 전이다 |
| `source_channel` | `NAVER_SMARTSTORE` (역수집 상품 식별) |

### 15.4 중복 방어 — 여기가 정확성의 핵심 ★

상품 생성 **전에** `channel_product_mapping` 을 원상품번호로 조회한다.

| 매핑 상태 | 처리 |
|---|---|
| 없음 | 새 상품 생성 (`source_type='CHANNEL_IMPORT'`) |
| `BUILDER` (우리가 네이버에 올린 것) | **되가져오지 않는다.** 가져오면 같은 상품이 우리 몰에 두 개 생긴다. 목록에서 체크박스 자체를 주지 않고, 서버도 거부한다 |
| `CHANNEL_IMPORT` | 이미 가져왔다 → **건너뜀**(실패가 아니다). 재실행이 idempotent 하다 |

> ⚠ **"건너뜀"은 갱신이 아니다.** 상세설계 §10 은 "기존 매핑 있으면 갱신"으로 적혀 있으나
> 1차는 **skip** 으로 구현했다. 가져온 뒤 관리자가 손본 상품명·가격·이미지를 네이버 값으로
> 되덮으면 편집이 사라지기 때문이다. 대신 **네이버에서 상품이 바뀌어도 재가져오기로 반영되지 않는다** —
> 갱신이 필요하면 어떤 필드를 덮을지 정하는 설계가 먼저다(2차).

> ⚠ **상세 HTML 안의 인라인 이미지는 네이버 CDN 을 그대로 참조한다.** 내려받는 것은
> 대표 이미지와 추가 이미지뿐이다. 네이버가 이미지를 내리면 상세가 깨지고, 몰을 다른 곳으로
> 포팅할 때도 따라오지 않는다. 1차 한계로 둔다(상세 HTML 전체를 재작성해야 하는 작업이다).

## 16. 재고 연동 — 우리 몰 → 네이버

화면 `/admin/sourcing/sync` · 구현 `services/sourcing/channel/naverStockSync.js`

**재고의 정본은 우리 몰이다**(`services/catalog/sellableStock.js` — 판매가능재고 = `sku.status='ON' AND stock_managed=1` 인 재고의 합).
네이버 값을 우리 DB 에 덮어쓰지 않는다. 화면 이름이 "가져오기"라고 해서 역방향으로 쓰면 결제·주문 재고가 조용히 어긋난다.

- **전송(쓰기)** — 우리 판매가능재고 → 네이버
- **대사(읽기)** — 네이버 현재값을 조회만 해 우리 값과 나란히 보여 준다. DB 에 쓰지 않는다.

1차는 **수동 버튼**이다(설계 §12.3). 배치·스케줄러는 2차.

### 16.1 전송 경로가 상품 유형에 따라 다르다 ★★

이번 작업에서 가장 값비싼 발견이다. **하나의 엔드포인트로 통일되지 않는다.**

| 유형 | 경로 | 본문 |
|---|---|---|
| **옵션상품** | `PUT /v1/products/origin-products/{no}/option-stock` | `{ optionInfo: { useStockManagement: true, optionCombinations: [...] }, stockQuantity: N, productSalePrice: { salePrice: 기준가 } }` |
| **단일상품** | `GET /v2/products/origin-products/{no}` → `originProduct.stockQuantity` 교체 → `PUT /v2/products/origin-products/{no}` | 되읽은 원본 그대로 + 재고만 교체 |

실호출로 확인한 함정 세 가지:

1. **`optionInfo` 는 옵션이 없어도 필수다.** 빼면 400 `optionInfo: 데이터를 입력해 주세요.`
   그리고 옵션상품은 **`productSalePrice` 도 필수**다. 빼면 400 `옵션가 수정 시 판매가를 필수 입력해 주세요.`
   옵션 `price` 가 판매가 대비 추가금액이라 기준가 없이는 해석할 수 없기 때문이다.
   → ★ **옵션상품은 재고만 보낼 수 없고 판매가가 함께 나간다.** 우리 SKU 최저가가 그대로
   네이버 판매가가 되므로, 우리 몰에서 가격을 고치면 재고 전송 시 네이버 가격도 바뀐다.
2. **등록 페이로드처럼 `originProduct` 로 감싸면 같은 400** 이 난다. option-stock 의 본문은 **평평한 구조**다.
3. ★ **단일상품에 option-stock 을 쓰면 200 을 주고도 재고가 바뀌지 않는다.**
   빈 조합(`optionCombinations: []`)으로 100 → 77 을 요청하고 되읽으면 **여전히 100**이었다.
   원상품 전체 수정으로 바꾸니 77 이 반영됐고, 100 으로 원복까지 확인했다.
   → **"200 성공 ≠ 반영"** 이 등록(§10)뿐 아니라 재고에도 그대로 적용된다.

> ⚠ 단일상품 경로는 **원상품을 통째로 되읽어 재고만 갈아 끼운다.** 우리가 만든 페이로드로 덮어쓰지 않는다 —
> 네이버에서 직접 등록·수정한 값(상세HTML·고시·배송)을 우리 값으로 조용히 밀어 버리면 안 되기 때문이다.

> ⚠ **단일상품 경로는 원상품 전체 수정이라 재검수를 유발할 수 있다.**
> 검증은 **의류** 상품으로 했고 즉시 되읽었을 때 `SALE` 이었다. 그런데 네이버 검수는 비동기라
> (§10 — 등록 후 10~30분) 즉시 조회로는 배제되지 않는다. **건강기능식품처럼 판매권한이 필요한
> 규제 카테고리**는 재검수에서 `PROHIBITION`·`UNADMISSION` 으로 떨어질 수 있으므로,
> 전송 뒤 시간을 두고 판매상태를 한 번 더 확인할 것. 이 몰의 표준 예시가 건강식품몰이라 실사용에서 바로 만난다.

### 16.2 보내는 값과 스킵 판정

- 옵션상품 — `channel_sku_mapping` 으로 우리 SKU ↔ 네이버 옵션조합을 이어 **배열 전체를 한 번에** 보낸다(N+1 금지).
  옵션 `price` 는 판매가 대비 추가금액이므로 등록 때와 같은 규칙(판매 가능 SKU 최저가)으로 기준가를 다시 낸다.
  우리 몰에서 끈 SKU 는 `usable:false` 로 보낸다.
- 단일상품 — 판매가능재고 합만 보낸다. 전송 기록은 `channel_sku_mapping` 에 `naver_option_id IS NULL` 행으로 남긴다.
  **이걸 남기지 않으면 비교 근거가 없어 화면에서 영원히 "확인 필요"로 보이고 변경없음 스킵도 동작하지 않는다.**
- `last_sent_stock` 과 같으면 **호출하지 않고 건너뛴다**(호출 한도는 유한하다). [강제 전송] 으로 무시할 수 있다.
- ⚠ `useStockManagement=false` 로 보내면 네이버가 수량을 **9,999 로 덮는다.** 항상 `true`.

## 17. 가져온 상품 → 스마트스토어 바로 전송

화면 `/admin/sourcing/staging` [스마트스토어로 바로 전송] · 구현 `services/sourcing/channel/naverDirectPublish.js`

```
supplier_product ──①──▶ products(우리 몰) ──②──▶ 네이버
              publishService        naverPublishService
```

### 17.1 중간 테이블에서 네이버로 곧장 쏘지 않는다 ★

버튼 이름은 "바로 전송"이지만 **우리 몰 상품을 반드시 거친다.** 건너뛸 수 없는 이유가 셋이다.

1. `channel_product_mapping.product_id` 가 **NOT NULL** 이고 `channel_sku_mapping.sku_id` 는 `product_sku` 를 가리킨다.
   상품 행이 없으면 매핑을 만들 수 없고, 그러면 **재고 전송(§16)·주문 역추적이 통째로 죽는다.**
2. 네이버 페이로드는 옵션 축·조합·SKU 코드·정제된 상세HTML을 요구하는데 그 정규화가 곧 ① 단계다.
   중간 테이블에는 공급처 원문(`supplier_variant`)만 있다.
3. 등록 파이프라인이 두 벌이 되면 가격·재고 규칙이 갈라진다.

즉 이 기능은 **두 버튼을 한 번에 누르는 편의**이지 새 경로가 아니다. 사용자에게는 한 동작, 데이터 모델은 그대로.

### 17.2 실패를 감추지 않는다 ★

`naverPublishService.publishMany` 는 전제 조건이 없으면(자격증명 없음·등록 기본값 미작성)
**전건을 통째로 throw** 한다. 그대로 흘려보내면 화면에는 오류만 뜨고 **우리 몰에는 ①이 만든 상품이 조용히 쌓인다.**

그래서 ②를 잡아 두 단계 결과를 함께 돌려주고, 화면에 이렇게 표시한다.

> 우리 몰에는 N건이 등록됐지만 네이버 전송은 시작하지 못했습니다: (사유)
> — 설정을 마친 뒤 [스마트스토어 등록] 화면에서 해당 상품을 다시 보내세요.

### 17.3 카테고리·상한

- 네이버 리프 카테고리는 화면에서 **검색해 고른다**(선택). 비우면 우리 카테고리에 연결된 리프를 상속한다.
- 상속은 **이번에 만든 상품에만** 적용한다. `naverCategoryInherit.applyInherit()` 는 **몰 전체**를 돌기 때문에
  그대로 쓰면 사용자가 누른 버튼의 범위를 넘어 수천 건을 건드린다 — 같은 규칙으로 범위만 좁힌 쿼리를 따로 둔다.
- 리프 검증은 ① 을 돌리기 **전에** 한다. 나중에 막히면 우리 몰에만 상품이 생기고 네이버엔 아무것도 안 올라간다.
- 1회 **10건**. ②가 건당 2~4초라 화면 타임아웃을 넘기지 않는 선이다.
- 상품 상태를 `OFF`(판매중지)로 두면 네이버에서도 `SUSPENSION`(전시중지)로 올라간다(`displayStatusOf`).

### 실호출로 드러난 것

1. **카테고리 판매권한.** 건강기능식품(홍삼) 상품은 400 `등록권한이 있어야만 판매가 가능합니다` 로 거부됐다. 의류로 바꾸니 즉시 성공. **규제 카테고리는 스마트스토어센터에서 권한 신청이 선행되어야 한다.** 이 몰의 표준 예시가 건강식품몰이므로 실사용 전 반드시 확인할 것.
2. **이미지가 외부 URL 이라 등록 자체가 불가능했다** — §5 참고. 이게 없었으면 전 상품 실패였다.
3. **`nodemon.json` 의 watch 에 `services/` 가 빠져 있었다.** 서비스 코드를 고쳐도 서버에 반영되지 않아 "코드 버그"로 오인하기 쉽다. `services`·`shared` 를 추가했다.

### ~~남은 설계 갭 — 고시 유형이 몰당 1개다~~ → 해결(§6.5)

아래는 발견 당시 기록이다. 2026-07-21 리프 카테고리별 지정으로 해결했다.

`naver_publish_profile.notice_type` 은 **몰 단위 값 하나**다. 그런데 실제로는 **상품 카테고리마다 고시 상품군이 달라야 한다**(의류=WEAR, 건강기능식품=DIET_FOOD…). 종합몰에서는 이 구조로 정상 등록이 불가능하다.

`mapper.buildNotice(profile, override)` 는 이미 상품별 override 를 받도록 돼 있으나 **`naverPublishService` 가 override 를 채우지 않고, 저장할 곳도 화면도 없다.** 다음 차수 과제:
- 상품(또는 우리 카테고리) 단위 고시 유형·값 저장
- 네이버 리프 카테고리로 `GET /v1/products-for-provided-notice?categoryId=` 를 불러 **상품군을 자동 추천**
