# 네이버 스마트스토어 카테고리/브랜드 참조 리소스 — 설계

> 상태: 골격 구현 완료(코드·스키마 파일). 마이그레이션 실행·자격증명 입력·크론 등록은 미적용.
> 관련: [도매꾹_온채널_스마트스토어_연동_상세설계.md](./도매꾹_온채널_스마트스토어_연동_상세설계.md) 의 `services/sourcing/` 골격을 확장한다.

## 1. 목적과 원칙

네이버 스마트스토어 상품 등록에 쓰이는 **전체 카테고리(및 브랜드)** 를 주기적으로 수집해
**참조 리소스**로 저장한다. 상품 등록 화면에서 관리자가 이 리소스를 검색·선택하면,
그 선택을 **근거**로 실제 몰 카테고리(`categories`)를 생성/매핑한다.

핵심 원칙:

- **참조 리소스 ≠ 몰 카테고리.** 수집된 네이버 카테고리는 `naver_category` 에만 적재되고
  몰 `categories` 에는 **자동 반영되지 않는다.** 반영은 오직 상품 등록 시 관리자의 선택으로만.
- **전역 리소스.** 네이버 카테고리는 판매자 공통이라 `mall_id` 없이 한 벌만 둔다.
  수집에는 유효한 네이버 자격증명 **1건**만 있으면 된다.
- **기존 자산 재사용.** 자격증명(`mall_channel_credential`·`NAVER_SMARTSTORE`),
  게이트(`mall_channel_setting.sourcing_enabled`·`sourcingFlag`),
  매핑 엔진(`services/catalog/taxonomyResolver`), 크론 표준(고정 셸 + DB 스케줄).

## 2. 데이터 모델 (scripts/migrate_sourcing_02_naver_taxonomy.sql)

| 테이블 | 역할 |
|---|---|
| `naver_category` | 네이버 전체 카테고리 스냅샷(전역). `naver_category_id`(PK), `whole_category_name`, `is_leaf`, `is_active`, `raw_json`, `fetched_at`. 상품 등록엔 **활성 리프만** 선택 가능. |
| `naver_brand` | 브랜드 목록(골격). 카탈로그 API 구조 확인 후 컬럼 확정. |
| `naver_taxonomy_sync_log` | 수집 회차 로그(관리자 현황 화면 근거). |
| `naver_taxonomy_schedule` | 수집 주기(단일 행). `interval_hours` 기본 24. |
| `products.naver_category_id` / `naver_brand_id` | 상품이 등록 근거로 삼은 네이버 참조 ID. 몰 카테고리 매핑(`category_id`)과 **별개**. |

수집분에 없어진 항목은 삭제하지 않고 `is_active=0`(soft) — 과거 매핑 추적을 보존한다.

## 3. 커머스 API 연동 (services/sourcing/channel/naverClient.js)

- 인증: 전자서명 OAuth2 `client_credentials`.
  `client_secret_sign = base64( bcrypt(`${clientId}_${timestamp}`, clientSecret) )`.
  (clientSecret 은 네이버가 주는 bcrypt salt 문자열.)
- 토큰: `POST /external/v1/oauth2/token`. `clientId` 기준 메모리 캐시(DB 저장 안 함).
- 카테고리: `GET /external/v1/categories` — 전체 트리 배열. 응답 필드는 버전차를 대비해
  sync 계층(`mapCategory`)에서 방어적으로 매핑(`id/name/wholeCategoryName/last/categoryLevel`).
- 연결 검증: `adapters.validateConnection` 이 네이버는 **실제 토큰 발급**으로 확인(스텁 해소).

## 4. 수집 (services/sourcing/naverTaxonomySync.js + scripts)

- `syncCategories()` : ACTIVE 네이버 자격증명 1건 선택 → 전체 조회 → `naver_category` upsert
  (이번 회차 `fetched_at=runStart`) → 그보다 오래된 행 `is_active=0` → 로그 기록.
  자격증명 없으면 **SKIPPED**(실패 아님).
- 배치: `scripts/sync_naver_taxonomy.js` (`_bootstrap` 먼저). `--scheduled` 는
  `naver_taxonomy_schedule.interval_hours` 도래 시에만 실행.
- 크론: `scripts/naver_taxonomy_cron.sh` — 서버 crontab 에 **한 줄** 고정 등록.
  `*/5 * * * * /data/yd-mall/scripts/naver_taxonomy_cron.sh`
  (주기 변경은 SSH 없이 관리자 화면에서.)

## 5. 관리자 · 상품 등록 연동

- 관리자 메뉴: **외부몰 연동 → 네이버 리소스 관리** (`/admin/sourcing/naver-taxonomy`).
  적재 통계(카테고리·브랜드)·수동 수집·스케줄·**수집 리소스 브라우징(카테고리/브랜드 탭, 검색+페이징)**·회차 로그.
  컨트롤러 `sourcingController`, 뷰 `admin/sourcing/naver_taxonomy`,
  목록 조회는 `naverTaxonomySync.listCategories` / `listBrands`.

  > **통합 이력.** 한때 `몰 관리 → 리소스 관리`(`/admin/resources`)가 같은 네이버 현황·수동수집·검색을
  > 중복 제공했으나, 수집 리소스를 보는 곳을 이 화면 하나로 통합하고 그 메뉴·컨트롤러·라우트·뷰를
  > 제거했다(`scripts/migrate_naver_resource_menu_merge.sql`).
  > **경로는 `naver-taxonomy` 그대로 유지**한다 — 상품 등록 폼이 `/search` 를 호출하고 있어서다.
- 상품 등록 폼(`views/admin/products/form.ejs`): "네이버 카테고리에서 찾기" 검색 위젯.
  선택 시 기존 **직접 입력 칸(`new_category_name`)** 을 네이버 리프명으로 채우고
  hidden `naver_category_id` 를 세팅한다. 저장 시:
  - `taxonomyResolver.resolveOrCreateCategory` 가 몰 카테고리를 생성/매핑 → `products.category_id`.
  - `products.naver_category_id` 에 근거 네이버 ID 저장.
  검색 API: `GET /admin/sourcing/naver-taxonomy/search?q=` (활성 리프만).

## 6. 남은 작업 (Go-live 체크리스트)

1. `scripts/migrate_sourcing_02_naver_taxonomy.sql` 실행(승인 후).
2. `/admin/sourcing/connections` 에서 네이버 `client_id/client_secret` 입력 → **[검증]** → ACTIVE.
3. `/admin/sourcing/naver-taxonomy` 에서 **[지금 수집]** 1회 → 적재 확인.
4. 서버 crontab 에 `naver_taxonomy_cron.sh` 한 줄 등록(자동 주기 수집).
5. 브랜드 수집: 실제 카탈로그/브랜드 API 응답 확인 후 `naver_brand`·adapter 확정.
