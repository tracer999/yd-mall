작성일: 2026-07-13 · 상태: **1차(기본형) 구현 완료 · 활성화 대기**

## 0. 한 줄 요약

| 신설 메뉴 | URL | 정체성 | 새 테이블 |
|---|---|---|---|
| **추천** `RECOMMEND` | `/recommend` | **"나에게 맞는 상품"** — 사람 축(개인화 + MD 큐레이션) | **0개** |

## 1. 배경 — 현재 GNB에 없는 축

현재 GNB 12개(카테고리 제외)는 **상품을 찾는 기준**이 네 가지로 수렴한다.

| 축 | 메뉴 |
|---|---|
| 싸다 | 오늘특가 · 아울렛 · 쿠폰 |
| 인기 있다 | 베스트 · 랭킹 |
| 새롭다 | 신상품 |
| 누가 만들었나 | 브랜드 |
| 한시적 행사다 | 기획전 · 이벤트 · 공동구매 · 쇼핑라이브 |

빠진 것은 두 축이다.

1. **"나에게 맞나"** — 고객 개인을 기준으로 상품을 좁히는 동선이 하나도 없다.
2. **"이 몰은 무엇을 파는 곳인가"** — 기획전은 전부 기간이 있어 **끝난다**. 몰의 정체성을 **상시**로 표현하는 공간이 없다.

각각에 메뉴 하나씩 대응시킨다.

### 2-1. 폐기한 접근 (되풀이 금지)

**❌ 추천 = `productController.getList` 재탕**
`/recommend?sort=best` 같은 목록 한 장은 `gnb_menu_design.md` §0-2가 지적한 **"상품목록 재탕"** 안티패턴 그 자체다. 추천은 "무엇을 보여주는가"가 아니라 **"왜 이걸 너에게 보여주는가"** 가 화면의 본질이므로, 섹션마다 근거 문구를 다는 **전용 랜딩**으로 만든다.

**❌ 전문관 = 새 테이블 + 새 모듈**
`exhibition` 이 이미 필요한 것을 전부 갖고 있다.

| 전문관 요구사항 | `exhibition` 의 현재 상태 |
|---|---|
| 상시 운영(종료 없음) | `end_at` **nullable** — `LIST_ORDER.ending_soon` 이 이미 `end_at IS NULL` 을 뒤로 보내는 코드를 갖고 있다 |
| 고정 대표 URL | `(mall_id, slug)` 유니크 |
| 공통 템플릿 | `detail_template_type` (`TAB_SHOP` / `CATEGORY_SHOP` / `BRAND_SHOP`) |
| 지속적 상품 교체 | `exhibition_product` (섹션·순서·고정·전용 배지·구매 차단까지) |
| 유형 구분 | `exhibition_type` **varchar(50)** — 값 추가에 DDL 불필요 |

새 테이블을 파면 관리자 화면 · 상품 매핑 · 섹션 · 배너 · SEO 를 **전부 두 벌** 유지하게 된다. 얻는 것은 없다.

**❌ `product_recommendations` 를 추천 메뉴의 목록 소스로 쓰기**
이 테이블(573행)은 **상품상세(PDP)용 item-to-item** 연관 데이터다(`product_id` → `related_id`, "함께 보면 좋은 상품"). 기준 상품 없이 단독으로 목록을 만들 수 없다. 추천 메뉴에서는 **개인화 섹션의 씨앗**(최근 본 상품 → 그 상품의 연관 상품)으로만 쓴다.

## 3. 두 메뉴의 경계 — 겹침 해소

| 구분 | 기획전 | **전문관** | **추천** |
|---|---|---|---|
| 축 | 시간(한시) | 시간(상시) | 사람(개인) |
| 운영 기간 | `start_at` ~ `end_at` | `end_at IS NULL` | 없음 |
| 목적 | 행사·매출 촉진 | 상품 탐색 · 몰 정체성 | 발견 · 전환 |
| 상품 구성 | 행사 단위 교체 | 지속 교체 | 자동 산출 + MD 지정 |
| 화면 | 행사별 자유 구성 | 공통 템플릿 | 섹션 랜딩 |
| GNB | 상시 노출(목록) | 상시 노출(목록) | 상시 노출(랜딩) |

**베스트와 추천의 차이**: 베스트는 *"모두에게 인기 있는 것"*(익명·집계), 추천은 *"너에게 맞는 것"*(개인·근거). 비로그인 사용자에게는 이 둘이 가까워지므로, 추천의 비로그인 화면은 **베스트를 복제하지 않고** MD 큐레이션 + 로그인 유도로 채운다(§4-3).

## 4. 추천 `/recommend`

### 4-1. 단계별 모드 (로드맵)

개인화 엔진이 없는 몰도 쓸 수 있어야 한다(빌더 범용성).

| 단계 | 방식 | 상태 | 필요한 것 |
|---|---|---|---|
| **기본형** | 운영자가 상품 직접 지정 | ✅ **1차 구현** | `product_badge='RECOMMEND'` 체크 (기존 관리자 상품 폼) |
| **개인화(얇게)** | 최근 본 상품 → 연관 상품 | ✅ **1차 구현** | `recent_views` + `product_recommendations` (둘 다 존재) |
| 규칙형 | 카테고리·판매량·조회수·할인율 자동 구성 | ⬜ 2차 | 추천 슬롯 정의 테이블(`recommend_slot`) |
| 개인화(제대로) | 구매 이력 · 연령/성별 세그먼트 · 협업 필터링 | ⬜ 3차 | `orders` 22건 · `likes` 11건 — **데이터가 없어 지금은 무의미** |
| AI형 | 취향 질문 · 자연어 검색 · 유사 사용자 | ⬜ 4차 | OpenAI 연동은 이미 있음(상품 설명 생성) |
| 상황별 | "부모님 선물" · "여름휴가 준비" | ⬜ 2차 | **전문관으로 대체 가능** — 상황 큐레이션은 전문관 하나로 만들면 된다 |

> 3차를 지금 하지 않는 이유는 명확하다. 주문 22건 · 좋아요 11건으로 만든 협업 필터링은 **추천이 아니라 난수**다. 데이터가 쌓이기 전에 엔진을 지으면 검증할 수 없는 코드만 남는다.

### 4-2. 데이터 가용성 (2026-07-13 실측, mall_id=1)

| 소스 | 행 수 | 추천에서의 역할 |
|---|---|---|
| `products.product_badge` LIKE `RECOMMEND` | **14건** | MD 추천 섹션 |
| `product_recommendations` | 573행 / 기준상품 **166종** | 개인화 씨앗 → 연관 상품 |
| `recent_views` | 187행 | 개인화 씨앗 → 최근 본 상품 |
| `products.view_count` | 전 상품 | 지금 뜨는 상품 섹션 |
| `orders` | 22건 | ❌ 부족 |
| `likes` | 11건 | ❌ 부족 |

### 4-3. 화면 구성

```
┌────────────────────────────────────────────┐
│ 배너 (group_key = 'menu:RECOMMEND')        │  ← 없으면 섹션 자체가 사라짐
├────────────────────────────────────────────┤
│ ① 회원님을 위한 추천            [로그인 시]│  최근 본 상품 → 연관 상품
│    "최근 보신 «상품명» 과 함께 많이 봤어요" │  이미 본 상품 제외
├────────────────────────────────────────────┤
│ ①' 로그인하고 맞춤 추천 받기  [비로그인 시]│  ①의 자리를 대체하는 CTA 카드
├────────────────────────────────────────────┤
│ ② MD 추천                                  │  product_badge = RECOMMEND
│    "MD가 직접 고른 상품"                   │  관리자 상품 폼에서 체크
├────────────────────────────────────────────┤
│ ③ 지금 많이 보는 상품                      │  view_count DESC
│    "최근 조회가 많은 상품"                 │  ①·② 에 나온 상품 제외(중복 방지)
└────────────────────────────────────────────┘
```

**모든 섹션이 비면** `COMING_SOON.recommend` 준비중 랜딩으로 되돌린다(빈 화면 방지 — 기획전·오늘특가와 같은 폴백 규칙).

### 4-4. 개인화 섹션 알고리즘 (1차)

```
seeds   = recent_views(user) ORDER BY viewed_at DESC LIMIT 5
related = product_recommendations WHERE product_id IN (seeds)
          → 노출가능(PUBLIC, status<>'OFF') 필터
          → 이미 본 상품(recent_views 전체) 제외
          → 씨앗 순서(최근 본 것 우선) × display_order 로 정렬
          → 최대 12건
```

- 결과가 0건이면 (연관 데이터가 없는 상품만 봤을 때) **씨앗 상품의 같은 카테고리 인기 상품**으로 폴백한다.
- 그래도 0건이면 섹션을 그리지 않는다. "추천할 게 없습니다" 는 보여주지 않는다.
- **근거 문구를 반드시 단다.** `"최근 보신 «{씨앗 상품명}»과 함께 많이 본 상품"` — 근거 없는 추천은 베스트와 구별되지 않는다.

### 4-5. 관리자

**신규 관리자 화면 없음.** MD 추천은 기존 **상품 관리 → 상품 뱃지 → `RECOMMEND` 체크**로 관리한다. 별도 큐레이션 화면을 파는 것은 2차(규칙형)에서 `recommend_slot` 을 도입할 때 함께 한다.

## 5. 전문관 `/specialty`

### 5-1. 구현 방식 — `exhibition` 재사용

```sql
-- DDL 없음. exhibition_type 은 varchar(50) 이라 값만 추가하면 된다.
exhibition_type = 'SPECIALTY'
end_at          = NULL          -- 무기한 = 상시
```

`services/exhibition/exhibitionService.js` 의 `TYPES` 상수에 한 줄 추가하는 것으로 관리자 폼에 즉시 나타난다.

### 5-2. 목록 분리 — 두 메뉴가 서로를 오염시키지 않게

| 화면 | 쿼리 |
|---|---|
| `/exhibition` (기획전 목록) | `exhibition_type <> 'SPECIALTY'` — **전문관은 기획전 목록에서 뺀다** |
| `/specialty` (전문관 목록) | `exhibition_type = 'SPECIALTY'` |

이 분리를 하지 않으면 전문관이 기획전 목록에 "종료일 없는 이상한 기획전"으로 섞여 나온다.

**상세 URL 은 전문관 전용으로 갈라놓는다.**

- 전문관 상세 → `/specialty/{slug}`
- 기획전 상세 → `/exhibition/{slug}`
- `/exhibition/{전문관-slug}` 로 들어오면 **301 → `/specialty/{slug}`** (정규 URL 1개 유지. SEO 중복 방지)
- 상세 렌더 코드는 `exhibitionController.getDetail` **하나를 공유**한다. `exhibitionService.decorate()` 가 `exhibition_type` 을 보고 `detailPath` 를 파생하므로, 목록 카드 링크는 자동으로 갈린다.

### 5-3. 관리자

**전용 메뉴 없음.** 기존 `/admin/exhibitions` 에서 유형을 **'전문관 (상시)'** 로 고르고 종료일을 비우면 그것이 전문관이다.

- 관리자 목록에 유형 배지가 이미 있으므로 기획전/전문관 구분이 눈에 보인다.
- 종료일을 비우는 것이 "상시"의 유일한 조건이다. 별도 플래그를 만들지 않는다 — **같은 사실을 두 곳에 저장하면 반드시 어긋난다**(기획전 설계의 `status` vs `phase` 와 같은 원칙).

### 5-4. 화면

전문관 목록은 기획전 목록 뷰를 **재사용하지 않는다.** 기획전 카드에는 `예정 / 진행중 / 종료` 기간 배지와 `종료임박순` 정렬이 있는데, 상시 운영되는 전문관에서는 전부 의미가 없다(전부 "진행중"으로 뜬다).

```
┌────────────────────────────────────────────┐
│ 전문관                                     │
│ 상시 운영되는 테마별 전문 매장             │
├──────────────┬──────────────┬──────────────┤
│ [썸네일]     │ [썸네일]     │ [썸네일]     │
│ 건강기능식품관│ 선물관       │ 프리미엄관   │
│ 상품 42      │ 상품 18      │ 상품 25      │
└──────────────┴──────────────┴──────────────┘
```

발행된 전문관이 0건이면 `COMING_SOON.specialty` 랜딩으로 폴백한다.

### 5-5. 초기 전문관 예시 (운영자가 관리자에서 생성)

이 몰은 건강식품 전문몰이므로 카테고리 복제가 아닌 **탐색 의도** 기준으로 나눈다.

| 전문관 | 성격 |
|---|---|
| 선물관 | 부모님·명절·집들이 — 제안서의 "상황별 추천"을 흡수한다 |
| 프리미엄관 | 고가·프리미엄 라인 |
| 이너뷰티관 | 콜라겐·이너뷰티 |
| 데일리케어관 | 유산균·비타민 등 상시 복용 |

> **씨드 데이터를 코드로 넣지 않는다.** 개발 DB = 운영 DB 이므로 스크립트가 만든 전문관이 그대로 고객에게 노출된다. 전문관 생성은 관리자 화면에서 운영자가 한다.

| `routes/feature.js` | `COMING_SOON` 에 `recommend` · `specialty` 항목 |
| `app.js` | 라우트 마운트 |

### 마운트 순서 함정 (중요)

`routes/feature.js` 는 `app.use('/', featureRoutes)` 로 **가장 먼저** 마운트된다. 따라서 `feature.js` 안에 `router.get('/recommend')` 를 두면 뒤에 오는 `app.use('/recommend', ...)` 가 **영영 닿지 못한다**. 두 메뉴 모두 하위 경로(`/specialty/:slug`)가 필요하므로 **별도 라우터로만** 등록한다. `feature.js` 에는 준비중 랜딩 정의(`COMING_SOON`)만 추가한다.

## 7. 배포 순서 (반드시 지킬 것)

개발 DB = 운영 DB 다. **`module_ready` 를 먼저 올리면 운영 GNB 에 404 링크가 뜬다.**

```
1. 마이그레이션 실행  → feature_menu 2행 (module_ready = 0)
                        mall_feature_menu 4행 (is_enabled = 1)
                        ⇒ 아직 GNB 에 안 뜬다 (navigationService 는 is_enabled AND module_ready)
2. 코드 커밋 → push  → GitHub Actions → 운영 배포
3. 배포 확인 후 활성화 SQL 실행 (§7-1)
```

### 7-1. 활성화 SQL — 배포 확인 후 수동 실행

```sql
UPDATE feature_menu SET module_ready = 1 WHERE feature_code IN ('RECOMMEND','SPECIALTY');
```

### 7-2. ⚠️ GNB 슬롯이 이미 꽉 찼다

`navigation_config.max_gnb_items = 12` 이고, 현재 활성 GNB 기능 메뉴가 **정확히 12개**다(카테고리 제외). 추천·전문관을 켜면 **뒤 2개가 잘려 사라진다**(`navigationService` 가 뒤에서 자른다).

선택지는 둘 중 하나다. **운영 GNB 가 바뀌는 변경이므로 사용자 승인 후 실행한다.**

**(A) 쿠폰·멤버십을 GNB 에서 내린다** — 제안서 권고안. 둘 다 마이페이지에 이미 동선이 있다(`/mypage/coupons`, `/mypage/points`).

```sql
UPDATE mall_feature_menu SET is_enabled = 0
 WHERE mall_id IN (1,2) AND feature_code IN ('COUPON','MEMBERSHIP');
```

**(B) 슬롯을 늘린다** — 14개. GNB 가 복잡해진다.

```sql
UPDATE navigation_config SET max_gnb_items = 14 WHERE mall_id IN (1,2);
```

> 별도로, **랭킹(`RANKING`)은 베스트에 흡수됐다.** 베스트가 랭킹 엔진(`best_ranking` 스냅샷 + 그룹×기간 탭)으로 재설계되면서 `/ranking` 은 준비중 랜딩으로 남아 있는 죽은 메뉴다. 이것부터 내리면 슬롯이 하나 빈다.
