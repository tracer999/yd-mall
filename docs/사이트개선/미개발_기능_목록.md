# 미개발 기능 목록 (매뉴얼 기준)

> ## ✅ 2026-07-23 개발 완료 — Tier 0 전체 · Tier 1 전체 · 2-1 / 2-6 / 2-7
> 아래 5장에서 추린 항목 중 **17건을 구현했습니다.** 각 항목의 완료 표시는 5장을 보세요.
> 1~4장의 전체 목록에서도 해당 항목은 `✅` 로 표시했습니다.
>
> **외부 계약이 필요해 "코드 완료"로만 표시한 2건**
> - **2-6 문자·알림톡** — 발송 코드와 설정 화면까지 완료. 실제 발송은 ① 문자 중계사 계약 + **발신번호 사전등록**(통신사 서류 심사) ② 카카오 채널 개설 + **템플릿 사전승인**(영업일 수일)이 끝나야 나갑니다. 설정이 비어 있으면 조용히 건너뛰므로 주문·클레임 처리에는 영향이 없습니다.
> - **1-3 택배사 API 추적** — 이 중 **배송완료 처리**(수동 일괄 + 발송 N일 후 자동)와 **택배사 배송조회 링크**는 완료. 택배사 API 로 실시간 배송 단계를 받아오는 것은 추적 서비스 계약이 필요해 미완입니다.

`docs/manual/admin` · `docs/manual/user` 전 문서에서 **매뉴얼 스스로 "아직 없다 / 지원하지 않는다"고 밝힌 항목**을 전수 수집한 것입니다.
근거는 36개 문서의 `## 아직 지원하지 않는 것` 전용 섹션 + 본문에 흩어진 미개발 서술입니다.

> 수집 기준 — 매뉴얼이 "없다"고 쓴 것 중 **의도된 설계 결정**(예: "외부 링크는 새 창 고정", "로그인 후 자격 전환 불가")과
> **트러블슈팅 FAQ**(예: "공지를 올렸는데 안 보입니다")는 미개발이 아니므로 제외했습니다.

# 5. 지금 당장 쇼핑몰을 운영해야 한다면 — 반드시 필요한 것

앞의 1~4장은 "매뉴얼이 없다고 밝힌 것 전부"입니다. 그중 **실제 고객을 받아 장사를 시작하는 순간
없으면 운영이 멈추거나 사고가 나는 것**만 따로 추렸습니다. 나머지는 없어도 장사는 됩니다.

판단 기준은 세 가지입니다.
① **매일 발생하는 일인가** ② **없을 때 우회 수단이 있는가** ③ **없으면 금전·법적 사고로 이어지는가**

---

## Tier 0 — 이게 없으면 개점 자체가 불가능

| # | 항목 | 왜 반드시 필요한가 | 현재 우회 수단 |
|---|---|---|---|
| 0-1 ✅ | **몰 도메인별 자동 접속** | 자기 도메인 없이 `?mall=코드` 로 장사할 수 없습니다. 명함·광고·검색 유입 전부 불가. 도메인 칸은 저장만 되고 동작하지 않습니다 | **없음** |
| 0-2 ✅ | **주문번호 · 주문자명 검색** | 고객이 전화로 "제 주문 어떻게 됐나요"를 물었을 때 찾을 방법이 없습니다. 주문 100건만 넘어도 CS 가 마비됩니다. 현재 몰·상태 필터만 있음 | 브라우저 Ctrl+F (페이지 넘어가면 끝) |
| 0-3 ✅ | **주문 목록 엑셀 내려받기** | 택배사 접수 파일 생성, 정산, 회계 제출의 시작점입니다. 이게 없으면 출고 자체를 못 돌립니다 | **없음** |
| 0-4 ✅ | **송장 일괄 등록(엑셀 업로드)** | 택배사가 돌려주는 송장 CSV 를 그대로 올리는 것이 표준 흐름입니다. 하루 30건만 넘어도 한 건씩 손으로 넣는 것은 불가능 | 건별 수기 입력 |
| 0-5 ✅ | **부분 취소 · 부분 반품** | 3개 주문 중 1개만 반품하는 것은 **매일 일어납니다.** 현재는 주문 단위라 전체 취소 후 재주문을 유도해야 하고, 이 과정에서 결제·재고·적립금이 어긋납니다 | 전체 취소 → 재주문 (사고 위험) |
| 0-6 ✅ | **교환** | 사이즈·불량 교환은 반품만큼 흔합니다. "반품 후 재주문 안내"는 고객이 배송비를 두 번 내는 구조라 분쟁이 됩니다 | 반품 + 재주문 안내 |

## Tier 1 — 개점 후 1~2주 안에 반드시 부딪히는 것

| # | 항목 | 언제 터지나 |
|---|---|---|
| 1-1 ✅ | **회원 비밀번호 초기화 · 회원 정보 수정** | "비밀번호를 못 찾겠어요", "주소를 잘못 넣었어요" — 관리자가 손댈 수 있는 수단이 전혀 없습니다 |
| 1-2 ✅ | **관리자 비밀번호 찾기 · 재발급** | 운영자가 자기 비밀번호를 잊으면 **개발자를 불러 DB 를 고쳐야 합니다.** 본인이 스스로 바꾸는 기능도 없습니다 |
| 1-3 ◑ | **배송 완료 자동 처리** ✅ · 택배사 API 추적(계약 필요) | 현재는 송장 번호를 기록만 합니다. 실제 배송이 끝나도 주문은 계속 `배송중` 이라, 구매확정·정산·리뷰 요청이 전부 멈춥니다 |
| 1-4 ✅ | **반품 회수 송장 등록 · 회수 추적** | 반품을 승인해도 물건이 언제 돌아오는지 시스템이 모릅니다. 환불 시점 판단을 사람 기억에 의존하게 됩니다 |
| 1-5 ✅ | **대시보드 매출 금액** | 지금은 주문 **건수**만 나옵니다. 사장이 "오늘 얼마 팔았나"를 볼 화면이 없습니다 |
| 1-6 ✅ | **약관 동의 이력 조회 + 예약 시행** | 개인정보 관련 분쟁·점검 시 "이 회원이 언제 무엇에 동의했는가"를 제시해야 합니다. 조회 화면이 없습니다. 약관 개정 시 **시행일 예약**도 안 돼 자정에 사람이 눌러야 합니다 |
| 1-7 ✅ | **포인트 유효기간(소멸) 관리** | 적립금이 소멸 없이 무한 누적되면 회계상 부채가 계속 쌓입니다. 나중에 도입하면 이미 지급된 포인트에 소급 적용이 어렵습니다 |
| 1-8 ✅ | **주문서 · 거래명세서 출력** | 사업자 고객(B2B)은 거의 항상 요구합니다. B2B 기능을 켜고 파는 이상 필수 |

## 정리 — 개점 최소 요건

> **Tier 0 여섯 가지(도메인 접속 · 주문 검색 · 주문 엑셀 · 송장 일괄등록 · 부분취소/반품 · 교환)는
> 대체 수단이 없어 우회가 불가능합니다.** 이 여섯 개를 채우면 "장사는 돌아가는" 상태가 되고,
> Tier 1 여덟 가지를 채우면 "CS 와 정산이 사람 기억에 의존하지 않는" 상태가 됩니다.

Tier 0 + Tier 1 = **14개 항목**이 실질적인 운영 개시 요건입니다.
전체 미개발 목록(1~4장 약 180개) 대비 **8% 수준**이므로, 나머지는 운영을 시작한 뒤 순차적으로 채워도 됩니다.
