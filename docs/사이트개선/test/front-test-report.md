## 2. 발견된 문제

### 2.1 [치명] `/checkout/fail` 이 **항상 500** — 결제 실패 고객이 안내 화면을 볼 수 없다 ✅ **수정 완료**

결제가 실패하면 고객이 도착하는 페이지가 **어떤 경우에도 렌더되지 않는다.** 뷰의 EJS 문법 오류라 `reason` 값과 무관하게 컴파일 단계에서 터진다.

```
GET /checkout/fail → 500
Unexpected token 'else' in views/user/checkout/fail.ejs while compiling ejs
```

원인은 `views/user/checkout/fail.ejs:8-12` 에서 **`if` 블록을 먼저 닫아 버려 뒤따르는 `else` 가 고아가 된 것**이다.

```ejs
<% if (reason === 'cancel') { %>결제가 취소되었습니다.<% } %>      ← 여기서 이미 닫힘
<% else if (reason === 'stock') { %>재고가 부족하여 …<% } %>       ← 짝 없는 else
<% else if (reason === 'amount') { %>결제 금액이 일치하지 않습니다.<% } %>
<% else if (reason === 'config') { %>결제 설정이 완료되지 않았습니다.<% } %>
<% else { %>오류가 발생했습니다. 잠시 후 다시 시도해 주세요.<% } %>
```

**수정 방향** — 블록을 이어 붙인다.

```ejs
<% if (reason === 'cancel') { %>결제가 취소되었습니다.
<% } else if (reason === 'stock') { %>재고가 부족하여 결제를 완료할 수 없습니다.
<% } else if (reason === 'amount') { %>결제 금액이 일치하지 않습니다.
<% } else if (reason === 'config') { %>결제 설정이 완료되지 않았습니다.
<% } else { %>오류가 발생했습니다. 잠시 후 다시 시도해 주세요.
<% } %>
```

**영향 범위가 넓다.** `/checkout/success` 도 파라미터가 없으면 `/checkout/fail?reason=missing` 으로 보내므로, 그 경로도 함께 500이 된다. 즉 **결제 이탈·취소·재고부족·금액불일치 전 상황에서 고객이 흰 오류 화면을 보게 된다.**

> **수정 결과** — 위 방향대로 블록을 이어 붙였다. 5개 분기 전부 200 + 올바른 메시지를 확인했다.
>
> | `reason` | 출력 |
> | --- | --- |
> | `cancel` | 결제가 취소되었습니다. |
> | `stock` | 재고가 부족하여 결제를 완료할 수 없습니다. |
> | `amount` | 결제 금액이 일치하지 않습니다. |
> | `config` | 결제 설정이 완료되지 않았습니다. |
> | 그 밖 · 없음 | 오류가 발생했습니다. 잠시 후 다시 시도해 주세요. |

> 같은 유형(`<% } %>` 뒤에 `<% else`)이 다른 뷰에도 있는지 확인하기 위해 **전체 뷰 298개를 EJS 컴파일**했다. 결과는 이 파일 1건 + 미사용 partial 2건(관리자 리포트 §2.2)뿐이다.

### 2.2 [높음] 몰 빌더로 만든 신규 몰의 **홈 히어로가 렌더되지 않는다** ✅ **수정 완료**

몰을 만들면 홈 최상단이 비어 있고 바로 "베스트" 섹션부터 시작한다. 테마1(좌 히어로 + 우 상품 카드)을 골라도 히어로가 없다.

**원인은 프로비저닝과 샘플 시딩의 축이 어긋난 것이다.** `theme_hero` 는 배치와 콘텐츠를 두 축으로 나눠 쓴다.

| 축 | 저장 위치 | 신규 몰의 값 |
| --- | --- | --- |
| ① 배치 | `page_section.config_json.layout` | `split_feature` ✅ 프리셋대로 설정됨 |
| ② 콘텐츠 | `site_settings.hero_variant` | **`full_banner`** ← 컬럼 DEFAULT 값이 그대로 |

`full_banner` 는 **`banners`(banner_type='MAIN')** 를 읽는데, 샘플 시더(`services/mall/sampleSeeder.js`)는 **`hero_slide` 만 채우고 `banners` 는 채우지 않는다**. 리졸버는 "노출할 게 하나도 없으면 섹션을 통째로 스킵"하는 설계라, 결과적으로 히어로가 사라진다.

실제 확인한 값 (mall_id=32):

```
site_settings.hero_variant = 'full_banner'
banners      WHERE mall_id=32 AND banner_type='MAIN'  →  0 건
hero_slide   WHERE mall_id=32 AND slot='MAIN'         →  4 건 (전부 is_active=1)
page_section  theme_hero  sort_order=1, is_active=1
```

`mallProvisioner` 는 `hero_variant` 를 전혀 설정하지 않아 DB 기본값 `full_banner` 가 그대로 남는다(`site_settings.hero_variant` 컬럼 DEFAULT = `full_banner`). 새로 만든 두 몰(32·33) 모두 동일했다.

**영향**: 몰 빌더의 첫인상이 "히어로 없는 홈"이 된다. 납품받은 사용자가 원인을 스스로 찾기 어렵다 — 히어로 슬라이드는 등록돼 있는데 화면에는 안 나오기 때문이다.

> 참고: 페이지빌더의 섹션 미리보기(`?type=theme_hero`)는 정상 렌더된다. 미리보기는 별도 샘플 config 를 쓰기 때문으로, **미리보기가 정상이라 문제를 알아채기 더 어렵다.**

**수정 결과** — `mallProvisioner.applySiteSettings()` 가 `site_settings` 행을 만들 때 콘텐츠 축을 함께 정하도록 했다. **MAIN 슬라이드가 있고 MAIN 배너가 없을 때만** `product_showcase` 로 맞추므로, 배너를 이미 등록해 둔 몰의 선택은 덮지 않는다. `INSERT IGNORE` 라 기존 몰의 설정도 건드리지 않는다.

처음에는 `sampleSeeder` 쪽에 넣었는데 동작하지 않았다. **샘플 시딩이 프로비저닝보다 먼저 돌아** 그 시점엔 `site_settings` 행이 아직 없어 `UPDATE` 가 0행이었기 때문이다. 그래서 행을 만드는 주체인 프로비저너로 옮겼다.

> 시더에 남겼던 보정 코드는 **삭제했다.** `seedSampleData()` 호출부를 전수 확인한 결과 **몰 생성 경로 한 곳뿐**이라(기존 몰에 샘플을 다시 심는 UI 는 없다 — `[샘플 데이터 관리]` 화면은 샘플 *카탈로그* 를 관리한다) 도달할 수 없는 코드였다. 같은 로직이 두 곳에 흩어지면 나중에 어긋나므로, 프로비저너 한 곳에서만 결정하고 시더에는 왜 여기서 하지 않는지 주석만 남겼다.

검증: 같은 조건(테마1 + 샘플 포함)으로 몰을 새로 만들어 `hero_variant=product_showcase` 로 저장되는 것과, 홈 최상단이 **좌 히어로 + 우 상품 카드**로 렌더되는 것을 스크린샷으로 확인했다. 콘솔·네트워크 오류 0건.

### 2.3 [중간] "최근 본 상품"이 **몰 경계를 넘는다** (실측 확인) ✅ **수정 완료**

`qatest` 에서 상품을 본 뒤 `qaempty`(상품 0건인 빈 몰) 홈에 들어가면, **다른 몰에서 본 상품이 "최근 본 상품" 섹션에 그대로 노출**된다.

**클라이언트·서버 두 경로를 분리해 실측했다.**

| 조건 | 결과 |
| --- | --- |
| 비로그인 + localStorage 있음 | 빈 몰 홈에 타 몰 상품 노출 (클라이언트 경로) |
| **로그인 + `localStorage.clear()` 후 재방문** | **여전히 노출** → **서버측 누출 확정** |

두 번째 조건은 브라우저에서 localStorage 를 완전히 비운 뒤 빈 몰 홈을 다시 연 것으로, 클라이언트 이력이 전혀 없는 상태에서도 서버가 타 몰 상품을 내려보낸다는 뜻이다. DB 에도 몰 구분 없이 기록돼 있다.

```
recent_views  id=526  user_id=51  product_id=11146   (해당 상품의 products.mall_id = 32)
→ mall_id=33 (qaempty) 홈에서 그대로 노출됨
```

원인이 클라이언트·서버 양쪽에 있다.

| 경로 | 문제 |
| --- | --- |
| 비로그인 | `views/partials/sections/recent_product.ejs:78` — `localStorage.getItem('yd_recent_products')` 를 **몰 구분 없는 단일 키**로 읽는다 |
| 로그인 | `services/display/resolvers/recent_product.js` 의 쿼리에 **`p.mall_id` 조건이 없다**. `recent_views` 테이블에도 `mall_id` 컬럼이 없다(`id · user_id · product_id · viewed_at`) |

다른 리졸버들은 `WHERE p.mall_id = ?` 로 몰을 격리한다(`_shared.js` 참고). 이 섹션만 빠져 있다.

**영향**: 몰 A 상품이 몰 B 홈에 뜨고, 클릭하면 그 몰에 없는 상품 페이지로 이동한다. 서로 다른 고객사에 납품된 몰이 한 브라우저에서 열릴 때 특히 어색하다.

**수정 결과** — 두 경로를 모두 막았다.

| 경로 | 수정 |
| --- | --- |
| 서버 | `recent_product.js` 쿼리에 `AND p.mall_id = ?` 추가 (`shared.mallId` — 다른 리졸버와 같은 규칙) |
| 클라이언트 | localStorage 키를 `yd_recent_products:<mallId>` 로 분리. 기록(`products/detail.ejs`)·우측 레일(`right_utility.ejs`)·홈 섹션(`recent_product.ejs`) 3곳을 같은 규칙으로 통일 |

검증(브라우저 실측):

| 조건 | 수정 전 | 수정 후 |
| --- | --- | --- |
| 로그인 + localStorage 비움 → 타 몰 홈 | 타 몰 상품 노출 | **노출 안 됨** |
| 비로그인 몰A 열람 → 몰B 홈 | 타 몰 상품 노출 | **노출 안 됨** (키: `yd_recent_products:32`) |
| 비로그인 몰A 열람 → **몰A 홈** | 노출 | **정상 노출** (기능 유지) |

> `recent_views` 테이블에 `mall_id` 컬럼을 추가하는 것은 스키마 변경이라 이번에는 하지 않았다. 상품의 `mall_id` 로 거르는 현재 방식으로 충분히 격리된다.

### 2.4 [낮음] favicon 이 404 — 모든 페이지에서 콘솔 오류 ✅ **수정 완료**

```
GET /favicon.ico → 404
```

신규 몰은 파비콘이 설정돼 있지 않아 브라우저 탭 아이콘이 비고, 전 페이지에서 콘솔에 404가 남았다. 레이아웃이 `favicon_url`·`logo_url` 이 있을 때만 `<link rel="icon">` 을 넣고, 없으면 아무것도 넣지 않아 브라우저가 기본 `/favicon.ico` 를 찾아가기 때문이다.

**수정 결과** — 중립적인 쇼핑백 아이콘 `public/favicon.svg` 를 추가하고, `main_layout`·`admin_layout` 양쪽에 폴백을 걸었다. 운영자가 파비콘·로고를 등록하면 그쪽이 그대로 우선한다. 신규 몰 홈에서 콘솔·네트워크 오류 0건을 확인했다.

### 2.5 [낮음] 신규 몰 푸터 소개 문구가 **기본몰 문구 그대로** ✅ **수정 완료**

`QA빈몰테스트` 푸터에 아래 문구가 그대로 노출된다.

```
자연에서 온 순수한 재료로 만드는 건강한 식탁.
와이디몰은 당신의 건강한 라이프스타일을 응원합니다.
```

몰 이름(`QA빈몰테스트`)은 정상 치환되는데 **소개 문구만 기본값**이라, 다른 업종 몰을 만들면 "건강식품" 문구가 남았다. DB 값이 아니라 `views/layouts/main_layout.ejs` 의 **하드코딩 폴백**이었다(신규 몰의 `site_settings.slogan` 은 `NULL`).

**수정 결과** — 폴백을 업종·브랜드 중립 문구(`<회사명> 쇼핑몰에 오신 것을 환영합니다.`)로 바꿨다. 운영자가 `[사이트 설정]` 에서 슬로건을 입력하면 그쪽이 우선한다.

### 2.6 [낮음] noindex 정책이 sitemap 과 rss 에 다르게 적용된다 — **미수정(정책 판단 필요)**

| 경로 | 색인 차단 상태에서 |
| --- | --- |
| `/sitemap.xml` | 404 (`isIndexingAllowed()` 가 막음) — 의도된 동작 |
| `/rss.xml` | **200** (52KB, 상품 목록 전체 노출) |

sitemap 을 막는 이유가 "이 서버는 색인되면 안 된다"라면 rss 도 같은 기준을 따르는 것이 일관적이다.

**손대지 않은 이유** — 이건 버그가 아니라 **정책 선택**이다. RSS 피드는 검색 색인 산출물이라기보다 구독용 배포 채널이라, "색인 차단 = 피드도 차단"이 자명하지 않다. 지금 피드를 소비하는 쪽이 있으면 막는 순간 끊긴다. 어느 쪽이 맞는지는 운영 판단이라 임의로 바꾸지 않았다.

막기로 정한다면 `routes/sitemap.js` 의 `/rss.xml` 핸들러 앞에 `/sitemap.xml` 과 같은 `isIndexingAllowed()` 가드를 한 줄 넣으면 된다.

## 3. 정상 확인 항목

### 3.1 전 라우트 접근

프론트 GET 라우트 87개를 전수 프로브했다. `/checkout/fail`(§2.1) 을 제외하고 전부 정상이며, 인증이 필요한 화면(`/cart` · `/mypage/*` 12종 · `/inquiries` · `/quotes` · `/b2b/*`)은 **예외 없이 `/auth/login` 으로 리다이렉트**되어 접근 제어가 지켜지고 있었다. 이 화면들의 실제 렌더는 §3.5 에서 로그인 상태로 별도 확인했다.

정상 렌더 확인: 홈 · 상품목록 · 상품상세 · 검색 · 베스트 3종 · 신상품 · 쇼핑특가 · 공동구매 · 쿠폰 · 멤버십 · 기획전 · 아울렛 · 전문관 · 추천 · 이벤트 · 라이브 · 브랜드 · 공지 · 고객센터 2종 · 약관 3종 · 이용안내 · 로그인 · 회원가입 · 결제수단 선택.

### 3.2 온라인 매뉴얼 63건

`/manual/admin/*` 40건, `/manual/user/*` 18건, `/manual/mall_builder/*` 5건 전부 정상 렌더. 깨진 문서·404 없음.

### 3.3 빈 몰(데이터 0건)에서의 동작 — 납품 직후 상태

샘플 데이터를 넣지 않은 `qaempty` 로 주요 18개 화면을 확인했고 **전부 200 + 정상 렌더**됐다. 상품·특가·기획전·이벤트·라이브·공지가 모두 0건인 상태에서도 화면이 깨지지 않는다.

```
/ · /products · /best · /new · /deals · /brands · /search · /cs · /outlet
/exhibition · /event · /coupon · /membership · /group-buy · /live · /specialty
/recommend · /notices
```

다만 홈은 §2.2 의 히어로 문제와 데이터 0건이 겹쳐 **사실상 빈 화면**에 가깝다(본문 텍스트 326자). 렌더 오류는 아니지만, 납품 직후 첫 화면으로는 아쉽다.

### 3.4 브라우저 실렌더 · 콘솔 에러

| 화면 | 결과 |
| --- | --- |
| 홈 / 상품 상세 / 상품 목록 / 쇼핑특가 / 빈 몰 홈 / 고객센터 / 로그인 / 회원가입 | **JS 에러 0건** (favicon 404 제외) |

몰 전환(`?mall=`)도 정상 동작한다. 몰 코드는 세션에 고정되므로, 리다이렉트로 쿼리가 사라져도(`/deal/today` → `/deals`, `/ranking` → `/best`) 몰 컨텍스트가 유지된다.

### 3.5 로그인 고객 전용 화면 — 실제 렌더 확인

가드(로그인 리다이렉트)만이 아니라 **실제 화면**을 확인하기 위해, `/auth/signup` 으로 테스트 고객을 만들어 로그인 상태로 전부 열어 봤다.

| 확인 방식 | 대상 | 결과 |
| --- | --- | --- |
| HTTP 프로브 | 마이페이지 12종 · 장바구니 2종 · 문의 2종 · B2B 2종 · 견적 · 결제 진입 | 21건 전부 정상 |
| 브라우저 실렌더 + 콘솔 수집 | 마이페이지 · 주문내역 · 포인트 · 쿠폰 · 찜 · 프로필 · 리뷰 · 클레임 · 최근본상품 · 장바구니 · 문의작성 · B2B신청 · 회원탈퇴 | 13화면 **JS 에러 0건** |

주문·포인트·쿠폰·리뷰가 모두 0건인 신규 고객 상태에서도 화면이 깨지지 않고 빈 상태 안내가 정상 출력됐다.

**장바구니 담기도 실제로 동작**한다. 상품(11146)을 담아 `carts` 테이블에 정상 기록되고 장바구니 화면에 반영되는 것까지 확인했다.

> `/checkout` 을 직접 열면 `/products` 로 보내는데, 이는 결제 화면이 **주문 대상(상품 또는 선택 항목)을 받아야 열리는** 구조이기 때문이다(장바구니에서 [주문하기] POST 로 진입). 결함이 아니다.

## 5. 테스트 중 생성한 데이터 — 정리 방법

### 5.1 테스트 몰 2개

관리자 리포트 §5 와 동일하다. `qatest`(32) · `qaempty`(33) 두 몰 모두 기본몰이 아니므로 **`[몰 관리 > 몰 리스트 관리]` 에서 삭제**할 수 있다. 별도 스크립트·SQL 은 필요 없다.

### 5.2 테스트 고객 계정 1개

로그인 고객 화면(§3.5)을 검증하려면 계정이 필요해, **`/auth/signup` 회원가입 폼으로** 아래 계정을 만들었다.

| 항목 | 값 |
| --- | --- |
| 이메일 | `qa.tester.0724@example.com` |
| 이름 | QA테스트고객 |
| `users.id` | 51 |

이 계정이 남긴 데이터는 **`carts` 1건(상품 11146), `recent_views` 1건**뿐이다. 계정은 관리자 `[회원 관리]` 에서 삭제할 수 있다. 그 밖에 주문·결제·리뷰는 생성하지 않았다.

> DB 를 직접 수정해 만든 데이터는 없다. 몰 2개와 고객 1개 모두 **화면의 폼**으로 생성했다.
