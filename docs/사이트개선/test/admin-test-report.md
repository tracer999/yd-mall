## 0. 판정 기준 — "200 OK"를 통과로 보지 않았다

이번 테스트의 출발점이 된 아울렛 화면 오류는 **HTTP 200을 반환하면서 화면만 깨지는** 유형이었다.
관리자 기본 레이아웃이 전역으로 지정돼 있지 않아(`app.js:105` 이 `layouts/main_layout` = 사용자용으로 설정), 컨트롤러가 `layout: 'layouts/admin_layout'` 을 빼먹으면 **관리자 화면이 고객 페이지 헤더/푸터에 감싸인 채 200으로 응답**한다.

따라서 모든 화면에 대해 아래를 함께 확인했다.

| 검사 | 방법 |
| --- | --- |
| 관리자 레이아웃으로 렌더됐는가 | 응답 HTML에 `id="sidebarMenu"` 존재 |
| 고객 레이아웃이 잘못 씌워졌는가 | 응답 HTML에 `id="scrollTopBtn"` 부재 |
| 렌더 중 예외가 문자열로 새어나왔는가 | `Server Error` / `Cannot read propert…` / `ER_XXX` / `>undefined<` / `NaN원` / `Invalid Date` 패턴 |
| 브라우저에서 실제로 동작하는가 | 콘솔 JS 에러 · 4xx/5xx 리소스 요청 수집 |

## 2. 발견된 문제

### 2.1 [확인] 아울렛 화면 오류는 **코드는 고쳐졌으나 서버에 반영되지 않은 상태**였다

- 테스트 시작 시점 `/admin/outlet`, `/admin/outlet/categories` 는 **여전히 고객 레이아웃으로 렌더**되고 있었다(200 응답, `sidebarMenu` 없음 / `scrollTopBtn` 있음).
- 원인은 코드가 아니라 **구동 중이던 서버가 옛 코드**였던 것. 프로세스 기동 시각 `00:01:45` vs `controllers/admin/outletController.js` 수정 시각 `10:44:00`. 이 서버는 `node app.js` 직접 실행이라 nodemon 자동 반영이 없었다.
- **최신 코드로 재기동한 뒤 두 화면 모두 정상**(사이드바·헤더·빈 상태 안내 정상 렌더)임을 HTTP 마커와 스크린샷으로 확인했다.

> 참고: 테스트를 위해 로컬 서버를 재기동했다(`node app.js`, PID 903724). 코드 변경은 하지 않았다.

### 2.2 [중간] 미사용 partial 2건이 EJS 컴파일 불가 — include 하는 순간 500 ✅ **수정 완료**

전체 뷰 298개를 컴파일한 결과 아래 2건이 깨져 있다. **현재 어디에서도 include 되지 않아 사용자 영향은 없으나, 재사용하려고 include 하는 순간 그 화면 전체가 500** 이 된다.

| 파일 | 오류 |
| --- | --- |
| `views/partials/modal_overlay.ejs` | `Could not find matching close tag for "<%-"` |
| `views/partials/user_search_modal.ejs` | `missing ) after argument list` |

원인은 동일하다. 두 파일 모두 **EJS가 지원하지 않는 `<%-- … --%>` 주석 문법**(JSP/ASP 문법)으로 시작한다. EJS 주석은 `<%# … %>` 다. 파서가 `<%-`(raw 출력)로 읽어 뒤 내용을 표현식으로 해석하면서 깨진다.

```
views/partials/modal_overlay.ejs:1
<%--                       ← EJS 는 이 문법을 모른다
  공통 모달 오버레이
  - body: <%- body %> ...  ← 주석이 아니므로 이 태그까지 파싱된다
--%>
```

**수정 결과** — 두 파일의 주석을 EJS 주석(`<%# … %>`)으로 바꾸고, 왜 JSP 식 주석을 쓰면 안 되는지 한 줄 남겼다. **뷰 298개 전부 컴파일 통과**(수정 전 3건 실패 → 0건).

> 작업 중 겪은 함정 하나: EJS 주석 **안에 `%>` 문자열이 들어가면 거기서 주석이 닫힌다.** 설명에 태그 예시를 그대로 적었다가 같은 오류가 재발했다. 문구를 풀어 쓰는 방식으로 해결했다.

### 2.3 [낮음] 외부몰 자격증명 미설정에 500을 반환 — 같은 상황의 다른 API는 400 ✅ **수정 완료**

`[외부몰 연동 > 상품 가져오기]` 화면(`/admin/sourcing/import`)에 **들어가기만 해도** 브라우저 콘솔에 500이 찍힌다.

| 엔드포인트 | 자격증명 없을 때 | 응답 본문 |
| --- | --- | --- |
| `/admin/sourcing/domeggook-categories` | **500** | `도매꾹 자격증명이 없습니다 — [공급처/채널 연결]에서 Open API Key 를 등록하세요.` |
| `/admin/sourcing/publish/addressbooks` | 400 | `네이버 자격증명이 없습니다. …` |

메시지 자체는 사용자가 할 일을 정확히 안내하고 있어 좋다. 다만 **"설정이 아직 안 됨"은 서버 장애가 아니므로 500이 아니라 400번대**가 맞고, 바로 옆 기능(네이버)이 이미 400을 쓰고 있어 **일관성도 어긋난다**. 500은 모니터링·로그에서 진짜 장애와 섞인다.

**수정 결과** — `importService.resolveCredential()` 이 던지는 에러에 `code = 'NO_CREDENTIAL'` 을 붙이고, 컨트롤러가 그 코드일 때 **400** 을 반환하도록 했다. 메시지 문자열로 분기하지 않으므로 다른 호출부에서도 같은 방식으로 쓸 수 있다.

검증: 자격증명이 **없는** 몰에서 `400` + 기존 안내 메시지, 자격증명이 **있는** 몰에서는 종전대로 `200` + 카테고리 트리를 확인했다.

### 2.4 [낮음] 주문서 인쇄 화면의 마크업이 불완전 ✅ **수정 완료**

`/admin/sales/:id/invoice` 응답이 `<title>` 로 시작했다. `layout: false` 로 렌더되는 독립 문서인데 `<!DOCTYPE html>` · `<html>` · `<head>` 선언이 없어, 브라우저가 quirks mode 로 해석하면 인쇄 레이아웃이 환경에 따라 달라질 수 있었다. (관리자 레이아웃이 없는 것 자체는 인쇄용이라 정상)

**수정 결과** — `<!DOCTYPE html>` · `<html lang="ko">` · `<head>`(charset·viewport 포함) · `<body>` 를 갖춘 온전한 문서로 만들었다. 실제 응답이 `<!DOCTYPE html>` 로 시작하는 것을 확인했다.

### 2.5 [낮음] 라우트 디렉터리에 남은 뷰 파일 잔재 5개 ✅ **수정 완료**

`routes/` 는 라우트만 있어야 하는데 EJS 파일이 섞여 있었다. **어느 코드에서도 참조되지 않아**(전수 grep 재확인) 삭제했다. 최초 커밋부터 있던 잔재다.

```
routes/detail.ejs        routes/layout.ejs        routes/list.ejs
routes/admin/detail.ejs  routes/admin/list.ejs
```

### 2.6 [낮음] 관리자 화면 6곳이 외부 CDN에 의존 — **미수정(별도 작업 권장)**

`/admin/products/seo/view/:id` 가 아이콘을 `https://cdn.jsdelivr.net/npm/bootstrap-icons@1.10.0/…` 에서 불러온다. 폐쇄망·오프라인 납품 환경에서 아이콘이 전부 깨진다.

다만 **이 화면만의 문제가 아니다.** 확인해 보니 아래 6개 화면이 같은 CDN을 쓴다.

```
views/admin/products/seo_preview.ejs   views/admin/login.ejs
views/admin/dashboard.ejs              views/admin/forgot_password.ejs
views/admin/visitors/stats.ejs         views/admin/traffic_sources_detail.ejs
```

`bootstrap-icons` 는 현재 `node_modules` 에 없어서, 고치려면 **패키지 추가 + 폰트 파일 배포 + 6개 화면 일괄 수정**이 필요하다. 이번 수정 범위(발견된 버그 정정)를 넘어서고 빌드 산출물에도 영향을 주므로 **손대지 않았다.** 폐쇄망 납품이 예정되면 별도 작업으로 진행하기를 권한다.

## 3. 정상 확인 항목

### 3.1 전 메뉴 접근 · 렌더

`admin_menus` 에 등록된 66개 메뉴 전체가 200 + 관리자 레이아웃으로 응답했다. 아래는 데이터가 없는 신규 몰에서도 화면이 깨지지 않는지가 관건인 항목들로, **전부 빈 상태 안내가 정상 출력**됐다.

수정을 마치고 서버를 재기동한 뒤, 처음과 **같은 방법으로 전부 다시 돌렸다.**

| 검증 | 수정 전 | 수정 후 |
| --- | --- | --- |
| EJS 뷰 컴파일 (298개) | 실패 3건 | **실패 0건** |
| 관리자 메뉴 66건 | 문제 0건 | **문제 0건** |
| 파라미터 없는 관리자 라우트 132건 | 문제 6건(전부 오탐·정상) | **동일** (`domeggook` 만 500→400) |
| 관리자 상세·수정 27건 | 문제 2건(독립 문서) | **동일** |
| 프론트 라우트 87건 | `/checkout/fail` 500 | **500 소멸** |
| 브라우저 실렌더 20화면 | JS 오류 2화면 | **1화면** (자격증명 미설정 400 — 의도된 응답, 화면에 안내 표시됨) |

**새로 생긴 문제는 없다.** 수정 후에도 남은 항목은 전부 §4 에 오탐·정상으로 분류해 둔 것들이다.

## 7. 스코프 한계 (과대 해석 방지)

- `qatest` 는 몰 등록 폼의 기본값대로 **샘플 데이터를 포함해** 생성했다. 따라서 "데이터 0건에서의 정상 동작"은 `qaempty`(§5) 로 별도 확인했고, 그 결과는 프론트 리포트 §3.3 에 있다.
- **관리자 쪽 저장(POST) 동작은 몰 생성 2건 외에는 실행하지 않았다.** 이번 테스트는 전 화면의 렌더·접근·오류 노출 검증이 목적이다. (고객 쪽은 회원가입·장바구니 담기까지 실제로 저장해 확인했다 — 프론트 리포트 §3.5)
- **외부몰(네이버·도매꾹·Shopify) 실전송은 일절 하지 않았다.** 관련 화면은 조회·렌더만 확인했다.
