# 네이버 기반 글로벌 카테고리 재구성 설계

> 목적: 우리 **글로벌 카테고리**(`categories`, `type='NORMAL'`, `mall_id=0`)의 **뼈대를 네이버 스마트스토어 표준 분류(L1~L3)로 재구성**한다. 이후 신규 상품 등록 시 네이버 참조가 우리 카테고리에 정확히 매핑되고, 네이버 주기 수신분이 우리 카테고리에 지속 반영되게 한다.
>
> 짝 문서: [`카테고리_브랜드_글로벌화_설계.md`](./카테고리_브랜드_글로벌화_설계.md), [`네이버_카테고리_리소스_설계.md`](./네이버_카테고리_리소스_설계.md), [`네이버_마스터데이터_수신_설계.md`](./네이버_마스터데이터_수신_설계.md)

---

## 0. 정책 전환 (중요 — 기존 원칙 뒤집기)

이 설계는 짝 문서들의 아래 원칙을 **의도적으로 전환**한다. 기존 문서를 볼 때 혼동하지 않도록 명시한다.

| 항목 | 기존 원칙 | 이번 전환 |
|---|---|---|
| 카테고리 생성 시점 | "분류를 먼저 만들지 않는다. **상품 등록 행위가** 카테고리를 만든다" (`product_easy_registration_design_and_development.md`) | **네이버 L1~L3 표준 뼈대를 선(先)-시드**한다. 상품 등록은 이제 "생성"보다 **"기존 표준 노드에 매칭"**이 기본 |
| 네이버 → 우리 반영 | "`naver_category`는 자동 반영 안 함. 순수 참조 리소스" (`네이버_카테고리_리소스_설계.md`) | 네이버 → 우리 글로벌 카테고리로 **선-시드 + 지속 동기화**(additive) |
| 우리 것 vs 네이버 | "우리 것은 우리 것, 네이버는 별도" | **뼈대는 네이버 표준을 채택**. 단 우리 고유(사용자 생성)분은 병존·보존 |

전환의 이유: 현재 우리 글로벌 카테고리(389개)는 네이버와 정렬돼 있지 않아(대분류 정확 일치 4/31, 뼈대 상이), 상품 등록 시 네이버 경로가 엉뚱한 곳에 매칭되거나 카테고리를 남발한다. 표준 뼈대를 먼저 세워 이 어긋남을 제거한다.

## 1. 확정 규칙 (사용자)

1. **뼈대 = 네이버 L1~L3.** 네이버 표준 분류 2,347개(L1 11 · L2 242 · L3 2,094)를 우리 글로벌 카테고리로 세운다. 우리 몰은 **최대 3단계**이므로 네이버 L4(3,468개)는 시드하지 않는다(전송 시점 선택 — 짝 문서 §32-4).
2. **교체 = 뼈대 재구성이지 데이터 파괴가 아님.** 기존 것을 무조건 지우는 게 아니라, 네이버 표준을 정본 뼈대로 삼고 기존을 그 위로 흡수·정리한다.
3. **상품은 네이버 표준 노드로 공격적 재매핑**(사용자 선택 B). 완전일치 + **퍼지 유일후보까지 자동 편입**(실측 약 42%). 단 퍼지 자동은 오매칭이 섞이므로 **전 이동을 로그에 남기고 관리자가 되돌릴 수 있게** 한다. 모호(복수후보)·무매칭은 수동/보존.
4. **사용자 고유·사용 중 카테고리는 보존.** 네이버에 대응이 없는(사용자가 만든) 카테고리, 그리고 어디선가 참조되는 카테고리는 남긴다. **어디에서도 안 쓰이는 것(현재 16개)만 제거.**
5. **지속 업데이트.** 네이버 주기 수신 시 신규 표준 노드를 우리 글로벌 카테고리에 반영(additive). 이때도 "미사용만 교체" 원칙 유지.
6. **사용자 생성분 검증은 후속**("이후에 검증"). 이번엔 `origin='user'`로 **태깅만** 하고 재조정 로직은 만들지 않는다.

## 2. 현황 (DB 실측)

| 축 | 네이버(`naver_category`, active) | 우리 글로벌(`categories` NORMAL, mall_id=0) |
|---|---:|---:|
| L1 / depth1 | 11 | 31 |
| L2 / depth2 | 242 | 120 |
| L3 / depth3 | 2,094 | 238 |
| L4 | 3,468 | — (최대 3뎁스) |
| **합계** | 5,815 (리프 4,999) | **389** |

- **정렬 안 됨**: 대분류 정확 일치 4/31(가구·인테리어/스포츠·레저/식품/패션잡화), 전체 이름 일치 151/389(39%). 뼈대 상이(네이버 `패션의류>여성의류>티셔츠` vs 우리 `여성패션>티셔츠`; 네이버 `식품>건강식품>영양제` vs 우리 `영양제`(대분류 직속)).
- **상품 사용 실태**: NORMAL 389개 중 **373개 사용 중**(미사용 16개). 상품이 리프뿐 아니라 **대분류·중분류에 직접** 붙음(예: `영양제` 92건, `건강식품(기타)` 94건).
- **네이버 역참조 미사용**: 상품 10,010건 중 `naver_category_id` 채워진 건 **2건** → 재매핑은 상품 자체 네이버ref가 아니라 **현재카테고리→네이버 이름매칭**으로 해야 함.
- **시드 데이터 품질**: `parent_naver_id`는 전량 비어 있으나(API 미제공), `whole_category_name`의 `>` 세그먼트 수 = `category_level`과 **정확히 일치** → **경로 파싱으로 트리 재구성 가능**.
- **참조 없음 상태**: `categories.naver_category_id`·출처(origin) 컬럼 **없음** → 신설 필수(§3).

### 2-1. 재매핑 자동화 가능성 (실측 — 설계의 핵심 근거)

우리 389개 이름을 네이버 노드(active, level≤3)에 매칭하고 **상품 수로 가중**한 결과:

| 분류 | 조건 | 카테고리 | 상품 | 비중 |
|---|---|---:|---:|---:|
| **자동이동** | 정규화 이름이 네이버에서 **유일** 매칭 | 87 | 2,655 | **27%** |
| **수동 검토** | 이름이 복수 네이버 노드에 매칭(모호) | 42 | 1,429 | 14% |
| **보존(무매칭)** | 네이버에 대응 이름 없음 | 260 | 5,925 | **59%** |

**퍼지(부분포함) 매칭을 무대응 260개에 추가 적용**(사용자 선택 = 공격적 편입):

| 구분 | 카테고리 | 상품 | 예시 |
|---|---:|---:|---|
| 퍼지 유일후보(자동편입) | 61 | 1,505 | `다이어트`→`식품>다이어트식품`, `단백질`→`식품>다이어트식품>단백질보충제` (양호) |
| 퍼지 복수후보(수동) | 91 | 1,920 | `지갑`→`남성지갑/여성지갑/…` |
| 그래도 무매칭(보존) | 108 | 2,500 | `건강식품(기타)` 등 |

> ⚠ **결론(사용자 선택 B — 공격적 편입 반영):**
> - 완전일치 자동(27%, 2,655건) **+ 퍼지 유일후보(1,505건) = 약 42%(4,160건)까지 자동 편입 가능**.
> - 나머지 58%는 여전히 수동 검토(모호) 또는 보존(무매칭).
> - **퍼지 유일후보도 오매칭이 섞인다**(예: `콜라겐/이너뷰티`→`여가>자기계발>뷰티`는 명백한 오매칭). 부분포함은 짧은 이름에서 노이즈가 큼.
> - 원인: ① 우리 뼈대가 네이버와 어긋남(`여성패션>티셔츠` vs `패션의류>여성의류>티셔츠`) ② 우리 리프(L3)가 네이버에선 비-리프 중간노드 많음(네이버 리프 4,999 중 L4가 3,468) ③ 마케팅 묶음 이름은 네이버에 없음.
> **따라서 (B)를 택해도 본체는 "표준 뼈대 선-시드 + 퍼지 자동편입 + 관리자 사후 검토·롤백"이다.** 퍼지 자동은 반드시 `category_remap_log`에 남기고 관리자가 되돌릴 수 있어야 한다(§5 Phase 2).

## 3. 데이터 모델 변경

### 3-1. `categories` 컬럼 추가 (멱등 가드)

```sql
ALTER TABLE categories
  ADD COLUMN naver_category_id VARCHAR(64) NULL COMMENT '이 노드가 대응하는 네이버 카테고리 ID(origin=naver 일 때)' AFTER shopify_collection_id,
  ADD COLUMN origin ENUM('naver','user') NOT NULL DEFAULT 'user' COMMENT '출처 — naver:표준시드 / user:사용자·상품등록 생성',
  ADD KEY idx_categories_naver (naver_category_id),
  ADD KEY idx_categories_origin (type, origin);
```

- `origin`은 **"사용자 생성분 보존"의 집행 근거**. 없으면 규칙을 강제할 수 없다(현재가 그 상태).
- 기존 389개는 마이그레이션에서 origin을 정한다(§5 Phase 1): 네이버 매칭 성공분은 `naver`로 승격, 나머지는 `user` 유지.
- **`naver_category_id`는 이미 `products`에도 있음**(등록 근거 참조). 컬럼명이 겹치지만 축이 다르다: `categories.naver_category_id`=이 우리 노드↔네이버 노드 매핑, `products.naver_category_id`=이 상품이 근거로 삼은 네이버 리프. 혼동 주의.

### 3-2. 매핑 원장 (선택 — 감사·역추적용)

재매핑 이력을 남겨 롤백·검증을 돕는다(파괴적 이동을 하므로 권장).

```sql
CREATE TABLE IF NOT EXISTS category_remap_log (
  id            BIGINT NOT NULL AUTO_INCREMENT,
  phase         VARCHAR(32) NOT NULL,           -- 'SEED' | 'REMAP' | 'PRUNE' | 'SYNC'
  from_category_id INT NULL,                    -- 이동 전 우리 카테고리
  to_category_id   INT NULL,                    -- 이동 후 우리 카테고리(네이버 리프)
  naver_category_id VARCHAR(64) NULL,
  product_count INT NULL,                       -- 이동한 상품 수
  match_kind    ENUM('EXACT','PATH','FUZZY','MANUAL','NONE') NULL,
  score         DECIMAL(4,3) NULL,
  note          VARCHAR(500) NULL,
  created_at    TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  KEY idx_crl_phase (phase, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

## 4. "사용 중" 판별 — 삭제 안전 체크리스트 (필수)

미사용 제거(Phase 3)는 **단일 컬럼이 아니라 아래 전부**를 확인한 뒤에만 지운다. 하나라도 참조되면 보존.

| # | 참조처 | 위치 | 방식 |
|---|---|---|---|
| 1 | `products.category_id` | FK `products_ibfk_1` (ON DELETE SET NULL) | 컬럼 |
| 2 | `banners.category_id` | FK `fk_banners_category` | 컬럼 |
| 3 | `category_option.category_id` | FK `fk_catopt_cat` | 컬럼 |
| 4 | `custom_menu` `link_type='CATEGORY'` + `link_target` | 값 참조(FK 아님) | `navigationService`가 해석 |
| 5 | `product_group.filter_condition_json.category_id` | JSON 값 | `productGroupService` |
| 6 | `coupons.scope_json`의 `include/exclude.categoryIds` | JSON 배열 | `discountCalculator` |
| 7 | `mall_category_visibility.category_id` | (mall_id,category_id) 행 | 몰별 숨김 override |
| 8 | 파생 유효집합 조상 | `categoryScope.validCategoryIdSet` | 자손에 상품 있으면 조상도 "사용 중" |

> **원칙**: `origin='user'`이거나 위 1~8 중 하나라도 걸리면 **보존**. 전부 미참조 + `origin`이 시드로 대체 가능일 때만 제거. 현재 미사용 후보는 16개지만, 재매핑(Phase 2)으로 상품이 빠져나가 새로 비는 노드가 생기므로 판정은 **재매핑 이후에 1회** 수행한다.

## 5. 마이그레이션 단계 (Phase)

각 Phase는 독립 실행·재실행(멱등) 가능해야 하며, 파괴적 단계(2·3) 전에는 스냅샷을 뜬다.

### Phase 0 — 준비 (비파괴)
- `categories`에 `naver_category_id`·`origin` 컬럼 + `category_remap_log` 테이블 추가(§3).
- **스냅샷**: `categories`, `products(id, category_id)` 를 백업 테이블(`_bak_YYYYMMDD`)로 덤프. 롤백 근거.

### Phase 1 — 네이버 L1~L3 시드 (비파괴, additive)
`naver_category` (is_active=1, level≤3) 2,347개를 우리 글로벌 카테고리로 세운다.

1. `whole_category_name`을 `>`로 파싱, **경로가 짧은 것부터**(L1→L2→L3) 정렬.
2. 각 세그먼트를 `resolveOrCreatePath` 패턴으로 생성/재사용하되, **네이버 노드는 정확 매칭(dedup)**으로 기존과 합친다:
   - 같은 부모 아래 **정규화 이름 완전일치** 노드가 있으면 재사용(신규 생성 안 함) → 기존 `식품`·`가구/인테리어` 등 4개 대분류가 네이버 노드로 흡수됨.
   - 재사용/생성한 노드에 `origin='naver'`, `naver_category_id`= 해당 네이버 ID 기록.
3. dedup은 taxonomyResolver의 `normalizeName`/`findChildByName`을 재사용(퍼지 아님 — 시드는 **정확 매칭만**, 오합치 방지).
4. 결과: 글로벌 카테고리에 네이버 표준 트리가 `origin='naver'`로 존재. 기존 사용자 노드는 아직 그대로(`origin='user'`).

> depthGuard 경유 필수(최대 3뎁스). 네이버는 L3까지만 넣으므로 위반 없음.
> **display_order**: 네이버 응답의 배열 순서를 시드 시 `display_order`로 보존한다(형제 정렬). 미보존 시 순서가 임의가 됨.

### Phase 2 — 상품 재매핑 (파괴적: products.category_id 이동) — 공격적 편입(B)
기존 카테고리를 네이버 노드에 매칭해 상품을 이동한다. **완전일치 + 퍼지 유일후보까지 자동 편입**하되(실측 약 42% — §2-1), 퍼지 자동은 오매칭이 섞이므로 **전량 로그 + 롤백**을 전제로 한다.

| 신뢰도 | 조건 | 처리 |
|---|---|---|
| **PATH** | 기존 카테고리의 **전체 경로**(`대>중>소`)가 네이버 `whole_category_name`과 정규화 완전일치 | **자동 이동**(안전) + 로그 |
| **NAME-UNIQUE** | 이름이 네이버 level≤3에서 **정규화 완전일치 & 유일** | **자동 이동**(안전) + 로그 |
| **FUZZY-UNIQUE** | 완전일치 없음. 부분포함/유사도 ≥0.85 후보가 **딱 1개** | **자동 편입** + 로그(`match_kind='FUZZY'`, score). **오매칭 가능 → 관리자 롤백 대상**. `origin='user'` 유지(네이버 승격 아님, 잠정 편입) |
| **NAME/FUZZY-MULTI** | 동명/유사 후보가 **2개 이상**(예: `영양제`=식품&반려동물, `지갑`=남성/여성/…, `기타`·`세트` 전역 반복) | **수동 큐**. 자동 금지 — 임의 선택은 silent 오배치 |
| **NONE** | 이름·부분포함 후보 전무(`건강식품(기타)` 등) | **보존**. `origin='user'`, 상품 그대로 |

> **롤백이 (B)의 안전장치**: 퍼지 자동은 `콜라겐/이너뷰티→여가>자기계발>뷰티` 같은 오매칭을 만든다. 그래서 ① 퍼지 이동은 전량 `category_remap_log`(from/to/score) 기록 ② 관리자 화면에서 **일괄/개별 되돌리기**(원 카테고리로 복원) 제공 ③ 퍼지 편입 노드는 `origin='naver'`로 승격하지 않고 `user`로 두어, 사후 검토에서 확정/취소한다.
> **bare-name 함정**: 네이버 이름은 트리 전역 반복(`기타` 19곳·`eBook` 22곳). 완전일치·퍼지 모두 **후보가 유일할 때만** 자동, 복수면 사람이 고른다.

**non-leaf 상품 착지 정책** (상품이 대분류·중분류 직속인 hazard):
- 네이버는 리프 선택이 원칙이나, 우리는 3단계까지만 쓰므로 매칭 대상은 **네이버 level≤3 노드**(리프 여부 무관)다. 우리 `영양제`↔네이버 `식품>건강식품>영양제`(L3, 비-리프)도 정당한 매핑.
- 대응 노드가 없거나 모호하면 → **자동 이동 안 함**, 수동 큐. 상품을 임의 노드에 떨구지 않는다.
- 어떤 경우에도 `category_id=NULL`로 만들지 않는다(목록·검색에서 사라짐 방지). 애매하면 원위치 유지.

### Phase 3 — 미사용 정리 (파괴적: 빈 노드 제거)
Phase 2로 상품이 빠져 **완전히 비고(§4 체크리스트 1~8 전부 미참조)**, `origin='naver'`로 대체 가능한 노드만 제거. `origin='user'`는 비어도 이번엔 **제거하지 않고 보존**(후속 검증 대상).
- 제거는 depthGuard 규칙 준수: **자식 있으면 삭제 차단**. 리프부터 상향 정리.
- `category_remap_log`에 `phase='PRUNE'` 기록.

### Phase 4 — 지속 동기화 컴포넌트 (신규, additive·멱등)
`naverTaxonomySync`(24h 크론) 이후 훅으로 **네이버→우리 카테고리 반영**을 붙인다. 별도·멱등·비파괴 컴포넌트.
- **신규 네이버 노드**(우리에 `naver_category_id` 없음) → Phase 1 규칙으로 우리 글로벌에 추가(`origin='naver'`).
- **네이버에서 사라진 노드**(`naver_category.is_active=0`) → 대응 우리 노드를 **soft 비활성(`is_active=0`)만** 한다(무인 크론이라 hard delete 안 함 — 오판정해도 복구는 `is_active=1` 한 번). 크론의 참조 판별은 FK 3종만, **hard 제거는 관리자 prune 스크립트**(§4 전체 체크리스트)가 담당. `origin='user'`는 불변.
  > ⚠ 구현 주의: `categories.naver_category_id`(general_ci) ↔ `naver_category`(unicode_ci) **collation 불일치**로 JOIN 시 에러 → JOIN 키에 `COLLATE utf8mb4_unicode_ci` 명시 필수(무인 경로라 누락 시 매 sync 조용히 실패).
- 사용자 생성(`origin='user'`)은 동기화가 **건드리지 않음**.
- 매 회차 `category_remap_log`에 `phase='SYNC'` 요약 기록.

## 6. taxonomyResolver 연동 변경

뼈대가 선-존재하므로, 상품 등록 경로(`resolveOrCreatePath`/`resolveOrCreateCategory`)의 동작을 "생성 우선"에서 **"표준 노드 매칭 우선"**으로 조정한다.

- 등록 위젯에서 네이버 리프를 선택하면(`products.form.ejs`), `products.naver_category_id`가 오므로 → **먼저 `categories`에서 `naver_category_id`로 정확 매핑**을 찾아 붙인다(경로 파싱·퍼지 이전에).
- 그 매핑이 없을 때만 기존 경로(퍼지 매칭 → 없으면 생성)로 폴백. 새로 생성되는 노드는 `origin='user'`.
- 결과: 표준 네이버 리프로 등록하는 상품은 **항상 같은 우리 노드**에 모인다(남발 종식).

## 7. 범위 밖 (이번 설계에서 다루지 않음 — 기존 부채)

- **레거시 몰 스코프 읽기 경로**(`navigationService`·홈 섹션 리졸버 등이 `mall_id = ?`로 남아 NORMAL/BRAND에 stale). 글로벌화 설계 Phase D 소관. 시드를 막지 않으므로 분리.
- **`mall_category_visibility` 문서(`is_visible`) vs 구현(`hidden`) drift**, `tables.sql` 미반영. 별도 정리.
- **사용자 생성분 재조정 로직**("이후에 검증"). 이번엔 `origin='user'` 태깅까지만.
- **BRAND 재구성**. 이 설계는 NORMAL만. 브랜드는 `naver_brand`(현재 0건) 확보 후 별도.

## 8. 리스크 & 검증

| 리스크 | 대응 |
|---|---|
| 상품 오이동(FUZZY 오매칭) | (B)는 퍼지 유일후보를 자동 편입하므로 오매칭 발생 상수화. **완화**: 전량 `category_remap_log`(from/to/score) + 관리자 일괄/개별 롤백 + 퍼지 편입은 `origin='user'`로 잠정 처리(사후 확정). 복수후보는 자동 금지 |
| non-leaf 상품이 갈 곳 없음 | 정확 대응 리프 없으면 원위치 유지(§5 Phase 2) |
| category_id NULL 유실 | 어떤 단계도 NULL 만들지 않음. 미분류 폴백 유지 |
| 시드 중복 노드 | dedup은 정규화 정확일치. 시드 후 `SELECT parent_id,name,COUNT(*)>1` 로 중복 0 검증 |
| 공용 개발 DB 오염 | Phase 0 스냅샷 + `_bak` 테이블. 파괴 단계는 트랜잭션 |

**검증 쿼리(각 Phase 후)**
- 시드: `SELECT depth,COUNT(*) FROM categories WHERE origin='naver' GROUP BY depth;` → 11/242/2094 근사(dedup 흡수분만큼 차이).
- 재매핑: `SELECT COUNT(*) FROM products WHERE category_id IS NULL;` → 0 유지. `category_remap_log`에서 이동 건수 합 = 확인.
- 정리: 제거 노드가 전부 미참조였는지 `category_remap_log(phase='PRUNE')`로 감사.
