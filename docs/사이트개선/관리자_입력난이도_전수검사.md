## A. JSON·CSS 코드 직접 입력 — 최우선

### A-1. 소싱 > 채널 연결 : 부가 설정 JSON ★
`views/admin/sourcing/connections.ejs:176-179`

```html
<label>부가 설정(JSON, 선택)</label>
<input type="text" name="extra_json" placeholder='{"channelId":"..."}' class="... font-mono">
```

- CLAUDE.md §33 "관리자 화면에 JSON 입력란을 새로 만들지 않는다" 정면 위반.
- 중괄호·따옴표 하나 틀리면 채널 연동이 조용히 실패한다.
- **권장**: 채널 메타(`chans`)에 채널별 부가 필드 정의를 두고 개별 input 을 렌더 → 서버가 JSON 조립. 네이버 고시(§6.4)에서 이미 검증된 패턴 그대로.

> ✅ **조치 — 입력칸 자체를 제거했다.**
> 수정에 앞서 소비처를 추적했더니 **이 값을 읽어 쓰는 코드가 하나도 없었다.** `credential.js` 가 저장·복사만 하고
> 어떤 어댑터도 참조하지 않으며, `mall_channel_credential` 의 `extra_json` 은 **전 행 NULL** 이었다.
> 그래서 필드를 만들어 주는 대신 없앴다 — 실제로 필요한 부가값이 생기면 그때 **이름 있는 필드 하나**로 받고 서버가 조립한다.
> `connections.ejs` · `sourcingController.postConnectionSave` · `adapters.js`(`extraHint`) 에서 제거.

### A-2. 쇼핑몰 관리 > 디자인 스타일 : CSS 값 7종 ★
`views/admin/theme-settings/edit.ejs:81` + `controllers/admin/themeSettingsController.js:27-33`

| 필드 | 요구 입력 |
|---|---|
| `fontFamily` | `'Pretendard', sans-serif` — **CSS 폰트 스택 문법** |
| `buttonRadius` / `cardRadius` / `inputRadius` | `0.5rem`, `8px`, `0` |
| `pillRadius` | `9999px` |
| `sectionSpacing` | `3rem` |
| `containerWidth` | `72rem` |

- 전부 자유 텍스트(`font-mono`). 값이 검증에 걸리면 화면에 "⚠ 기본값 대체"만 뜬다 — 사용자는 왜 안 먹는지 알 수 없다.
- `rem`·`px`·폰트 스택은 일반 사용자 어휘가 아니다.
- **권장**: 폰트 = 설치 폰트 select / 모서리·간격·너비 = 3~5단계 프리셋 select(각지게·보통·둥글게) 또는 px 슬라이더. rem 문자열 조립은 서버가.

> ✅ **조치 — 7종 전부 select 로 교체.**
> 화면에는 "각지게 / 보통 / 둥글게", "좁게 / 보통 / 넓게" 같은 **말**만 보이고, `0.5rem` 같은 CSS 문자열은
> 컨트롤러의 옵션 정의(`value`)에만 있다. 값 형식이 예전과 같아 `themeService.TOKENS[].test` 검증·기존 저장값과 100% 호환된다.
> - **폰트는 레이아웃이 실제로 불러오는 것만** 노출한다(Pretendard · 나눔명조 · Playfair Display · 기기 기본).
>   임의 폰트를 고르게 하면 "저장은 됐는데 안 바뀌는" 먹통 설정이 된다.
> - 저장된 값이 선택지에 없으면(예전 자유 입력분) **"현재 값 유지"** 옵션으로 맨 위에 얹어 저장 한 번에 조용히 바뀌지 않게 했다.
> - 검증: 실제 DB 값 `sectionSpacing: 4.5rem`(목록에 없는 값)이 "현재 값 유지"로 뜨고, 저장 왕복 후 `config_json` 이 **완전히 동일**(TOKENS 밖 키 `skin`·`fontDisplay` 보존).

## B. 생 HTML 직접 작성 — 제품 내 불일치

TinyMCE(WYSIWYG)는 이미 **9개 화면**에 적용돼 있다 — 공지·FAQ·쿠폰·약관(3종)·상품·이벤트·사이트설정.
그런데 아래 화면들은 **에디터 없이 생 HTML textarea**(전부 `font-mono`)다. 같은 제품 안에서 기준이 갈린다.

| 화면 | 위치 | 필드 |
|---|---|---|
| 기획전 편집 | `exhibitions/edit.ejs:103` | 상세 설명 "(HTML 허용)" |
| 기획전 편집 | `exhibitions/edit.ejs:262` | 안내·유의사항 "(HTML 허용)" |
| 기획전 편집 | `exhibitions/edit.ejs:346`, `545` | 섹션 `HTML 내용` |
| 공동구매 편집 | `group-buys/edit.ejs:74` | 상세 본문 "HTML 을 쓸 수 있습니다" |
| 라이브 편집 | `lives/edit.ejs:76` | 상세 본문 "HTML 을 쓸 수 있습니다" |
| 아울렛 설정 | `outlet/list.ejs:292` | 아울렛 공통 고지 `notice_html` |
| 페이지 빌더 | `services/display/sectionRegistry.js:209-216` | `custom_html` 섹션 |

- **권장**: 상품 상세와 동일하게 TinyMCE 적용. `custom_html` 섹션은 의도된 최후수단(escape hatch)이므로 **유지하되 "개발자용" 라벨**을 붙여 일반 사용자 동선에서 분리.
- 새니타이즈는 이미 저장·렌더 이중으로 되어 있어 보안 문제는 아니다. **순수 입력 난이도** 문제.

> ✅ **조치 — `views/partials/admin/tinymce.ejs` 신설 후 6곳 적용 + 페이지빌더 1곳 표기 변경.**
> - 대상 textarea 에 `data-rich` 를 달고 페이지 끝에서 파티셜을 include 한다.
> - **동적으로 추가되는 기획전 섹션**도 `window.ydRichEditor.attach(el)` 로 붙는다(페이지 로드 후 생긴 요소라 직접 붙여야 한다).
> - **제출 직전 `tinymce.triggerSave()`** 를 캡처 단계에서 건다. JS 로 `form.submit()` 하면 자동 저장이 걸리지 않아 **빈 값이 저장되는** 사고가 난다.
> - `TINYMCE_KEY` 가 없으면 에디터 없이 평범한 textarea 로 남는다(`font-mono` 제거 — 코드처럼 보이지 않게).
> - `custom_html` 은 존치하되 이름을 **"커스텀 HTML (개발자용)"** 으로 바꿔 일반 사용자 동선에서 분리했다.
> - 검증(브라우저): 기획전 편집에서 에디터 3개 초기화 → 섹션 추가 시 4번째 자동 부착 → `triggerSave` 후 textarea 에 HTML 반영 확인.
>
> ⚠ 같은 화면의 "유의사항"(`group-buys:81` · `lives:83` · `events/form:59`)은 **평문 그대로 두었다.** 원래도 HTML 안내가 없던 평문 필드다(B-부록 ③).

#### B 전용 추가 검증 — 새니타이저 × 툴바 대조

B 만 저장되는 **내용의 모양**이 달라진다(평문/생 HTML → 표·이미지·정렬이 섞인 리치 HTML).
그래서 "에디터 → textarea" 만으로는 부족하고 **textarea → 새니타이즈 → DB** 까지 확인했다.
툴바가 약속한 서식을 서버가 조용히 버리면, 사용자는 표를 넣고 저장했는데 사라지는 — 예전 생 HTML 박스보다 나쁜 상황이 된다.

| 툴바가 만드는 것 | 새니타이저 통과 |
|---|---|
| `<strong>` `<em>` `<p>` | ✓ |
| `<ul>/<ol>/<li>` | ✓ |
| `<a href>` | ✓ (외부 링크는 `target=_blank rel=noopener` 자동 부여) |
| `<img src alt width height>` | ✓ |
| `<table><tbody><tr><td>` | ✓ |
| `text-align` 정렬 | ✓ |
| `<script>` | ✓ 제거됨 |
| `style="border: expression(...)"` | ✓ 차단됨 |

**한 가지 고쳤다** — 표 테두리(`border` · `border-collapse`)가 `allowedStyles` 에 없어 **표는 저장되는데 선만 사라졌다.**
이제 4개 화면에 [표] 버튼을 노출했으므로 표현용 border 계열을 허용 목록에 추가했다(`htmlSanitizer.js`).

**아웃렛 `notice_html` 은 저장 시 새니타이즈가 아예 없었다.** 원래 평문 박스라 필요가 없었지만, 이제 HTML 을 뱉으므로
다른 3개 화면과 같은 규칙으로 `sanitize()` 를 걸었다(`outletController.postSetting`).
스토어프론트는 `<%- noticeHtml %>` 로 raw 렌더 중이라 **에디터 출력이 정상 표시된다**(렌더 시에도 한 번 더 새니타이즈 — 이중 방어).
다만 마크업이 `<i>아이콘</i>텍스트` 인라인 구조라 `<p>` 가 들어오면 아이콘 아래로 밀렸다 → flex 로 감싸 정렬을 유지했다(`views/user/outlet/list.ejs`).

검증: 실제 POST 로 굵게+표+`<script>` 를 저장 → DB 에 `<strong>`·표·테두리 보존, `<script>` 제거 확인.
**검증용으로 만들어진 `outlet_setting` 행은 삭제해 원복**했고, 삭제 후에도 화면이 200 이라 위 폴백 버그 수정도 함께 재확인됐다.

### B-부록. textarea 38개 전수 분류 (신호 grep 이 아니라 **열거**)

§6.4 가 "JSON textarea 는 한 곳뿐"이라 단정했다가 `<input>` 을 놓친 전례가 있어, textarea 축만은 38개 전부를 열거해 분류했다. **13 + 7 + 18 = 38, 잔여 없음.**

**① WYSIWYG(TinyMCE) 적용 — 13개 ⭕**
`notices/form:30` · `faqs/form:51` · `coupons/form:114,118` · `policies/form:14,20` · `policies/create:36` · `policies/edit:33` · `products/form:535,542` · `events/form:54` · `settings/form:137,148`
(셀렉터 확인: `textarea.tinymce-editor` / `#editor` / `#content` / `#detailEditor, #noticeEditor` / `textarea#short_description` / `textarea#description` / `#slogan` / `#company_intro`)

**② 생 HTML 직접 작성 — 7개 ❌** (위 표와 동일)

**③ 평문 자유 텍스트 — 18개 ⭕** (HTML 안내·`font-mono` 없음. 사용자가 코드를 쓰지 않는다)

> ✅ **조치**
>
> **원산지(1건) — 별도 작업에서 이미 해결됨.** 이 감사 직후 진행된 네이버 연동 작업이 `naver_origin_area`(535건 수집)와
> 검색 UI 를 넣었다. 현재 `publish.ejs` 는 이름으로 검색해 고르고 코드는 hidden 으로 넘어간다. **추가 조치 불필요.**
>
> **주소록(2건) — `naverAddressBook.js` + [주소록 불러오기] 버튼 신설.** (커밋 `3dd2b89`)
> - 화면에서 버튼을 누르면 `GET /admin/sourcing/publish/addressbooks` 로 네이버 주소록을 받아 두 select 를 채운다.
>   고르면 hidden(`release_address_no` · `refund_address_no`)에 번호가 들어간다.
> - **저장하지 않는다.** 주소록은 몰마다 다르고 판매자센터에서 언제든 바뀌는 운영 데이터라 시드하면 납품된 몰에서 틀린 값이 된다(§31).
> - 저장된 번호가 주소록에 없으면(판매자센터에서 삭제) **"현재 설정 유지"** 로 남겨 조용히 버리지 않는다.
> - 응답 필드명 흔들림은 `normalize()` **한 함수에만** 가뒀다(고시 스키마와 같은 방침).
> - 검증: 로컬에서 **실제 네이버 API 호출 성공** — 주소록 2건(`상품출고지`/RELEASE, `반품교환지`/REFUND_OR_EXCHANGE) 수신,
>   브라우저에서 버튼 → select 채움 → hidden 반영까지 확인. (실응답에서 확인된 `REFUND_OR_EXCHANGE` 를 라벨 표에 반영)
