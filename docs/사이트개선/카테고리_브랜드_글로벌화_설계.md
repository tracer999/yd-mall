# 카테고리·브랜드 글로벌화 설계

> 목적: 지금은 `categories.mall_id` 로 **몰별**로 관리되는 카테고리·브랜드를, **전체 몰 공통(글로벌) 한 벌**로 바꾼다. 몰별 중복 관리를 없앤다.
>
> 짝 문서: [`네이버_마스터데이터_수신_설계.md`](./네이버_마스터데이터_수신_설계.md), [`쇼핑몰_상품_옵션_세트_묶음_관리구조_정리.md`](./쇼핑몰_상품_옵션_세트_묶음_관리구조_정리.md) §32.

---

## 1. 확정 규칙 (사용자)

1. **카테고리·브랜드 = 글로벌 한 벌.** 몰별 중복 관리 폐지. 관리 메뉴는 전 몰 공통 카탈로그 하나.
2. **각 몰 스토어프론트의 "유효 카테고리" = 그 몰에 상품이 있는 카테고리**(`products.mall_id` 기준, 조상 포함). 무관 카테고리는 그 몰에 안 뜬다.
3. **유효 카테고리 안에서 몰별 표시 on/off 선택 가능.** (예: 건강식품관에 유효하지만 특정 카테고리는 숨김) → **몰별 표시 override**.
4. 카테고리는 우리 몰 기준 **최대 3단계**(§32). 네이버 4단계는 상품 전송 시 선택.

| mall | NORMAL | BRAND | THEME | OUTLET |

- `categories.mall_id`/타입 스코핑 코드 **36개 파일**(productController·categoryController·taxonomyResolver·navigationService·brandService·brandController·outletController·categorySync 등).

## 3. 목표 데이터 모델

### 3.1 categories = 글로벌
- `categories.mall_id` 를 **글로벌 sentinel(예: 0)** 로 통일(컬럼은 유지, 스코핑만 폐지). 계층(`parent_id`,`depth`)·타입(`NORMAL/BRAND/THEME/OUTLET`) 그대로.
- 관리자 카테고리/브랜드 화면은 `adminMallId` 스코핑 제거 → 전 몰 공통 카탈로그.

### 3.2 유효 카테고리(몰) — 파생
```text
validCategories(mall) = { c : ∃ product(mall_id=mall, category_id=c) } ∪ (그 조상)
validBrands(mall)     = { b : ∃ product(mall_id=mall, brand_category_id=b) }
```
관리자 §10 필터(상품 있는 것만)와 동일 로직을 **몰 기준**으로.

### 3.3 몰별 표시 override — 신설
```sql
CREATE TABLE mall_category_visibility (
  mall_id     BIGINT NOT NULL,
  category_id INT    NOT NULL,
  is_visible  TINYINT(1) NOT NULL DEFAULT 1,   -- 유효 카테고리 중 이 몰에서 숨길지
  PRIMARY KEY (mall_id, category_id)
);
```
- 기본: 유효 카테고리는 **표시(visible)**. 이 표에 `is_visible=0` 행이 있으면 그 몰에서 숨김.
- 스토어프론트 노출 = `validCategories(mall)` − `{override is_visible=0}`.

## 4. 마이그레이션 (비가역 — 신중)

> ⚠️ **공용 DB. Phase B(products 재지정)부터는 되돌리기 어려움.** 각 단계 전 백업/검증.

- **Phase A — 글로벌 카테고리 셋 구성(매핑만, 파괴 없음)**
  `(type, 계층 경로)` 기준으로 중복 병합 규칙 결정 → **old category_id → global category_id 매핑표** 생성. 병합 정책:
  - NORMAL: 같은 **경로**(대>중>소)면 동일 카테고리로 병합. 다르면 별개.
  - BRAND: 같은 **이름**이면 병합(브랜드는 평면).
  - 병합 시 대표 1개를 글로벌로 승격, 나머지는 매핑 대상.
- **Phase B — products 재지정**
  `products.category_id/brand_category_id/theme_category_id` 를 매핑표로 일괄 UPDATE. (크로스몰 참조 0 이라 안전한 재지정)
- **Phase C — 코드 스코핑 제거(36 파일)**
  카테고리/브랜드 조회에서 `mall_id` 필터 제거. 관리자 카테고리/브랜드 화면 글로벌화. `taxonomyResolver`(생성)도 글로벌로.
- **Phase D — 스토어프론트 내비 재구성**
  `navigationService` 를 **유효 카테고리(몰) + 표시 override** 기준으로. 관리자에 몰별 표시 on/off UI.
- **Phase E — 잔여 정리**
  매핑으로 흡수된 중복 카테고리(참조 0 확인 후) 삭제. sentinel(mall_id=0)로 정리.

## 5. 리스크·주의

- **규모**: 36 파일 + 1,700+ 카테고리 병합 + 내비 재설계. 다단계 프로젝트(SKU 작업 규모).
- **비가역**: Phase B 이후 products 참조가 바뀜. 공용 개발 DB라 서버에도 즉시 반영.
- **THEME/OUTLET**: mall 1 THEME 2, mall 2 OUTLET 4 — 글로벌화 대상에 포함할지 결정 필요(우선 NORMAL·BRAND만, THEME/OUTLET은 후속).
- **기존 §10 필터**: 관리자 "상품 있는 것만"은 지금 `adminMallId` 기준 → 글로벌화 후 "전 몰 통틀어 상품 있는 것"으로 의미 변경. 관리자 카탈로그는 전 몰, 스토어프론트는 몰별 유효.
- **멀티몰 원칙(CLAUDE.md)** 과의 관계: mall_id 스코핑은 유지되는 축(products·menu·orders)과 폐지되는 축(categories/brands)이 나뉜다 — 문서로 명확히.
