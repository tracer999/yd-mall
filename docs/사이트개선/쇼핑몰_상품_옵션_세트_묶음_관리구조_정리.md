# 쇼핑몰 상품·옵션·세트·묶음 관리 구조 정리

## 1. 문서 목적

이 문서는 쇼핑몰 빌더의 상품 등록 기능을 설계할 때 필요한 다음 내용을 정리한 것이다.

- 단일상품과 옵션상품의 구분
- 상품 옵션명과 옵션값 관리 방식
- 카테고리별 옵션 추천 구조
- 상품 속성과 판매 옵션의 구분
- 수량별 가격과 묶음상품의 처리
- 세트상품과 기획상품의 관리
- 기본 상품 풀과 판매 상품 풀의 구분
- Product, Variant, SKU 중심의 권장 데이터 모델

---

# 2. 핵심 결론

쇼핑몰 상품 구조는 다음과 같이 설계하는 것이 적합하다.

```text
상품 마스터(Product)
 ├─ 단일상품
 │    └─ SKU 1개
 │
 ├─ 옵션상품
 │    └─ SKU 여러 개
 │
 ├─ 묶음상품
 │    └─ 기존 SKU × 수량
 │
 └─ 세트·기획상품
      └─ 기존 여러 SKU 조합
```

운영상으로는 다음 두 개의 풀로 구분한다.

```text
기본 상품 풀
= 단일상품 + 옵션상품의 SKU

판매 상품 풀
= 기본 상품 + 묶음상품 + 세트상품 + 기획상품
```

핵심 기준은 다음과 같다.

- 고객에게 보여주는 단위는 `Product`
- 실제 재고와 거래의 기준은 `SKU`
- 단일상품도 내부적으로 SKU 1개를 가진 상품
- 옵션상품은 여러 Variant 또는 SKU를 가진 상품
- 세트·묶음·기획상품은 기존 SKU를 참조하여 구성
- 카테고리별 옵션 정보는 강제 규칙보다 추천 템플릿으로 사용

---

# 3. 단일상품과 옵션상품

## 3.1 단일상품

옵션이 없는 상품이다.

예시:

```text
상품: 생수 500ml
 └─ SKU: 생수 500ml 단품
```

단일상품도 내부적으로는 하나의 SKU를 가진다.

```text
Product ID: PRODUCT-100
SKU ID: SKU-100-01
재고: 100개
```

즉, 시스템적으로는 다음과 같이 이해할 수 있다.

```text
단일상품 = SKU가 1개인 상품
```

---

## 3.2 옵션상품

색상, 사이즈, 용량, 향 등 고객이 선택할 수 있는 옵션을 가진 상품이다.

예시:

```text
상품: 기본 반팔 티셔츠

 ├─ SKU: 블랙 / M
 ├─ SKU: 블랙 / L
 ├─ SKU: 화이트 / M
 └─ SKU: 화이트 / L
```

재고는 상품 전체가 아니라 각 SKU별로 관리한다.

| SKU | 재고 |
|---|---:|
| 블랙/M | 10 |
| 블랙/L | 8 |
| 화이트/M | 15 |
| 화이트/L | 5 |

옵션 조합별로 다음 정보를 관리할 수 있다.

- SKU
- 재고
- 판매가
- 원가
- 바코드
- 공급처 상품코드
- 판매 상태

---

# 4. 옵션명과 옵션값 관리 방식

## 4.1 완전 고정형은 적합하지 않음

상품 카테고리와 상품 특성이 매우 다양하기 때문에 옵션명을 완전히 고정해서는 안 된다.

예를 들어 같은 화장품 카테고리에서도 상품별 옵션이 다르다.

- 립스틱: 색상
- 파운데이션: 색상, 호수
- 샴푸: 용량, 향
- 마스크팩: 종류, 수량
- 기획세트: 구성

따라서 모든 상품에 동일한 옵션 구조를 강제하면 예외 상품을 처리하기 어렵다.

---

## 4.2 완전 자유 입력형도 적합하지 않음

옵션명과 옵션값을 모두 직접 입력하도록 하면 판매자마다 표현이 달라진다.

예시:

```text
색상
컬러
Color
색깔
```

또는:

```text
2개
2개입
2EA
2팩
```

이 경우 검색, 필터, 외부 마켓 연동, 데이터 분석이 어려워진다.

---

## 4.3 권장 방식: 반정형 구조

가장 적절한 방식은 다음과 같다.

```text
표준 옵션명 제공
+ 카테고리별 추천
+ 직접 입력 허용
```

예시:

```text
카테고리: 패션 > 여성의류 > 티셔츠

추천 옵션
☑ 색상
☑ 사이즈
□ 기장
□ 핏
+ 직접 옵션 추가
```

다른 카테고리:

```text
카테고리: 식품 > 음료 > 커피

추천 옵션
☑ 용량
☑ 수량
□ 맛
□ 원두 종류
+ 직접 옵션 추가
```

---

# 5. 옵션명과 옵션값의 차이

## 5.1 옵션명

가능한 한 표준 사전을 제공한다.

예시:

- 색상
- 사이즈
- 용량
- 수량
- 재질
- 향
- 맛
- 길이
- 구성
- 모델
- 타입
- 호수

표준 목록에 없는 경우 직접 입력을 허용한다.

---

## 5.2 옵션값

옵션값은 상품마다 다르기 때문에 직접 입력을 기본으로 하되 추천값을 제공한다.

예시:

```text
옵션명: 색상

추천값:
블랙 / 화이트 / 그레이 / 네이비 / 베이지

직접 입력:
민트블루 / 로즈핑크
```

옵션값 역시 내부적으로 ID 기반으로 관리하는 것이 좋다.

예시:

```text
OPTION: COLOR
VALUE: BLACK
DISPLAY_NAME: 블랙
```

화면 표시명과 내부 코드를 분리하면 다음 작업에 유리하다.

- 다국어
- 옵션명 변경
- 외부 마켓 매핑
- 검색 및 필터
- 통계 및 분석

---

# 6. 상품 속성과 판매 옵션의 구분

상품 속성과 판매 옵션은 반드시 구분해야 한다.

## 6.1 상품 속성

검색, 필터, 상품 설명에 사용되는 정보이다.

예시:

- 제조사: 삼성전자
- 원산지: 한국
- 재질: 면 100%
- 화면 크기: 15.6인치
- 피부 타입: 건성

고객이 주문 과정에서 선택하지 않는다.

---

## 6.2 판매 옵션

고객이 구매 시 선택하며 가격, 재고, SKU가 달라질 수 있다.

예시:

- 색상: 블랙 / 화이트
- 사이즈: M / L
- 수량 구성: 1개입 / 2개입
- 용량: 300ml / 500ml

같은 항목이라도 상품에 따라 속성 또는 옵션이 될 수 있다.

예시:

```text
재질 = 상품 설명용 속성
재질 = 고객이 선택하는 옵션
```

따라서 시스템에서는 속성과 옵션을 별도 엔티티로 관리하는 것이 좋다.

---

# 7. 카테고리별 옵션 매칭 테이블

카테고리별 옵션 매칭 테이블을 만드는 것은 좋은 설계다.

다만 이 테이블은 강제 규칙보다 상품 등록을 돕는 추천 템플릿으로 사용하는 것이 적합하다.

---

## 7.1 카테고리 테이블

```text
category
- category_id
- parent_category_id
- category_name
- depth
- active
```

---

## 7.2 옵션 정의 테이블

```text
option_definition
- option_id
- option_code
- option_name
- option_type
- active
```

예시:

| option_code | option_name | option_type |
|---|---|---|
| COLOR | 색상 | SELECT |
| SIZE | 사이즈 | SELECT |
| CAPACITY | 용량 | SELECT |
| QUANTITY | 수량 | SELECT |
| LENGTH | 길이 | SELECT |
| CUSTOM_TEXT | 문구 입력 | TEXT |

---

## 7.3 카테고리-옵션 매칭 테이블

```text
category_option_mapping
- category_id
- option_id
- required
- recommended
- display_order
- allow_custom_value
- inherit_to_children
```

예시:

| 카테고리 | 옵션 | 필수 | 추천 | 직접입력 |
|---|---|---:|---:|---:|
| 티셔츠 | 색상 | N | Y | Y |
| 티셔츠 | 사이즈 | Y | Y | Y |
| 샴푸 | 용량 | N | Y | Y |
| 샴푸 | 향 | N | Y | Y |
| 노트북 | 색상 | N | Y | N |
| 노트북 | 메모리 | N | Y | Y |

---

# 8. 카테고리 옵션 상속 구조

모든 세부 카테고리에 옵션을 각각 연결하면 관리량이 지나치게 커진다.

따라서 상위 카테고리 옵션을 하위 카테고리에 상속하는 구조가 필요하다.

예시:

```text
패션
 └─ 여성의류
     └─ 티셔츠
```

옵션 매핑:

```text
패션 → 색상
여성의류 → 사이즈
티셔츠 → 기장
```

티셔츠 등록 화면에는 다음이 합쳐져 노출된다.

```text
색상 + 사이즈 + 기장
```

상속 옵션에는 하위 카테고리에서 다음 기능도 필요하다.

- 상속 유지
- 상속 제외
- 노출 순서 변경
- 필수 여부 재정의
- 추천 여부 재정의

---

# 9. 카테고리별 옵션 상태

옵션 매칭 상태는 최소한 다음처럼 구분하는 것이 좋다.

| 상태 | 의미 |
|---|---|
| 필수 | 반드시 입력 |
| 추천 | 기본으로 노출 |
| 선택 | 사용자가 필요할 때 추가 |
| 미사용 | 해당 카테고리에서 숨김 |

일반적인 쇼핑몰 빌더에서는 필수 옵션을 많이 지정하지 않는 것이 좋다.

같은 티셔츠 카테고리라도 단일 색상, 프리사이즈 상품이 존재할 수 있기 때문이다.

---

# 10. 옵션값 추천 테이블

카테고리별로 자주 쓰이는 옵션값도 추천할 수 있다.

```text
category_option_value_mapping
- category_id
- option_id
- option_value_id
- recommended
- display_order
```

예시:

```text
티셔츠 → 색상 → 블랙, 화이트, 네이비
티셔츠 → 사이즈 → S, M, L, XL
신발 → 사이즈 → 230, 240, 250, 260
샴푸 → 용량 → 300ml, 500ml, 1L
```

단, 옵션값은 브랜드와 상품별 편차가 크므로 직접 입력을 항상 허용하는 것이 안전하다.

---

# 11. 카테고리 템플릿과 실제 상품 옵션 분리

카테고리 옵션 매칭 정보는 상품 생성 시 참고하는 템플릿일 뿐이다.

실제 상품 옵션은 상품별 테이블에 별도로 저장해야 한다.

```text
product_option
- product_option_id
- product_id
- option_id
- custom_option_name
- display_order
```

```text
product_option_value
- product_option_value_id
- product_option_id
- value
```

```text
product_variant
- variant_id
- product_id
- sku
- price
- stock
```

처리 흐름:

```text
카테고리 선택
→ 추천 옵션 조회
→ 판매자가 추가·삭제·수정
→ 상품별 옵션으로 확정 저장
```

카테고리 템플릿이 나중에 변경되더라도 기존 상품에 자동 반영되면 안 된다.

기존 상품 변경은 별도의 일괄 수정 기능이나 관리자 승인 절차를 통해 수행해야 한다.

---

# 12. 수량 선택, 수량 할인, 묶음상품의 구분

`2개`, `5개`와 같은 수량은 상황에 따라 처리 방식이 다르다.

---

## 12.1 일반 구매수량

고객이 동일 SKU를 여러 개 구매하는 방식이다.

```text
샴푸 1개 가격: 10,000원
구매수량: 3
총액: 30,000원
```

별도 Variant를 만들 필요가 없다.

---

## 12.2 수량별 할인

동일 SKU를 일정 수량 이상 구매할 때 가격을 할인하는 방식이다.

```text
1개: 10,000원
2개 이상: 개당 9,500원
5개 이상: 개당 8,500원
```

이 경우에도 별도의 SKU를 만들기보다는 가격 정책으로 처리하는 것이 일반적이다.

---

## 12.3 동일상품 묶음

동일 SKU 여러 개를 하나의 판매상품으로 구성한다.

예시:

```text
상품: 샴푸 3개 세트
구성: 샴푸 SKU × 3
판매가: 27,000원
```

재고는 샴푸 SKU 3개가 차감된다.

---

## 12.4 독립 SKU로 관리해야 하는 묶음

다음 조건에 해당하면 묶음을 독립 SKU로 관리할 수 있다.

- 제조사에서 별도 박스로 입고
- 별도 바코드 존재
- 별도 공급처 상품코드 존재
- 별도 원가 존재
- 별도 포장 및 배송 단위
- 물류센터에서 독립 재고로 관리

예시:

```text
생수 500ml 낱병
생수 500ml 20병 공장 포장 박스
```

20병 박스가 실제 독립 재고라면 별도 SKU이다.

판매 시 낱병 20개를 직접 포장하는 구조라면 단품 SKU × 20으로 구성한다.

---

# 13. 세트상품의 정의

세트상품은 기존 상품 또는 SKU 여러 개를 묶어 새로운 하나의 판매상품으로 만든 것이다.

예시:

```text
헤어케어 3종 세트

구성:
- 샴푸 1개
- 린스 1개
- 트리트먼트 1개
```

세트상품도 고객에게는 하나의 Product로 노출된다.

내부적으로는 기존 SKU를 참조한다.

```text
product

id = 10
상품명 = 헤어케어 3종 세트
type = SET
```

구성 테이블:

```text
product_set_component

set_product_id = 10
component_sku_id = SKU-SHAMPOO
quantity = 1

set_product_id = 10
component_sku_id = SKU-RINSE
quantity = 1

set_product_id = 10
component_sku_id = SKU-TREATMENT
quantity = 1
```

---

# 14. 세트상품과 묶음상품의 구분

| 구분 | 설명 | 예시 |
|---|---|---|
| 세트상품 | 서로 다른 SKU 조합 | 샴푸 + 린스 |
| 묶음상품 | 동일 SKU 또는 동일 계열 복수 구성 | 생수 6병 |
| 기획상품 | 캠페인 또는 이벤트 목적의 구성 | 여름 기획세트 |
| 선택형 세트 | 고객이 구성품을 선택 | 선물세트 직접 구성 |

---

# 15. 세트·기획·묶음도 상품 단위인가

세트·묶음·기획상품도 판매 화면에서는 하나의 Product이다.

다만 기본 상품과 달리 자체 재고를 직접 보유할 수도 있고, 구성 SKU의 재고를 참조할 수도 있다.

권장 기준:

```text
기본 상품
- 재고 기준 SKU 보유

세트·묶음·기획상품
- 구성 SKU 참조
- 별도 판매가 및 상세페이지 보유
```

외부 마켓에 등록할 때도 세트상품은 하나의 판매상품으로 전송되지만 내부 재고 차감은 구성 SKU 기준으로 처리한다.

# 16. 기본 상품 풀의 기준

기본 상품 풀은 상품 페이지 단위가 아니라 재고를 독립적으로 차감할 수 있는 SKU 단위로 이해하는 것이 가장 명확하다.

# 20. 재고 계산 예시

## 20.1 동일상품 묶음

```text

```

```text

= floor(20 ÷ 6)
= 3세트
```

## 20.2 서로 다른 상품 세트

```text
세트 판매 가능 수량
```

# 22. 최종 권장 원칙

1. 단일상품과 옵션상품은 모두 Product와 SKU 구조로 통합한다.
2. 단일상품은 SKU 1개, 옵션상품은 SKU 여러 개로 본다.
3. 기본 상품 풀의 실질적 기준은 SKU이다.
4. 세트·기획·묶음상품은 기본 SKU를 참조해 별도 Product로 구성한다.
5. 옵션명은 표준 사전과 직접 입력을 함께 제공한다.
6. 옵션값은 추천값과 직접 입력을 함께 제공한다.
7. 카테고리별 옵션 매칭은 강제 규칙이 아니라 추천 템플릿으로 사용한다.
8. 상위 카테고리 옵션을 하위 카테고리로 상속한다.
9. 카테고리 템플릿 변경은 기존 상품에 자동 반영하지 않는다.
10. 상품 속성과 판매 옵션을 분리한다.
11. 일반 구매수량과 수량 할인은 별도 SKU를 만들지 않고 정책으로 처리한다.
12. 실제 물류 단위가 다를 때만 별도 SKU를 만든다.
13. 외부 채널 연동은 내부 SKU를 기준으로 매핑한다.

## 25. 설계 결정 — "단일상품도 항상 SKU 1개"

Part 1 §3.1 의 원칙(*단일상품 = SKU 1개인 상품*)을 **구현 레벨에서도 그대로 채택**한다. 즉 **모든 상품은 `product_sku` 에 최소 1행을 가진다**(단일상품 = `is_default=1` SKU 1행).

### 25.1 두 대안

| 대안 | 내용 | 장점 | 단점 |
|---|---|---|---|
| **A. 항상 SKU** (채택) | 단일상품도 SKU 1행. 재고·가격·원가·상태의 유일 기준은 항상 `product_sku` | 장바구니·주문·재고차감·복합상품 구성이 **단일 코드 경로**. 마이그레이션이 1회성 백필 | `products` 단순 조회에도 SKU 조인 1회 추가 |
| B. 옵션 있을 때만 SKU | 단일상품은 `products` 가 재고·가격의 집, 옵션상품만 SKU 생성 | 단일상품 조인·백필 부담 없음 | 재고 경로가 **둘로 분기**(products vs sku). **복합상품 구성이 SKU 를 참조**하는데 단일상품이 SKU 가 없으면 `composite_component` 가 다형(product_id \| sku_id)이 되거나 지연 생성 필요 → 더 복잡 |

### 25.2 A 를 택한 이유

이 문서의 목적 자체가 **복합상품(묶음/세트/기획)** 이고, 복합상품 구성은 반드시 **SKU 를 참조**한다(§13, §19.4). B 를 택하면 "단일상품이 세트 구성품이 되는 순간" SKU 가 필요해져 결국 다형 참조나 지연 생성이 생긴다. **조인 1회 비용 < 재고·구성 경로 이원화 비용**이고, 몰 빌더는 같은 구조를 반복 생성하므로 **경로가 하나인 쪽**이 장기 유지비가 낮다.

> 단, `products.price`·`stock`(+ `purchase_price`) 는 **당장 제거하지 않는다**. 대표 SKU 값을 **단방향 미러링**(SKU 쓰기 → products 미러)하는 하위호환 컬럼으로 유지하다가 Phase 6 에서 폐기(읽기 폴백 전용)한다. **`products.status` 는 폐기 대상이 아니다**(상품 생명주기 게이트로 계속 사용). → §29 마이그레이션.

## 26. 목표 스키마 (to-be DDL)

> 컨벤션: 상품/SKU 참조 FK 는 `INT`, `mall_id BIGINT NOT NULL DEFAULT 1` 스코핑, `utf8mb4_general_ci`, snake_case. 기존 `products`/`order_items` 결에 맞춤.

### 26.1 상품 유형 플래그 (products 확장)

```sql
ALTER TABLE products
  ADD COLUMN product_type ENUM('SINGLE','OPTION','BUNDLE','SET','GIFT_SET','BUILD_SET')
      NOT NULL DEFAULT 'SINGLE' COMMENT '상품 형태' AFTER category_id;
-- price/stock/purchase_price 는 유지(대표 SKU 미러, Phase 6 폐기 예정)
-- status 는 폐기 대상 아님 — 상품 생명주기로 계속 사용
```

### 26.2 SKU — 재고·거래의 기준 단위

```sql
CREATE TABLE product_sku (
  id            INT NOT NULL AUTO_INCREMENT,
  mall_id       BIGINT NOT NULL DEFAULT 1,
  product_id    INT NOT NULL COMMENT '소속 상품(products.id, INT)',
  sku_code      VARCHAR(100) DEFAULT NULL COMMENT '내부 SKU 코드',
  barcode       VARCHAR(100) DEFAULT NULL,
  supplier_code VARCHAR(100) DEFAULT NULL COMMENT '공급처 상품코드',
  purchase_price INT DEFAULT 0 COMMENT '원가',
  price         INT NOT NULL COMMENT '판매가',
  stock         INT NOT NULL DEFAULT 0,
  stock_managed TINYINT(1) NOT NULL DEFAULT 1 COMMENT '0=재고를 구성 SKU에서 파생(복합상품 대표 SKU)',
  status        ENUM('ON','OFF') NOT NULL DEFAULT 'ON' COMMENT 'SKU(variant) on/off 만. 생명주기는 products.status',
  is_default    TINYINT(1) NOT NULL DEFAULT 0 COMMENT '단일상품/대표 SKU 여부(상품당 1행)',
  display_order INT NOT NULL DEFAULT 0,
  created_at    TIMESTAMP NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  UNIQUE KEY uk_sku_code (product_id, sku_code),
  KEY idx_sku_product (product_id),
  KEY idx_sku_mall (mall_id),
  CONSTRAINT fk_sku_product FOREIGN KEY (product_id) REFERENCES products (id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_general_ci COMMENT='상품 SKU(재고·거래 단위)';
```

> **SKU 로 내리는 건 `purchase_price`·`price`·`stock` 셋.**
> - `products.status` 는 **상품 생명주기**(`ON/OFF/SOLD_OUT/COMING_SOON/RESTOCK`, 5값)라 **Product 에 그대로 둔다**. SKU 의 `status` 는 variant on/off(2값)일 뿐이다.
> - **구매가능 판정 = `products.status` 게이트(구매가능 상태) AND `sku.status='ON'` AND (재고관리 시 `sku.stock>0`)** 의 합성. status 를 SKU 로 내려 5값을 3값에 뭉개는 실수를 피한다.
> - `products.original_price`·`discount_rate` 도 **표시 전용**이라 상품에 남긴다.

### 26.3 옵션 사전 (표준 옵션명/옵션값 — mall 스코프)

```sql
  option_name  VARCHAR(50) NOT NULL COMMENT '기본 표시명(색상)',
  input_type   ENUM('SELECT','TEXT') NOT NULL DEFAULT 'SELECT',
  is_active    TINYINT(1) NOT NULL DEFAULT 1,
  display_order INT NOT NULL DEFAULT 0,
  PRIMARY KEY (id),
  UNIQUE KEY uk_option_def (mall_id, option_code)

CREATE TABLE option_value_definition (
  id            INT NOT NULL AUTO_INCREMENT,
  option_definition_id INT NOT NULL,
  value_code    VARCHAR(50) NOT NULL COMMENT 'BLACK',
```

### 26.8 복합상품 구성 (묶음/세트/기획, §13·§19.4)

복합상품은 `products.product_type ∈ (BUNDLE,SET,GIFT_SET,BUILD_SET)` 인 **Product** 다. 자기 자신도 **대표 SKU 1행**(`is_default=1`, `stock_managed=0`)을 가져 판매가·장바구니·주문 경로를 일반 상품과 동일하게 태운다. **재고는 보유하지 않고 구성 SKU 에서 파생**한다(§15).

```sql
CREATE TABLE composite_component (
  id            INT NOT NULL AUTO_INCREMENT,
  composite_product_id INT NOT NULL COMMENT '복합상품 products.id',
  component_sku_id INT NOT NULL COMMENT '구성 SKU(product_sku.id)',
  quantity      INT NOT NULL DEFAULT 1 COMMENT '세트당 필요 수량',
  is_optional   TINYINT(1) NOT NULL DEFAULT 0 COMMENT '선택형 세트(BUILD_SET)용',
  display_order INT NOT NULL DEFAULT 0,
  PRIMARY KEY (id),
  UNIQUE KEY uk_comp (composite_product_id, component_sku_id),
  KEY idx_comp_sku (component_sku_id),
  CONSTRAINT fk_comp_product FOREIGN KEY (composite_product_id) REFERENCES products (id) ON DELETE CASCADE,
  CONSTRAINT fk_comp_sku FOREIGN KEY (component_sku_id) REFERENCES product_sku (id) ON DELETE RESTRICT
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_general_ci COMMENT='복합상품 구성(기존 SKU 참조)';
```

> `ON DELETE RESTRICT`: 세트에 묶인 구성 SKU 는 함부로 못 지운다(세트가 조용히 깨지는 것 방지). Part 1 카테고리 부모삭제 가드와 같은 사고방식.

### 26.9 외부 채널 매핑 — **후속 과제(이번 범위 제외)**

> **Shopify 논외.** 아래는 SKU 도입이 완료된 뒤 외부 연동을 다시 켤 때의 방향만 기록한다. **이번 개발에서는 구현하지 않는다.**

Part 1 §21 의 `channel_*_mapping` 은 **개념 예시**다. yd-mall 엔 이미 `shopify_product_mappings`·`mall_channel_credential`·`mall_channel_setting`·`naver_category`·`naver_brand` 가 있으므로, 훗날 연동을 재개하면 **새 테이블을 만들지 말고 기존 매핑을 SKU 레벨로 확장**한다(예: `shopify_product_mappings` 에 `sku_id` 추가, `uk_product_id` 를 `(product_id, sku_id)` 로 완화). 지금은 손대지 않는다.

## 27. 재고·가용수량 계산 규칙

| 상품 형태 | 대표 SKU `stock_managed` | 가용수량 |
|---|---|---|
| 단일/옵션 | 1 | 각 SKU 의 `stock` 직접 |
| 복합(묶음/세트/기획) | 0 | 아래 파생식 |

**복합상품 가용수량**(§20):

```text
available(composite) = min over 구성 c of  floor(sku_stock(c.component_sku_id) / c.quantity)
                       (단, is_optional=1 인 구성은 min 계산에서 제외)
```

**주문 확정 시 재고 차감**(§29 Phase 3):
- 단일/옵션: 주문된 SKU 의 `stock -= quantity`
- 복합: 대표 SKU 는 건드리지 않고 **각 구성 SKU** `stock -= quantity × component.quantity`

**정합성**: 결제 트랜잭션 안에서 `SELECT ... FOR UPDATE` 로 대상 SKU(복합이면 구성 SKU 전부) 잠금 후 차감. 현행 `checkoutController.js` 의 트랜잭션 블록을 그대로 활용한다.

## 29. 마이그레이션 단계 (비파괴 → 전환 → 폐기)

라이브 개발 데이터가 있으므로 **비파괴 증분**으로 간다. 각 Phase 는 독립 배포 가능.

- **Phase 0 — 스키마 추가(무중단)**: §26 의 `CREATE TABLE` + `ALTER ... ADD COLUMN` 전부. 기존 컬럼·경로 그대로. 앱 동작 변화 없음.
- **Phase 1 — 백필**: 상품 전건에 대해 (재실행 안전 — 대표 SKU 없는 상품만)
  ```sql
  INSERT INTO product_sku (mall_id, product_id, sku_code, purchase_price, price, stock, status, is_default)
  SELECT p.mall_id, p.id, p.product_code, p.purchase_price, p.price, p.stock,
         IF(p.status = 'OFF', 'OFF', 'ON'),   -- SKU status 는 on/off 만. 생명주기(SOLD_OUT/COMING_SOON/RESTOCK)는 products.status 유지
         1
  FROM products p
  WHERE NOT EXISTS (
    SELECT 1 FROM product_sku s WHERE s.product_id = p.id AND s.is_default = 1
  );
  ```
  `products.product_type` 은 전건 `SINGLE` 유지. `products.status` 는 그대로 둔다(SKU 로 안 내림). (Shopify 매핑 백필은 논외.)
- **Phase 2 — 읽기 경로 SKU 우선**: 조회 시 대표 SKU 값을 우선 사용. 쓰기는 **단방향 미러**(SKU 쓰기 → `products.price/stock` 미러 갱신). 전환기 동안 레거시 코드의 products 직접 쓰기만 폴백 허용. `products.status` 는 미러 아님(원천 그대로).
- **Phase 3 — 쓰기 경로 SKU 기준**: §28 의 재고차감·복원·cart·checkout 을 SKU 기준으로 전환. **옵션상품 등록 UI** 오픈(`product_option`/`product_option_value`/`sku_option_value` CRUD). (Shopify 웹훅 경로는 논외.)
- **Phase 4 — 복합상품**: `composite_component` 기반 BUNDLE/SET/GIFT_SET 등록·가용수량 파생·구성 SKU 차감. 대표 SKU `stock_managed=0`.
- **Phase 5 — 카테고리 옵션 템플릿**: `option_definition`/`category_option` 시드 + 상속(§8) + 등록화면 추천 노출(§18). 선택형 세트(BUILD_SET)·`category_option_value`(옵션) 는 여기서.
- **Phase 6 — 레거시 폐기**: `products.price/stock/purchase_price` 를 **읽기 폴백 전용**으로 강등(신규 코드 쓰기 금지, 문서화). 물리 삭제는 하지 않는다. **`products.status` 는 폐기 대상이 아니다**(생명주기 게이트로 계속 사용).

각 Phase 후 스모크: 상품 목록/상세 → 장바구니 → 결제 → 재고 차감/복원 경로. (Shopify 논외.)

## 30. MVP 범위 제안

빌더 데모로 "옵션·세트가 도는 것"을 보여주는 최소 범위:

1. Phase 0~1 (스키마 + 백필) — 기존 상품 무영향 확인
2. Phase 3 옵션상품 (색상/사이즈 정도) — 상세페이지 옵션 선택 → SKU 재고/가격
3. Phase 4 세트상품 1종 — 구성 SKU 파생 재고 + 주문 시 구성 차감

카테고리 옵션 상속(Phase 5)은 **후속**, 외부 채널(Shopify) SKU 매핑은 **범위 제외**. 이 3개만으로 Part 1 의 핵심(단일·옵션·세트 + SKU 재고 기준)이 실물로 증명된다.

## 31. 기본상품 vs 파생상품 — 관리 메뉴 분리 (Part 1 §2·§15·§16 반영)

Part 1 은 상품을 **두 풀**로 나눈다(§2).

```text
기본 상품 풀 = 단일상품 + 옵션상품           ← 재고·원가의 원천(SKU)
판매 상품 풀 = 기본 상품 + 묶음·세트·기획상품  ← 기본 SKU 를 참조해 만든 파생상품
```

즉 **세트·묶음·기획상품은 기본상품을 원천으로 "따로 만드는" 파생상품**(§15·§16)이다. 따라서 관리 화면도 둘로 나눈다.

### 31.1 분류 기준 (product_type)

| 구분 | product_type | 관리 메뉴 | 재고·가격 |
|---|---|---|---|
| **기본상품** | `SINGLE`, `OPTION` | `상품관리` (/admin/products) | 자기 SKU 가 원천 |
| **파생상품** | `BUNDLE`, `SET`, `GIFT_SET`, `BUILD_SET` | `세트·기획상품` (/admin/derived-products) | 구성 SKU 에서 파생 |

> **옵션상품이 기본상품에 남는 이유**: 옵션은 "같은 상품을 여러 SKU 로 쪼갠 것"(단일상품의 확장)이지 다른 상품을 묶은 게 아니다. 반면 세트·묶음은 **서로 다른(또는 복수) 기본 SKU 를 조합**한 별개 상품이다(§14). 그래서 옵션은 `상품관리`에 함께, 파생은 별도 메뉴로 둔다.

### 31.2 관리자 메뉴 변경

- **`상품관리`(/admin/products)** — 목록을 `product_type IN ('SINGLE','OPTION')` 으로 **기본상품만** 노출. 파생상품은 제외. 상세의 "세트·묶음 구성" 진입은 제거(기본상품을 세트로 바꾸지 않는다). "옵션·SKU 관리"는 유지.
- **`세트·기획상품`(/admin/derived-products)** — 신규 메뉴(admin_menus 그룹 32 하위). 파생상품 전용 목록·생성·구성.
  - 목록: 파생상품만(`product_type IN (BUNDLE,SET,GIFT_SET,BUILD_SET)`) + 유형 배지 + 구성 개수 + **파생 가용수량**.
  - 생성: `새 세트·기획상품` 폼(상품명·유형·판매가) → `products`(product_type) + 대표 SKU(`stock_managed=0`) 생성 → 구성 편집기로 이동.
  - 구성 관리: `compositeService` 편집기(구성 SKU 검색·수량). 진입점이 파생상품 메뉴로 이동(경로 `/admin/derived-products/:id/compose`).
  - 기본정보(상품명·가격·이미지·상세) 수정은 상품 편집 폼을 공유(같은 `products` 행).

### 31.3 데이터 모델은 그대로

스키마 변경 없음. `product_type` 이 이미 분류 축이고, 파생 구성은 `composite_component`(§26.8) 그대로다. **분리는 "관리 UI 층"에서만** 이뤄진다 — 고객 스토어프론트는 둘 다 하나의 Product 로 노출된다(§15).

### 31.4 파급

- 기본 `상품관리` 목록 쿼리에 유형 필터 1줄 추가.
- 파생상품 목록은 재고를 **파생 가용수량**으로 표기(기본 목록의 `products.stock` 부정확 이슈를 파생 쪽에서 정본화).
- `composite` 라우트·진입점을 `/admin/products/composite/*` → `/admin/derived-products/:id/compose` 로 이동.

## 32. 카테고리 계층 — 몰 카테고리 ↔ 네이버 카테고리 분리

### 32.1 우리 몰 카테고리 vs 네이버 카테고리 (별개 축)

| | 우리 몰 카테고리 | 네이버 카테고리 |
|---|---|---|
| 저장 | `categories`(mall 스코프, 계층) | `naver_category`(참조 리소스) + 상품의 `products.naver_category_id`(매핑) |
| 최대 단계 | **3단계** (`navigation_config.category_max_depth`, 기본 3) | **4단계** (대>중>소>세. 리소스 데이터 기준 level 1~4) |
| 성격 | 우리가 자유롭게 만드는 전시·분류 트리 | 네이버가 정의한 고정 트리(임의 생성 불가) |

**결론(사용자 방침)**: 우리 카테고리는 우리 것(최대 3단계)으로, 네이버 카테고리는 네이버용으로 **따로 저장**한다. 상품에는 `products.naver_category_id` 로 네이버 리프를 **매핑**만 한다.

### 32.2 네이버에 "우리 카테고리" 등록 불가 (확인됨)

네이버 커머스 API 상품 등록 시 `leafCategoryId` 는 **"전체 카테고리 조회 API"가 반환하는 유효한 네이버 카테고리 ID만** 허용한다(임의값 불가). 게다가 **등록 후 대분류 변경 불가**. → 우리 트리를 네이버에 만들 수 없고, 상품을 네이버 **기존 리프에 매핑**해야 한다. (출처: 네이버 commerce-api Discussions #246, #1881)

### 32.3 카테고리 계층 단계별 생성 (구현됨)

- 등록 폼 "직접 입력" 칸에 **`대>중>소` 경로**(구분자 `>`)를 넣으면 `taxonomyResolver.resolveOrCreatePath` 가 **부모-자식 체인을 단계별 생성/재사용**한다(같은 부모 아래 동명 재사용, depthGuard 경유).
- 네이버 위젯에서 리프를 고르면 `whole_category_name`(4단계) 중 **마지막 3단계**만 경로 칸에 채운다(대분류 자동 생략). 예: `패션잡화>여성신발>부츠/워커>미들부츠` → `여성신발>부츠/워커>미들부츠` 로 3단계 생성.
- 몰 최대 뎁스 초과분은 **앞(대분류)을 잘라** 가장 구체적인 리프를 보존.

### 32.4 카테고리 단계별 종속 셀렉트 (구현됨)

- 상품 등록/수정 폼의 카테고리 선택을 **1단계→2단계→3단계 종속 드롭다운**으로 제공. 상위 선택 시 하위가 채워지고, 가장 구체적으로 선택한 레벨의 id 가 `category_id` 로 제출된다.
- 편집 진입 시 상품의 `category_id` 로 조상 체인을 역추적해 셀렉트를 프리필.
- 제출용 원본 `select[name=category_id]` 는 숨겨 두고 종속 셀렉트가 값을 채운다(기존 import-url·네이버 위젯 JS 호환).

### 32.5 후속 과제 (미구현)

**상품 네이버 전송 시 카테고리 자동선택 안내** — 상품에 네이버 카테고리가 매핑돼 있지 않으면, **우리 카테고리 경로를 근거로 네이버 카테고리를 자동 추천**하고, 확정 가능한 단계까지 좁혀 **관리자가 최종 선택**하도록 안내하는 흐름. (전송 기능과 함께 별도 구현)
