# 쇼핑몰 빌더 메뉴 구조 설계 요약

## 1. 결론

쇼핑몰 빌더에서 모든 메뉴를 완전 동적으로 만드는 것은 과설계 가능성이 높다.  
현재 목표가 “기본 쇼핑몰 구조는 유지하되, 쇼핑몰별로 카테고리와 일부 메뉴 구성을 조정할 수 있는 구조”라면 다음 방식이 가장 적절하다.

```text
카테고리 메뉴 = 동적 관리
일반 메뉴 = 사전 정의된 메뉴를 사용 여부로 선택
커스텀 메뉴 = 제한된 개수만 허용
시스템 메뉴 = 고정
```

즉, 메뉴 설계의 방향은 **완전 자유형 메뉴 빌더**가 아니라 **통제된 동적 메뉴 시스템**이어야 한다.

---

## 2. 메뉴를 완전 동적으로 만들지 않는 이유

완전 동적 메뉴는 자유도가 높지만 쇼핑몰 빌더에서는 다음 문제가 발생할 수 있다.

| 문제 | 설명 |
|---|---|
| 운영자 실수 가능성 | 잘못된 URL, 중복 메뉴, 비활성 기능 연결 가능 |
| UX 일관성 저하 | 쇼핑몰마다 메뉴 구조가 과도하게 달라져 사용자 탐색성이 떨어짐 |
| QA 범위 증가 | 모든 메뉴 조합을 테스트해야 하므로 검증 비용 증가 |
| SEO 관리 어려움 | 중요 페이지 URL과 메뉴 노출 구조가 불안정해질 수 있음 |
| 기능 모듈 연결 복잡 | 메뉴와 실제 기능 페이지 간 매핑 관리가 복잡해짐 |
| 개발비 증가 | 메뉴 권한, 노출 조건, 라우팅, 예외 처리 개발 범위 증가 |

따라서 쇼핑몰 빌더에서는 “가능한 모든 메뉴를 자유 생성”하는 방식보다, **자주 쓰는 메뉴는 기능 모듈로 표준화하고 사용 여부만 선택**하는 구조가 더 안전하다.

---

## 3. 권장 메뉴 분류

쇼핑몰 메뉴는 다음 4가지로 분리하는 것이 좋다.

```text
1. 카테고리 메뉴
2. 일반 기능 메뉴
3. 커스텀 메뉴
4. 시스템 메뉴
```

| 구분 | 관리 방식 | 예시 |
|---|---|---|
| 카테고리 메뉴 | 동적 관리 | 패션, 식품, 뷰티, 가전, 생활용품 |
| 일반 기능 메뉴 | ON/OFF 선택 | 오늘특가, 베스트, 기획전, 이벤트, 브랜드 |
| 커스텀 메뉴 | 제한적 추가 | 시즌전, 브랜드위크, 오픈이벤트 |
| 시스템 메뉴 | 고정 | 로그인, 마이쇼핑, 장바구니, 검색, 고객센터 |

---

## 4. 카테고리 메뉴 설계

카테고리는 쇼핑몰마다 상품군이 다르기 때문에 반드시 동적으로 관리해야 한다.

### 예시

```text
패션몰
 ├─ 여성의류
 ├─ 남성의류
 ├─ 신발
 ├─ 가방
 └─ 액세서리

식품몰
 ├─ 신선식품
 ├─ 가공식품
 ├─ 건강식품
 ├─ 간편식
 └─ 정기배송

뷰티몰
 ├─ 스킨케어
 ├─ 메이크업
 ├─ 헤어케어
 ├─ 바디케어
 └─ 향수
```

### 권장 제한

```text
1depth: 5~10개 권장
2depth: 각 5~15개 권장
3depth: 필요 시 허용
4depth 이상: 비추천 -> 현재 카테고리 제한을 4-5 뎁스인데 이에 대해 3뎁스로 고정
```

카테고리는 자유롭게 구성하되, 메뉴 깊이와 개수는 제한해야 한다.  
카테고리 depth가 과도하게 깊어지면 메가메뉴, 모바일 메뉴, 검색 필터, SEO 구조 모두 복잡해진다.

---

## 5. 일반 메뉴 설계

일반 메뉴는 완전 동적 생성보다 **사전 정의된 기능 메뉴를 ON/OFF로 선택**하는 방식이 적절하다.

### 추천 일반 메뉴

| 메뉴 코드 | 기본 메뉴명 | 설명 | 기본 URL |
|---|---|---|---|
| TODAY_DEAL | 오늘특가 | 기간 한정 특가 상품 | `/deal/today` |
| BEST | 베스트 | 판매량/조회수/리뷰 기준 인기 상품 | `/best` |
| EVENT | 이벤트&혜택 | 이벤트, 쿠폰, 혜택 모음 | `/event` |
| EXHIBITION | 기획전 | 시즌전, 브랜드전, 테마전 | `/exhibition` |
| BRAND | 브랜드 | 브랜드별 상품 탐색 | `/brand` |
| NEW_PRODUCT | 신상품 | 최근 등록 상품 | `/new` |
| GROUP_BUY | 공동구매 | 목표 수량/기간 기반 공동구매 | `/group-buy` |
| LIVE | 쇼핑라이브 | 라이브 방송 또는 영상 상품 | `/live` |
| MEMBERSHIP | 멤버십 | 회원 등급, 적립, 혜택 안내 | `/membership` |
| COUPON | 쿠폰 | 다운로드 가능한 쿠폰 목록 | `/coupon` |
| OUTLET | 아울렛 | 할인/재고 소진 상품 | `/outlet` |
| RANKING | 랭킹 | 카테고리별 순위 상품 | `/ranking` |

### 일반 메뉴 관리 항목

```text
- 사용 여부
- 메뉴명 변경
- 노출 순서
- PC 노출 여부
- Mobile 노출 여부
- 로그인 필요 여부
- 노출 기간
- 연결 기능 모듈
```

### 예시

| 기능 메뉴 | 사용 여부 | 표시 메뉴명 | URL |
|---|---:|---|---|
| TODAY_DEAL | 사용 | 오늘특가 | `/deal/today` |
| BEST | 사용 | 베스트 | `/best` |
| EVENT | 사용 | 이벤트&혜택 | `/event` |
| LIVE | 미사용 | 쇼핑라이브 | `/live` |
| GROUP_BUY | 사용 | 공동구매 | `/group-buy` |

### 일반 메뉴를 ON/OFF 방식으로 관리해야 하는 이유

| 항목 | 완전 동적 메뉴 | ON/OFF 일반 메뉴 |
|---|---|---|
| URL 관리 | 운영자가 직접 입력 | 기능별 고정 URL 사용 |
| 기능 연결 | 수동 매핑 필요 | 모듈과 자동 연결 |
| QA | 조합이 많아짐 | 테스트 범위 축소 |
| 운영 안정성 | 낮음 | 높음 |
| SEO 구조 | 불안정 가능 | 예측 가능 |
| 관리자 편의성 | 복잡 | 단순 |

## 6. 커스텀 메뉴 설계

일반 메뉴를 ON/OFF 방식으로 제한하더라도 쇼핑몰별 개성을 위해 커스텀 메뉴는 필요하다.  
다만 커스텀 메뉴는 무제한이 아니라 **슬롯 방식**으로 제한하는 것이 좋다.

### 추천 커스텀 메뉴 예시

```text
- 설 선물대전
- 추석 선물대전
- 여름 클리어런스
- 브랜드 위크
- 오픈 이벤트
- 앱 전용 혜택
- 신규회원 전용관
- 타임세일
- 한정수량 특가
- 시즌 기획전
```

### 권장 제한

```text
Header GNB 커스텀 메뉴: 최대 3개
메가메뉴 내부 프로모션 링크: 최대 5개
Footer 메뉴: 비교적 자유롭게 허용
모바일 하단/퀵 메뉴: 최대 3~5개
```

### 커스텀 메뉴 관리 항목

```text
- 메뉴명
- 링크 유형: 내부 링크 / 외부 링크
- 연결 URL
- 노출 위치
- 노출 순서
- 노출 기간
- PC 노출 여부
- Mobile 노출 여부
- 로그인 필요 여부
- 새 창 열기 여부
```

### 커스텀 메뉴 위치 구분

| 위치 | 허용 수준 | 설명 |
|---|---|---|
| Header GNB | 제한적 허용 | 주요 메뉴 영역이므로 최대 3개 권장 |
| 메가메뉴 내부 | 허용 | 기획전/배너 링크 용도 |
| Footer | 비교적 자유 | 회사소개, 입점문의, 정책 문서 등 |
| 모바일 메뉴 | 제한적 허용 | 화면 공간이 좁으므로 최소화 |
| 퀵 메뉴 | 제한적 허용 | 쿠폰, 이벤트, 최근본상품 등 |

## 7. 시스템 메뉴 설계

시스템 메뉴는 쇼핑몰 핵심 기능과 직접 연결되므로 운영자가 자유롭게 삭제하거나 URL을 임의 변경하지 못하게 하는 것이 좋다.

### 고정 시스템 메뉴

```text
- 로그인
- 회원가입
- 마이쇼핑
- 장바구니
- 주문/배송조회
- 고객센터
- 검색
- 최근 본 상품
- 찜한 상품
```

### 관리 방식

| 시스템 메뉴 | 관리 방식 |
|---|---|
| 로그인 | 고정 |
| 회원가입 | 고정 또는 사용 여부 선택 |
| 마이쇼핑 | 고정 |
| 장바구니 | 고정 |
| 주문/배송조회 | 고정 |
| 고객센터 | 고정 또는 ON/OFF |
| 검색 | 위치/사용 여부만 선택 |
| 최근 본 상품 | 퀵 메뉴 사용 여부 선택 |
| 찜한 상품 | 사용 여부 선택 |

시스템 메뉴는 기능 안정성이 중요하므로 URL과 권한 체계는 고정하고, 노출 여부나 위치 정도만 관리하는 것이 좋다.

## 8. Header 메뉴 구조 제안

첨부 이미지와 유사한 기본 구조를 유지한다면 Header는 다음 구조가 적절하다.

```text
[로고] [검색창] [로그인] [마이쇼핑] [장바구니] [고객센터]

[카테고리] [일반 메뉴 1] [일반 메뉴 2] [일반 메뉴 3] [일반 메뉴 4] [커스텀 메뉴]
```

### 영역별 관리 방식

| 영역 | 관리 방식 | 설명 |
|---|---|---|
| 로고 | 설정 기반 | 몰별 로고 등록 |
| 검색창 | 고정 또는 사용 여부 | 위치는 템플릿에서 통제 |
| 로그인 | 시스템 고정 | 인증 모듈 연결 |
| 마이쇼핑 | 시스템 고정 | 마이페이지 연결 |
| 장바구니 | 시스템 고정 | 장바구니 연결 |
| 고객센터 | 시스템 메뉴 | 사용 여부 선택 가능 |
| 카테고리 | 동적 트리 | 쇼핑몰별 상품군 구성 |
| 일반 메뉴 | ON/OFF | 기능 메뉴 선택 |
| 커스텀 메뉴 | 제한적 추가 | 최대 3개 권장 |

## 9. DB 설계 방향

메뉴를 하나의 `menu` 테이블에 모두 넣는 방식은 초기에는 단순해 보이지만, 기능 메뉴와 카테고리, 시스템 메뉴, 커스텀 메뉴가 섞이면서 복잡도가 증가한다.

따라서 다음처럼 분리하는 것이 좋다.

```text
category
feature_menu
mall_feature_menu
custom_menu
navigation_config
```

### category

```text
category
 ├─ id
 ├─ mall_id
 ├─ parent_id
 ├─ name
 ├─ slug
 ├─ depth
 ├─ sort_order
 ├─ is_active
 ├─ pc_visible
 ├─ mobile_visible
 └─ seo_config
```

### feature_menu

```text
feature_menu
 ├─ id
 ├─ feature_code
 ├─ default_name
 ├─ default_path
 ├─ required_module
 └─ is_system
```

### mall_feature_menu

```text
mall_feature_menu
 ├─ id
 ├─ mall_id
 ├─ feature_code
 ├─ display_name
 ├─ sort_order
 ├─ is_enabled
 ├─ pc_visible
 ├─ mobile_visible
 ├─ login_required
 └─ visible_period
```

### custom_menu

```text
custom_menu
 ├─ id
 ├─ mall_id
 ├─ display_name
 ├─ link_type
 ├─ link_url
 ├─ sort_order
 ├─ location
 ├─ is_enabled
 ├─ pc_visible
 ├─ mobile_visible
 └─ visible_period
```

### navigation_config

```text
navigation_config
 ├─ id
 ├─ mall_id
 ├─ header_layout_type
 ├─ category_display_type
 ├─ max_gnb_items
 ├─ max_custom_items
 ├─ use_mega_menu
 ├─ use_search_bar
 └─ config_json
```

## 10. 관리자 화면 구성 제안

```text
메뉴 관리
 ├─ 카테고리 관리

 ├─ 커스텀 메뉴 관리
 ├─ 시스템 메뉴 노출 설정
 └─ Header/Footer 설정
```

### 카테고리 관리

```text
- 카테고리 추가/수정/삭제
- 상위/하위 카테고리 관리
- 노출 순서 변경
- PC/Mobile 노출 여부
- 메가메뉴 이미지/배너 설정
- SEO 제목/설명 설정
```

### 일반 메뉴 관리

```text
- 오늘특가 사용 여부
- 베스트 사용 여부
- 이벤트 사용 여부
- 기획전 사용 여부
- 브랜드 사용 여부
- 신상품 사용 여부
- 공동구매 사용 여부
- 라이브 사용 여부
- 쿠폰 사용 여부
- 멤버십 사용 여부
- 메뉴명 변경

```

### 커스텀 메뉴 관리

```text
- 최대 3~5개 추가
- 내부 링크/외부 링크 선택
- 노출 위치 선택
- 노출 기간 설정
- PC/Mobile 노출 여부
- 로그인 필요 여부
```

## 11. 최종 권장안

가장 현실적인 쇼핑몰 빌더 메뉴 구조는 다음과 같다.

```text
1. Header 구조는 기본 템플릿으로 고정
2. 카테고리는 쇼핑몰별 동적 트리로 관리
3. 일반 메뉴는 사전 정의된 기능 메뉴를 ON/OFF로 선택
4. 일반 메뉴의 메뉴명과 순서는 수정 가능
5. 커스텀 메뉴는 최대 3~5개만 허용
6. 로그인/마이쇼핑/장바구니/검색은 시스템 메뉴로 고정
7. Footer는 Header보다 자유롭게 관리
8. 모바일 메뉴는 별도 노출 설정만 허용
9. 모든 메뉴를 하나의 menu 테이블에 넣지 않고 용도별로 분리
10. 기능 메뉴는 URL과 모듈 연결을 표준화
```

## 12. 한 줄 요약

**카테고리는 유연하게, 일반 메뉴는 ON/OFF로 통제하고, 커스텀 메뉴는 제한적으로 허용하는 구조가 쇼핑몰 빌더에 가장 적합하다.**

이 방식이 개발비, 운영 안정성, UX, SEO, 확장성 측면에서 가장 균형이 좋다.
