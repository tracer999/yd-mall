## ⚠️ 설계와 실제 구현이 갈린 지점 (먼저 읽을 것)

상세설계·이 계획서의 아키텍처를 그대로 믿으면 안 된다. 구현하면서 아래를 **의도적으로 단순화**했다.

| 설계 | 실제 구현 | 이유 |
|---|---|---|
| `integration_outbox` + `worker.js` + PM2 2앱(`yd-mall-worker`) | **도입하지 않음.** 테이블·워커 파일·PM2 앱 모두 없다. 등록·수집은 요청 안에서 동기 처리 | 1차 범위가 "버튼을 누르면 그 시점에 실행"이라 비동기 큐가 필요 없었다. 2차(배치)를 켤 때 다시 판단한다 |
| `builder_product` / `builder_variant` 별도 네임스페이스 | **없음.** staging(`supplier_product`)에서 기존 `products` / `product_sku` 로 직접 승격한다(`publishService`) | 별도 상품 테이블을 두면 옵션·재고·가격 로직이 두 벌이 된다 |
| `category_mapping` · `pricing_policy` 테이블 | **없음.** 카테고리는 `taxonomyResolver` + `naver_category` 매핑으로, 가격은 승격 시점 계산으로 처리 | |
| `naver_category_notice_schema` / `builder_product_notice` | `naver_notice_schema`(36종) + `naver_publish_profile.notice_defaults_json` + `mall_notice_default` 로 대체 | 상품별 고시 저장소는 **아직 없다**(잔여 — 아래) |
| `channel_variant_mapping` | `channel_sku_mapping` 으로 명명 | |

실제 스키마: `mall_channel_credential` · `mall_channel_setting` · `supplier_product` · `supplier_variant` · `supplier_import_log` · `channel_product_mapping` · `channel_sku_mapping` · `channel_publish_log` · `channel_image_cache` · `naver_*` 7종.

## 완료 — 1차 (Phase 0~3)

| Phase | 상태 |
|---|---|
| **0 계약 확인 + 인증 스파이크** | ✅ 네이버 전자서명 OAuth 실호출 성공. 도매꾹 Open API Key 발급·조회 확인 |
| **1 기반 골격** | ✅ 자격증명(`mall_channel_credential`, `ENC:`) · `/admin/sourcing/connections`(CRUD·검증·다른 몰 복사) · 독립 대메뉴 · 몰별 사용여부 게이팅(`sourcingFlag`). **아웃박스·워커는 제외**(위 표) |
| **2 공급처 상품 가져오기** | ✅ **도매꾹 기준.** 어댑터(`supplier/domeggook/`) · `/admin/sourcing/import` · `/admin/sourcing/staging` · 재수집 멱등성 · 오류 처리 |
| **3 선택 등록 + 스마트스토어 등록** | ⬜ **부분.** 승격(staging → 우리 몰 상품)과 **네이버 등록**(`/admin/sourcing/publish`, `naverPublishService`)은 완료. 역방향 수집·재고/주문 가져오기는 **화면만 있고 미구현**(아래) |

**⏸ 온채널 L1(CSV 임포트) — 보류 (2026-07-20 결정).** 착수하지 않는다. 현재 온채널 선택 시 "API 연동이 아닌 수동·CSV 방식" 안내와 함께 차단된다. **재개 시 이 항목부터 시작**하면 된다.

### Phase 2 에서 얻은 제약 (계속 유효)

- **재판매 금지 상품 경고** — 도매꾹 `desc.license.usable=false` 인 상품을 스마트스토어에 올리면 제재 대상이다. 적재 시 `resale_allowed` 로 저장하고 목록·상세·가져오기 결과에 경고를 띄운다.
- **공급처 상세 HTML 은 외부 마크업** — 관리자 화면에서도 `services/display/htmlSanitizer` 를 통과시켜 렌더한다(저장형 XSS 방지).
- **도매꾹 호출 한도**(분당 180 / 일 15,000) — 프로세스 내 직렬 스로틀(350ms) + 1회 가져오기 상한 50건. ⚠️ 이 350ms 를 네이버에 그대로 쓰면 안 된다(네이버는 600ms — [`네이버_스마트스토어_연동.md`](./네이버_스마트스토어_연동.md) §3).
- 도매꾹은 재고를 `qty.inventory` + 옵션별 `qty` 로 실제 제공하므로 **가상재고 정책은 불필요**했다.
- 위탁가능/마진예상/인증위험 **필터는 미적용** — 재판매 가능여부는 필터 대신 적재 후 경고 배지로 노출.

## 잔여 — 1차 마무리

- **역방향 수집** (`/admin/sourcing/channel-import`) — 화면 라우트만 있고 `fetchChannelProducts`/`getChannelProduct` 가 없다. 스마트스토어에 직접 등록한 상품을 우리 몰로 가져오는 경로.
- **재고·주문 수동 가져오기** (`/admin/sourcing/sync`) — 마찬가지로 화면만 있다. 누르면 그 시점에 공급처 재고·네이버 주문을 조회·표시하는 버튼. **2차 배치가 이 함수를 그대로 재사용**하므로 여기서 함수 형태를 잘 잡아야 한다.
- **상품별 고시 입력 화면·저장소** — 지금은 몰 단위 기본값 + 전송 시점 override 뿐이다. 상품별 고시 값이 저장되면 필터 쪽 "경로 A"([`카테고리_브랜드_상품필터_설계.md`](./카테고리_브랜드_상품필터_설계.md))도 함께 풀린다.

## 잔여 — 2차 (Phase 4~7)

배치 동기화부터가 2차다. 1차의 수동 "가져오기" 로직을 스케줄러가 주기 호출하는 형태이므로 **로직 재작성은 없어야 한다.**

### Phase 4 — 재고·가격 동기화 (규모 M)

공급처 재고/가격 변화를 네이버에 자동 반영. **폴링 스케줄러를 여기서 켠다.** 안전재고·품절 처리·가격 정책 재계산. 스케줄은 `naver_taxonomy_schedule` 처럼 **DB 가 주기를 정하고 크론은 고정 한 줄**인 패턴을 따른다(SSH 없이 관리자 화면에서 주기 변경).

### Phase 5 — 주문 수집 + 발주 (규모 L)

네이버 주문 수집 → 공급처 발주(반자동 폴백 상시 유지) → 발주확인 SLA 관리. **리스크**: 개인정보 전달 경로.

### Phase 6 — 송장 + 클레임 (규모 M)

송장 등록·택배사 코드 매핑, 취소·반품·교환 처리, 감사로그.

### Phase 7 — 운영 안정화 (규모 M)

- 장애센터 — 실패 건 조회·재처리·외부응답 요약.
- 정합성 배치(일 1회) — 상품/재고/주문 대사 + 정산 대사.
- 일일 리포트 + KPI 대시보드.
- 개인정보 보존/비식별화 배치.
- 알림(중대 인증오류·발주실패·송장지연) — 기존 `emailService` 재사용.

## 배포·운영 고려

- **마이그레이션**: `scripts/migrate_sourcing_*.sql` 를 개발 DB 에 적용(로컬·서버 공용 1벌 — 신중히). `admin_menus` INSERT 포함 필수.
- **기능 토글**: `sourcingFlag` 로 미완성 화면을 숨긴다. Shopify 연동(비활성)과 독립.
- **시크릿**: 자격증명은 `ENC:` + `shared/crypto`. `ENCRYPTION_KEY`(`/etc/environment`) 없으면 앱 기동 불가.
- **테스트**: 프레임워크 미도입 상태다. 2차 착수 시 `services/sourcing/core/*`(순수 로직 — 가격·재고·옵션변환·상태매핑·멱등키·택배사코드) 우선으로 Jest 도입을 재검토한다. 화면·어댑터는 통합/수동 병행.
- **외부 전송 테스트는 1건이 기본, 최대 5건**(`CLAUDE.md`). 대량 등록 테스트는 하지 않는다.
- **PM2 는 fork·`instances: 1` 유지.** 스로틀이 프로세스 내 상태라 cluster 로 늘리면 rate limit 이 즉시 깨진다.
