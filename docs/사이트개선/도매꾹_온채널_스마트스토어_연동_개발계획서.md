## 0. 계획 개요

### 0.1 목표

몰(판매자)이 **도매꾹에서 위탁 상품을 소싱 → 빌더에서 편집(가격/고시/카테고리) → 네이버 스마트스토어에 등록 → 재고·가격·주문·발주·송장을 동기화**하는 통합 시스템을, **기존 yd-mall 인프라를 늘리지 않고** 모듈러 모놀리스 + 단일 워커로 구축한다.

### 0.2 원칙

- 인프라 신규 도입 최소화(큐 없이 **MySQL 아웃박스 + PM2 워커**).
- 기존 자산 재사용: `services/shopify` 어댑터 골격, `shared/crypto` 암호화, `mall_id` 스코핑, `admin` RBAC, `cron+DB스케줄` 패턴.
- **관리자 독립 대메뉴 + 몰별 사용 여부(유료) 게이팅.** 이 기능은 기존 상품/주문 흐름과 분리된 최상위 대메뉴로 개발하고, 몰마다 사용 여부를 켠 경우에만 활성(납품 시 추가 계약 대상). 상세 설계서 §2·§5·§21.
- **불확실 구간은 항상 수동/반자동 폴백**을 정식 경로로 유지(자동화는 계약 확인 후 승격).
- 각 Phase 끝에 **동작하는 얇은 수직 슬라이스**를 남긴다(가짜 완성 금지).

### 0.2.1 릴리스 차수 (중요)

| 차수 | 범위 | 특징 |
|---|---|---|
| **1차** | 공급처 **상품 가져오기 → 중간 테이블 적재 → 선택 등록 → 스마트스토어 등록** + **역방향 수집**(스마트스토어 직접등록 상품 → 우리 몰) + **재고·주문 수동 가져오기** | **배치(스케줄 자동 동기화) 없음.** 모든 동작은 화면의 **"가져오기" 버튼**으로 그 시점에 실행. Phase 1~3 |
| **2차** | **재고·주문 동기화 자동화(배치)**, (반)자동 발주, 송장·클레임 | 1차의 "가져오기" 동작을 워커 폴링 스케줄러가 자동 반복. Phase 4~7 |

> **1차의 재고·주문도 "가져오기" 버튼으로 처리한다.** 즉 사용자가 버튼을 누르면 그 시점에 공급처 재고·네이버 주문을 조회/동기화한다(자동 스케줄 아님). **2차는 이 버튼과 동일한 로직을 워커가 주기적으로 자동 호출**하는 것이므로, 1차의 수동 동기화 함수를 2차가 그대로 재사용한다(로직 재작성 없음). 상품 등록/역수집의 신뢰성용 아웃박스+워커 골격(Phase 1)은 1차에 포함하되, **스케줄 폴링 루프는 2차에서 켠다.**

### 0.3 결정 게이트 (Phase 0에서 확정)

| ID | 결정 | 기본값(미정 시) |
|---|---|---|
| D1 | 계약/권한(네이버 앱 유형·도매꾹 Private·온채널 제휴·자동결제) | 자사형 앱 / Open만 / 온채널 L1 / 수동결제 |
| D2 | 큐 방식 | **MySQL 아웃박스** |
| D3 | 테넌트 단위 | **판매자 = 몰 1개** |
| D4 | MVP 카테고리 범위 | 비규제 상품군 3~5개 |
| D5 | 테스트 프레임워크 | **Jest 도입** |

> D1은 외부 사업자 답변이 필요하므로 **Phase 0에서 병렬로 문의를 발송**하고, 답이 늦어도 나머지 Phase는 Mock Adapter로 진행한다(Phase 3의 Mock 우선 원칙).

## 1. 아키텍처 배치 (구현 지도)

```text
services/sourcing/
  core/                 # 도메인(공급처·채널 무관)
    normalize.js        # 공급처 상품 → BuilderProduct/Variant
    pricing.js          # 가격/마진 계산 (설계서 §11 계산식)
    inventory.js        # 안전재고·가상재고·예약재고 계산
    order.js            # 주문 조립·상태 전이
    procurement.js      # 발주 준비·자금·사전검사(부록 B)
  supplier/
    SupplierAdapter.js  # 공통 인터페이스(cred 주입형)
    domeggook/          # Open API 어댑터
    onchannel/          # L1: CSV 임포트 + 수동 발주 기록
  channel/
    SalesChannelAdapter.js
    naver/
      auth.js           # 전자서명 OAuth2 토큰 캐시(설계서 §8.2)
      client.js         # fetch 래퍼 + 에러 정규화(shopify/adminClient 참고)
      products.js inventory.js orders.js claims.js
      channelImport.js  # 역방향: 스마트스토어 채널상품 → 우리 몰 (설계서 §10)
  outbox/
    publish.js          # 트랜잭션과 함께 outbox 적재
    handlers.js         # TOPIC_HANDLERS 맵(shopify/webhookHandler 패턴)
  credential.js         # mall_channel_credential 로드 + 복호화 주입

worker.js               # PM2 app 2 진입점: outbox 소비 + 폴링 스케줄러
controllers/admin/sourcing*Controller.js
routes/admin/sourcing*.js         # routes/admin.js 에 requireMenuAccess 등록
scripts/migrate_sourcing_*.sql    # 스키마 + admin_menus INSERT
```

- `ecosystem.config.cjs` 의 `apps` 에 `yd-mall-worker`(script: `worker.js`, fork, instances:1) 추가.
- `services/shopify` 는 **수정하지 않고 참고만**(패턴 복제). 새 어댑터는 자격증명 주입형으로 신규 작성.

## 3. Phase별 계획

각 Phase: **목표 · 작업 · 산출물 · 완료기준(DoD) · 의존/리스크**. 규모는 상대값(S/M/L)로 표기(절대 일정은 팀 상황에 맞춰 부여).

---

### Phase 0 — 계약 확인 + 인증 스파이크 (규모 S)

**목표**: 개발 리스크가 큰 두 가지(외부 계약·네이버 인증)를 먼저 없앤다.

**작업**
- [ ] D1 문의서 발송(네이버/도매꾹/온채널) — 설계서 §32 체크리스트 사용, 병렬 진행.
- [ ] 네이버 커머스API **인증 PoC** — 전자서명(bcrypt) 토큰 발급 → 상품 1건 조회까지(실 개발 스토어).
- [ ] 도매꾹 Open API Key 발급 → 상품 목록/상세 1건 조회 PoC.
- [ ] D2~D5 결정 확정(미정은 기본값 채택).

**산출물**: PoC 스크립트(`scripts/_poc_naver_auth.js` 등, `_` 접두어로 비추적), 계약 확인 결과표, 결정 게이트 확정 문서.

**DoD**: 네이버·도매꾹 각 1건 실호출 성공 로그 확보. 결정 게이트 5개 값 확정.

**리스크**: D1 답변 지연 → 나머지 Phase는 Mock으로 진행(블로킹 아님).

**목표**: 자격증명 저장, 어댑터 인터페이스, 아웃박스+워커, 관리자 메뉴의 뼈대.

**작업**

- [ ] `services/sourcing/credential.js` — 로드 + `shared/crypto` 복호화 주입 헬퍼.

- [ ] `services/sourcing/outbox/{publish,handlers}.js` — 적재 API + `TOPIC_HANDLERS` 맵(초기 no-op).
- [ ] `worker.js` — 아웃박스 소비 루프(`FOR UPDATE SKIP LOCKED`, 백오프, DLQ) + 폴링 타이머 골격.
- [ ] `ecosystem.config.cjs` 에 `yd-mall-worker` 추가.

**의존**: Phase 0(인증 PoC). **리스크**: 워커 프로세스 배포(PM2 2앱) 검증 — `yd-mall.sh` 기동 흐름 점검.

### Phase 2 — 공급처 상품 가져오기 → 중간 테이블 적재 (규모 L) · **1차**

**목표**: 도매꾹/온채널에서 상품을 검색·가져와 **중간 테이블(`supplier_product`/`supplier_variant`)에만 적재**. 아직 판매 상품화(등록) 아님 — "가져오기 ≠ 상품 등록"(설계서 §7.1).

> ✅ **완료 (2026-07-20 · 도매꾹 기준).** 어댑터(`supplier/domeggook/`) · 가져오기 처리 · `/admin/sourcing/import` · `/admin/sourcing/staging` 구현. 실 API 로 검증(검색·상세·옵션 30건·재수집 멱등성·오류 처리). 스키마 `scripts/migrate_sourcing_02_supplier.sql`.
> 도매꾹은 재고를 `qty.inventory` + 옵션별 `qty` 로 실제 제공하므로 **가상재고 정책은 불필요**했다(설계서 §12.2 리스크 해소).

**⏸ 온채널 L1(CSV 임포트) — 보류 (2026-07-20 결정).** 착수하지 않는다. 현재 온채널 선택 시 "API 연동이 아닌 수동·CSV 방식" 안내와 함께 차단된다. **재개 시 이 항목부터 시작**하면 된다. 그래서 Phase 2 의 DoD 는 온채널을 뺀 **도매꾹 기준으로 충족**된 것으로 본다.

**Phase 2 에서 얻은 제약 (계속 유효)**
- **재판매 금지 상품 경고** — 도매꾹 `desc.license.usable=false` 인 상품을 스마트스토어에 올리면 제재 대상이다. 적재 시 `resale_allowed` 로 저장하고 목록·상세·가져오기 결과에 경고를 띄운다.
- **공급처 상세 HTML 은 외부 마크업** — 관리자 화면에서도 `services/display/htmlSanitizer` 를 통과시켜 렌더한다(저장형 XSS 방지).
- **도매꾹 호출 한도**(분당 180 / 일 15,000) — 프로세스 내 직렬 스로틀(350ms) + 1회 가져오기 상한 50건.
- 위탁가능/마진예상/인증위험 **필터는 미적용** — 마진은 가격정책(Phase 3), 재판매 가능여부는 필터 대신 적재 후 경고 배지로 노출.

### Phase 3 — 선택 등록 + 편집 + 스마트스토어 등록 + 역방향 수집 (규모 L) · **1차**

**목표**: 중간 테이블에서 **선택한 상품을 등록(빌더 승격)** → 편집(가격/고시/이미지) → 네이버 등록 → 검수 추적. 그리고 **역방향**(스마트스토어 직접등록 상품 → 우리 몰) 수집.

**작업 — 선택 등록·편집·채널 등록**
- [ ] `core/normalize.js` + staging "상품 등록" 승격 — 선택한 `supplier_product` → `builder_product`/`builder_variant`(`source_type=SUPPLIER`, DRAFT). "가져오기 ≠ 등록"의 등록 단계.
- [ ] `category_mapping` + 매핑 UI — 공급처→빌더 표준→네이버 **leaf** 3단(추천 우선순위 설계서 §9.4).
- [ ] `/admin/sourcing/products/:id` 편집 — 상품명·이미지·상세·**고시항목**·인증·배송/반품.
- [ ] `pricing.js` + `pricing_policy` — 마진 역산·라운딩·최소이익 가드(설계서 §11).
- [ ] `naver_category_notice_schema`/`builder_product_notice` — 상품군별 필수 고시 검증 게이트.
- [ ] 등록 전 검증(설계서 §22) — 위탁허용·leaf매핑·고시완료·인증·가격>0 하드 게이트.
- [ ] `channel/naver/products.js` — 원상품/채널상품 등록 + 옵션 매핑(`channel_variant_mapping`, ID 기반). 아웃박스 topic `channel.product.publish` 로 비동기.

**작업 — 역방향 수집(신규 요건)**

- [ ] `core/inventory.js` — 공급처 현재 재고 조회 + 안전재고 계산(스케줄 없이 함수만). **2차 배치가 이 함수를 재사용**.

**산출물**: 중간 테이블 상품을 골라 편집 후 네이버 등록(검수 통과 시 `PUBLISHED`). 스마트스토어 상품을 우리 몰로 역수집. 재고·주문을 버튼으로 수동 조회·동기화.

**DoD**: (a) 개발 스토어에 상품 1건 등록 성공 → 매핑 저장 → 검수 상태 반영, 고시 누락 시 차단. (b) 네이버 기존 상품 1건 역수집 → `builder_product`(CHANNEL_IMPORT) 생성. (c) "재고 가져오기"/"주문 가져오기" 버튼으로 그 시점 조회·표시.

**의존**: Phase 2. **리스크(최상)**: 고시·카테고리 필수값 — **D4로 카테고리 범위를 좁혀** 부담 축소.

> **1차는 여기까지.** 상품 가져오기·등록·역수집이 완성되면 M2(수동 운영) 달성. 아래 Phase 4~7은 **2차(배치 동기화)**.

### Phase 4 — 재고·가격 동기화 (규모 M) · **2차 (배치 시작)**

- [ ] `inventory.js` — `판매가능재고 = max(0, 공급처재고 - 안전재고 - 예약재고)`, 가상재고 정책.
- [ ] `channel/naver/inventory.js` — 옵션별 재고/가격 전송(아웃박스 `channel.inventory.update`).

**DoD**: 공급처 재고를 낮추면(테스트) 네이버 옵션 재고가 안전재고 반영해 갱신. 전체 품절 시 판매중지 제안.

## 4. 테스트 전략 (D5: Jest)

- **도입**: `jest` devDependency 추가(현재 테스트 프레임워크 없음). `services/sourcing/**` 우선 커버.
- **단위**(설계서 §27): 가격/재고/옵션변환/상태매핑/카테고리매핑/멱등키/택배사코드/전자서명 생성.
- **계약 테스트**: 각 Adapter가 공통 인터페이스 준수(Capability 기반).
- **Mock Adapter**: 도매꾹/온채널/네이버 Mock(품절·가격변경·토큰만료·주문중복·발주실패·송장지연) — **Phase 3~6는 Mock 우선으로 개발**, 실 API는 개발 스토어로 통합 확인.
- **장애 테스트**: "외부 API 성공 후 내부 저장 실패" 복구(외부 요청 ID + 조회 기반 재동기화) — 아웃박스 멱등으로 대응.
- 커버리지 목표는 `services/sourcing/core/*`(순수 로직) 우선 상향. 화면·어댑터는 통합/수동 병행.

## 5. 배포·운영 고려

- **PM2 2앱 배포**: `yd-mall`(웹) + `yd-mall-worker`(워커). `yd-mall.sh`/`ecosystem.config.cjs` 기동 흐름에 워커 추가 검증(로컬 PM2가 3006 점유 이슈 인지).
- **마이그레이션**: `scripts/migrate_sourcing_*.sql` 를 개발 DB에 적용(로컬·서버 공용 1벌 — 신중히). `admin_menus` INSERT 포함 필수.
- **기능 토글**: `sourcingFlag` 로 미완성 화면 숨김. Shopify 연동(비활성)과 독립.
- **시크릿**: 자격증명은 `ENC:`+`shared/crypto`. `ENCRYPTION_KEY`(/etc/environment) 없으면 앱 기동 불가 규칙 준수.
- **푸시 배포는 사용자 명시 요청 시에만**(main 푸시 = 개발 서버 즉시 반영).

| 마일스톤 | 차수 | 포함 Phase | 사용자 가치 |
|---|---|---|---|
| **M1 기반 검증** | 1차 | 0~1 | 인증 성공 + 대메뉴/사용여부/자격증명/아웃박스 골격 |

## 7. 즉시 착수 가능한 첫 스텝 (Phase 0)

1. D1 문의서 3사 발송(설계서 §32).
2. 네이버 개발 스토어 + 커머스API 애플리케이션 생성 → 인증 PoC.
3. 도매꾹 Open API Key 발급 → 조회 PoC.
4. 결정 게이트 D2~D5 확정(기본값 채택 가능).

이 4개가 끝나면 Phase 1(기반 골격) 코드 착수 조건이 갖춰진다.
