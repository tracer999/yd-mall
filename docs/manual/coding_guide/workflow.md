# 바이브코딩 워크플로우 – 기능 하나를 끝까지 넣는 순서

이 문서는 **비개발자 / 입문자**가 이 쇼핑몰 프로젝트에서

- 새 기능을 아이디어 단계에서
- 실제 화면·버튼이 동작하는 상태까지

**바이브코딩으로 완성하는 전체 흐름**을 정리한 워크플로우입니다.

기본 개념은 이미 다른 문서에 있습니다.

- 기술 스택: `tech_stack`
- Node.js/Express: `nodejs`
- MVC 구조: `mvc`
- MySQL/테이블: `mysql`
- 폴더 구조: `project_structure`
- 프롬프트 쓰는 법: `vibe_coding`

이 문서는 그 내용을 **“순서”**로 엮어 줍니다.

---

## 1. 먼저, 한 번만 하는 준비(프로젝트 세팅)

이 단계는 **처음 프로젝트를 받을 때 한 번만** 하면 됩니다.

1. 저장소 클론
   - `git clone` 으로 프로젝트를 내려받거나, 이미 받은 폴더를 엽니다.

2. 패키지 설치
   - 터미널에서 프로젝트 폴더로 이동 후:
   - `npm install`

3. 환경 변수(.env) 만들기
   - `.env.example` 가 있다면 복사해서 `.env` 로 만들고 값만 채웁니다.
   - 없다면 docs 나 README 를 보고 `MYSQL_HOST`, `MYSQL_USER` 등 필요한 항목을 채웁니다.

4. DB 생성 및 초기 스키마 적용
   - MySQL에 접속해서 프로젝트용 DB를 하나 만듭니다.
   - `tables.sql` 안의 내용을 실행하거나,
   - `scripts/init_db.js` 처럼 초기화 스크립트가 있으면 README 안내에 따라 실행합니다.

5. 개발 서버 실행 테스트
   - `npm run dev` (또는 `npm start`) 로 서버를 켠 뒤,
   - 브라우저에서 `http://localhost:3000` 이 뜨는지 확인합니다.

여기까지 되면, 이제부터는 **“기능 하나씩 추가하는 반복 루프”** 로 생각하면 됩니다.

---

## 2. 기능 하나를 추가할 때의 큰 흐름

어떤 기능이든, 이 프로젝트에서는 거의 항상 이 순서를 따릅니다.

1. **기획**: 무엇을 만들지, 누가 쓰는지, 대략 적기
2. **DB 설계**: 필요한 테이블/컬럼이 있는지 확인, 없으면 추가
3. **컨트롤러**: 어떤 데이터를 가져오고, 어떻게 처리할지 함수 작성
4. **라우터**: URL(경로)을 정하고, 컨트롤러와 연결
5. **뷰(EJS)**: 실제 화면(목록, 상세, 폼 등) 만들기
6. **메뉴/버튼 연결**: 사용자가 눌러 들어올 수 있도록 링크/버튼 추가
7. **테스트 & 수정**: 직접 눌러 보고 에러/UX를 고치기

바이브코딩에서는 이 7단계를 **AI에게 한 단계씩 요청**하면서 진행합니다.

---

## 3. Step 1 – 기능 기획 + 좋은 프롬프트 만들기

먼저, 노트에 짧게라도 써 봅니다.

- 예: "관리자가 FAQ(자주 묻는 질문)를 등록/수정/삭제할 수 있고, 사용자는 /faq 페이지에서 목록만 본다."

그다음 AI에게 이렇게 말합니다.

> "Node.js + Express + MySQL8 + EJS 로 만든 쇼핑몰 프로젝트야.
>  app.js + routes/ + controllers/ + views/ 구조이고, 관리자 기능은 routes/admin/, controllers/admin/ 에 있어.
>  `admin/products`, `admin/banners` 처럼, FAQ(자주 묻는 질문) 기능을 추가하고 싶어.
>  관리자: FAQ CRUD, 사용자: /faq 목록 페이지 정도야.
>  이 기능을 DB → 관리자 → 사용자 → 메뉴 순으로 단계별로 같이 만들어 보자. 1단계로 DB 설계부터 도와줘."

이렇게 하면, AI도 **“아, 이건 장기 프로젝트고, 여러 단계를 같이 해야겠구나”** 를 이해합니다.

---

## 4. Step 2 – DB 설계(테이블/컬럼)

### 4-1. 테이블이 이미 있는지 먼저 확인

- `tables.sql` 을 열어서 비슷한 테이블이 있는지 봅니다.
- 있다면 그 구조를 기준으로 요청하면 됩니다.

### 4-2. 새 테이블이 필요할 때 프롬프트 예시 (FAQ)

> "MySQL 8을 쓰고, tables.sql 에 여러 테이블이 정의되어 있어.
>  여기 FAQ(자주 묻는 질문)를 저장할 faqs 테이블을 추가하고 싶어.
>  id(PK, auto increment), question(varchar 255), answer(text), display_order(int, 정렬용), is_visible(tinyint 1, 기본 1), created_at, updated_at 컬럼이 필요해.
>  이 프로젝트의 tables.sql 스타일에 맞춰 CREATE TABLE 문을 작성해줘."

### 4-3. 컬럼 타입이 고민될 때

`mysql` 문서의 "자주 쓰는 MySQL 8 컬럼 타입" 섹션을 참고해서,

> "display_order 는 정렬용 숫자인데, int 로 해도 될까? tinyint 로 충분할까?"

처럼 물어보면 됩니다.

CREATE TABLE 문을 만들었으면, 실제 DB나 `tables.sql` 에 적용합니다.

---

## 5. Step 3 – 컨트롤러 작성

컨트롤러는 "요청이 들어왔을 때 **무엇을 할지**" 를 결정하는 부분입니다.

### 5-1. 관리자 컨트롤러 프롬프트 예시 (FAQ)

> "controllers/admin 디렉터리에 faqController.js 를 만들고 싶어.
>  이 안에 다음 함수를 넣어줘.
>  - getList: faqs 테이블에서 is_visible 과 상관없이 전체를 display_order, id 기준으로 조회해서 관리자 목록 뷰에 넘김
>  - getForm: 새로 등록할 때 쓸 폼
>  - postCreate: 새 FAQ INSERT
>  - getEdit: 특정 id의 FAQ를 조회해서 수정 폼에 넘김
>  - postUpdate: 특정 id FAQ UPDATE
>  - postDelete: 특정 id FAQ 삭제(또는 is_visible 0)
>  DB는 config/db.js 의 mysql2/promise pool 을 사용해줘.
>  controllers/admin/bannerController.js 패턴을 최대한 따라줘."

이때 **필수 정보 3가지**를 항상 같이 말해 주세요.

1. 파일 경로: `controllers/admin/faqController.js`
2. 사용하는 DB 연결: `config/db.js` 의 `pool`
3. 참고할 기존 패턴: `controllers/admin/bannerController.js`

이 세 가지만 잘 알려줘도, AI가 프로젝트 스타일을 훨씬 잘 맞춰 줍니다.

---

## 6. Step 4 – 라우터(URL) 연결

이제 "어떤 주소로 접속했을 때 어떤 컨트롤러를 부를지" 를 정합니다.

### 6-1. 관리자 라우터 프롬프트 예시

> "routes/admin/faq.js 파일을 새로 만들고 싶어.
>  - GET /       → faqController.getList
>  - GET /new    → faqController.getForm (등록 폼)
>  - POST /      → faqController.postCreate
>  - GET /:id/edit → faqController.getEdit
>  - POST /:id   → faqController.postUpdate
>  - POST /:id/delete → faqController.postDelete
>  형태로 만들어줘. routes/admin/banners.js 구조를 최대한 비슷하게 써줘."

그리고 `routes/admin.js` 에 이 라우터를 붙입니다.

> "routes/admin.js 에 /admin/faq 경로를 연결해줘.
>  products, banners 처럼 require 와 use 코드를 추가해줘.
>  필요한 경우 requireMenuAccess('/admin/faq') 도 넣어줘."

---

## 7. Step 5 – 뷰(EJS) 만들기

컨트롤러와 라우터가 준비되었다면, 이제 화면을 만듭니다.

### 7-1. 관리자 목록 뷰 프롬프트 예시

> "views/admin/faq/list.ejs 파일을 만들고 싶어.
>  layout 은 layouts/admin_layout 을 사용하고,
>  컨트롤러에서 faqs 배열을 넘겨준다고 가정해줘.
>  테이블로 question, display_order, is_visible, 수정/삭제 버튼을 보여줘.
>  상단에는 'FAQ 등록' 버튼이 있어서 /admin/faq/new 로 이동하게 해줘.
>  views/admin/banners/list.ejs 의 구조를 참고해서 작성해줘."

### 7-2. 관리자 폼 뷰 프롬프트 예시

> "views/admin/faq/form.ejs 를 만들어줘.
>  layout 은 layouts/admin_layout.
>  question 입력 필드, answer textarea, display_order 숫자 입력, is_visible 체크박스를 넣어줘.
>  신규 등록과 수정에 모두 쓸 수 있게, noticeController.form.ejs 나 banners/form.ejs 처럼 value 가 있을 때/없을 때를 처리해줘."

사용자용 뷰도 비슷한 방식으로

> "views/user/faq/list.ejs 를 만들어줘. layout 은 layouts/main_layout."

처럼 요청하면 됩니다.

---

## 8. Step 6 – 메뉴/버튼 연결

이제 기능은 만들어졌지만, **들어갈 길** 이 필요합니다.

- 관리자 사이드바 메뉴: `admin_menus` 테이블과 관련 화면
- 사용자 상단/하단 메뉴: `views/user/partials` 의 네비게이션 등

예를 들어 관리자 FAQ 메뉴는 이렇게 요청할 수 있습니다.

> "이 프로젝트 관리자 메뉴는 admin_menus 테이블로 관리하고 있어.
>  FAQ 관리 메뉴를 /admin/faq 로 연결하고 싶어.
>  적당한 INSERT SQL 예시를 만들어줘. name, path, display_order, icon_class 정도만 쓰면 돼."

또는,

> "관리자 메뉴 관리 화면에서 새 메뉴로 FAQ를 추가하고 싶어.
>  어떤 항목에 무엇을 입력해야 /admin/faq 로 연결되는지 설명해줘."

---

## 9. Step 7 – 테스트 & 에러 핸들링 워크플로우

마지막으로, 실사용처럼 테스트합니다.

1. 관리자 로그인
2. 새로 만든 메뉴(예: FAQ)를 클릭
3. 목록 → 등록 → 수정 → 삭제까지 한 번씩 해 보기
4. 사용자 페이지에서도 잘 보이는지 확인

### 9-1. 에러가 났을 때 프롬프트 패턴

> "방금 /admin/faq 페이지에 접속했더니 에러가 났어.
>  에러 메시지는 다음과 같아. (여기에 에러 전문 붙이기)
>  이 프로젝트는 app.js 에서 /admin 라우터를 routes/admin.js 로 연결하고,
>  FAQ 기능은 routes/admin/faq.js, controllers/admin/faqController.js 에 만들었어.
>  에러 원인과 수정 방법을 알려줘."

에러 메시지, 요청한 URL, 관련 파일 경로 **세 가지를 항상 같이** 알려 주면 AI가 원인을 찾기 쉽습니다.

## 10. 하루 공부·개발 루틴 예시

바이브코딩으로 학습 겸 개발을 할 때, 이런 루틴을 추천합니다.

1. 오늘 만들 기능을 한 줄로 정리
   - 예: "오늘은 FAQ 관리자 + 사용자 목록까지만 한다."
2. 이 워크플로우에 맞춰 Step 1 ~ Step 7 을 큰 틀로 잡기
3. 각 Step 에서 `vibe_coding` 문서의 좋은 프롬프트 예시를 참고해 AI에게 요청
4. 기능 하나 완성 후, 같은 패턴으로 **다른 기능(공지, 이벤트, 태그 등)** 에 반복 적용

이렇게 몇 번만 반복해 보면,

- "기능 아이디어가 떠오른다 → DB → 컨트롤러 → 라우터 → 뷰 → 메뉴 → 테스트" 의 흐름이 자연스럽게 몸에 익습니다.

## 12. 단계별 상세 가이드 및 시간 배분

### Step 1: 기능 기획 (5-10분) 📝

**목표:** 무엇을 만들지 명확히 하기

### Step 3: 컨트롤러 작성 (15-20분) 🎮

**목표:** 비즈니스 로직 구현하기

**관리자 CRUD 함수 체크리스트:**
- [ ] `getList`: 전체 데이터 조회 (페이징, 필터링)
- [ ] `getForm`: 등록 폼 렌더링
- [ ] `postCreate`: 새 데이터 INSERT + 검증
- [ ] `getEdit`: 수정 폼 렌더링 (기존 데이터 로드)
- [ ] `postUpdate`: 데이터 UPDATE + 검증
- [ ] `postDelete`: Soft delete (is_deleted = 1)

**코드 검토 포인트:**

```javascript
✅ 좋은 컨트롤러 코드 예시:

const pool = require('../../config/db');  // ← 경로 확인!

// 1. async/await 사용
exports.getList = async (req, res, next) => {
  try {
    // 2. 파라미터 검증
    const page = parseInt(req.query.page) || 1;
    const limit = 20;
    const offset = (page - 1) * limit;

    // 3. SQL injection 방지 (? 플레이스홀더)
    const [rows] = await pool.query(
      'SELECT * FROM reviews WHERE is_deleted = 0 LIMIT ? OFFSET ?',
      [limit, offset]
    );

    // 4. 뷰 경로 확인
    res.render('admin/reviews/list', {
      reviews: rows,
      currentPage: page
    });
  } catch (err) {
    // 5. 에러 처리
    next(err);
  }
};
```

**시간 단축 팁:**
- 기존 컨트롤러 코드 일부를 AI에게 보여주고 "이 패턴으로" 요청
- 복잡한 쿼리는 단계별로 요청 (먼저 단순 조회 → 조인 추가)

## 15. 자주 막히는 구간과 해결책

### 병목 1: DB 설계 단계에서 컬럼 타입 결정
**증상:** "이 필드는 VARCHAR? TEXT?"
**해결:**
- 짧은 텍스트(100자 이하) → VARCHAR(n)
- 긴 텍스트(본문) → TEXT
- 숫자 → INT (또는 BIGINT, TINYINT)
- 금액 → DECIMAL(10,2)
- 날짜/시간 → TIMESTAMP

### 병목 2: 컨트롤러 작성 시 async/await 이해 부족
**증상:** "await을 어디에 붙여야 하나요?"
**해결:**
```javascript
// ✅ async 함수 안에서만 await 사용 가능
exports.getList = async (req, res, next) => {
  //                ↑ async 선언

  const [rows] = await pool.query(...);
  //             ↑ DB 조회는 await

  res.render(...);  // 렌더링은 await 불필요
};
```

### 병목 3: 뷰에서 변수명 불일치
**증상:** "undefined" 또는 빈 화면
**해결:**
```javascript
// 컨트롤러
res.render('admin/faq/list', { faqs: rows });
//                              ↑ 이 이름과

// EJS
<% faqs.forEach(faq => { %>
//  ↑ 이 이름이 일치해야 함
```

### 병목 4: 라우터 순서 문제
**증상:** 특정 URL만 작동 안 함
**해결:**
```javascript
// ❌ 잘못된 순서
router.get('/:id', ...);      // 이게 먼저면
router.get('/new', ...);      // '/new'를 ':id'로 인식

// ✅ 올바른 순서
router.get('/new', ...);      // 구체적인 경로 먼저
router.get('/:id', ...);      // 파라미터 경로 나중
```

## 16. 마지막 체크리스트

기능 하나를 마무리할 때, 아래를 빠르게 확인해 보세요.

### 코드 품질 체크
- [ ] 필요한 데이터를 저장할 **테이블/컬럼** 이 있다
- [ ] **컨트롤러** 가 데이터를 잘 가져오고/저장한다
- [ ] **라우터(URL)** 가 컨트롤러를 잘 가리킨다
- [ ] **뷰(EJS)** 가 데이터를 예상대로 보여준다
- [ ] 관리자/사용자 **메뉴·버튼** 에서 새 기능으로 들어갈 수 있다
- [ ] 에러 처리가 되어 있다 (try-catch, 404 등)

### 보안 체크
- [ ] SQL injection 방지 (? 플레이스홀더 사용)
- [ ] XSS 방지 (EJS <%= %> 사용)
- [ ] CSRF 토큰 (폼 제출 시)
- [ ] 파일 업로드 검증 (확장자, 크기)
- [ ] 권한 체크 (관리자 전용 기능)

### 사용자 경험 체크
- [ ] 로딩 상태 표시 (필요시)
- [ ] 에러 메시지가 명확함
- [ ] 성공 메시지 표시
- [ ] 모바일에서도 잘 보임 (반응형)
- [ ] 접근성 (키보드 네비게이션)

이 체크리스트를 매번 돌리면, 바이브코딩으로도 충분히 **실서비스에 가까운 쇼핑몰 기능**을 만들어 갈 수 있습니다.
