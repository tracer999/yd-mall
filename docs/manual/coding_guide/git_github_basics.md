# Git/GitHub 기초와 소스 관리

이 문서는 Git/GitHub를 처음 접하는 분을 위해,

- Git, GitHub가 무엇인지
- 왜 소스 관리를 해야 하는지
- commit, push, pull 등 핵심 용어를 언제 쓰는지
- 실수했을 때 소스를 복원하는 기본 원리
- CI/CD의 개념과 최근 활용 흐름

을 간단히 정리한 입문 자료입니다.

---

## 1. Git과 GitHub는 무엇인가?

### Git
**Git**은 코드 변경 이력을 저장하는 버전 관리 도구입니다.

- 누가, 언제, 무엇을 바꿨는지 기록
- 이전 시점으로 되돌아가기 가능
- 기능별로 작업을 분리(브랜치)

### GitHub
**GitHub**는 Git 저장소를 인터넷에 올려 협업하는 서비스입니다.

- 팀원과 코드 공유
- Pull Request 기반 코드 리뷰
- 이슈/릴리스 관리
- CI/CD 파이프라인 연동

짧게 정리하면:

- Git: **버전관리 엔진**
- GitHub: Git을 팀으로 쓰기 위한 **협업 플랫폼**

---

## 2. 왜 쓰는가? (소스 관리의 핵심 이유)

1. 실수 복구
코드를 잘못 고쳐도 이전 커밋으로 복원할 수 있습니다.

2. 협업 안정성
여러 명이 동시에 작업해도 변경 이력을 기준으로 충돌을 관리할 수 있습니다.

3. 변경 추적
문제가 생겼을 때 어느 변경에서 시작됐는지 빠르게 추적할 수 있습니다.

4. 배포 신뢰성
"운영에 나간 코드"와 "개발 중 코드"를 구분하기 쉬워집니다.

---

## 3. 자주 쓰는 용어와 언제 사용하는가

| 용어 | 뜻 | 언제 사용하나 |
|------|----|----------------|
| `repository(저장소)` | 코드와 이력 전체가 담긴 공간 | 프로젝트 시작할 때 |
| `clone` | 원격 저장소를 내 PC로 복사 | 처음 프로젝트를 받을 때 |
| `branch` | 독립 작업 라인 | 기능/버그수정 작업을 분리할 때 |
| `commit` | 변경 스냅샷 저장 | 의미 있는 단위 작업이 끝났을 때 |
| `push` | 로컬 커밋을 원격(GitHub)으로 업로드 | 협업 공유/백업/PR 생성 전 |
| `pull` | 원격 최신 변경을 로컬로 반영 | 작업 시작 전, 충돌 방지 위해 |
| `merge` | 두 브랜치 변경을 합침 | 기능 완료 후 메인 반영 시 |
| `conflict` | 같은 부분을 다르게 수정해 충돌 | merge/pull 시 수동 정리 필요할 때 |
| `pull request(PR)` | 변경 반영 요청 및 리뷰 단위 | 팀 리뷰 후 병합할 때 |

## 4. 소스 복원 관점에서 보는 기본 관리법

### 4-1. 복원 단위는 "커밋"
소스 복원은 보통 "파일"보다 "커밋 단위"로 생각합니다.

- 커밋을 자주, 작게 나누면 복원이 쉬움
- "기능 1개 = 1~3 커밋" 정도로 유지하면 추적이 편함

### 4-2. 복원 시나리오 (개념)

1. 방금 수정한 파일만 되돌리고 싶다
- 아직 커밋 전이면 작업 파일 상태를 이전으로 되돌립니다.

2. 최근 커밋이 문제다
- 해당 커밋을 취소하는 새 커밋을 만들어 안전하게 복구합니다.
- 공유 브랜치에서는 "이력 재작성"보다 "취소 커밋" 방식이 보통 안전합니다.

3. 특정 시점으로 임시 점검하고 싶다
- 예전 커밋으로 이동해 동작을 확인한 뒤, 다시 현재 브랜치로 돌아옵니다.

핵심 원칙:
- 메인 브랜치에서 큰 작업을 직접 하지 않기
- 기능 브랜치에서 작업하고 PR로 병합하기
- 커밋 메시지는 "무엇을 왜 바꿨는지"가 보이게 쓰기

---

## 5. 권장 작업 흐름 (입문형)

1. `pull`로 최신 코드 동기화
2. 기능 브랜치 생성
3. 코드 수정 + 로컬 테스트
4. 작은 단위로 `commit`
5. `push` 후 PR 생성
6. 리뷰 반영
7. 메인 브랜치로 병합

이 흐름만 지켜도 "잃어버린 코드"와 "덮어쓰기 사고"가 크게 줄어듭니다.

---

## 6. CI/CD란 무엇인가? (개념만)

### CI (Continuous Integration)
개발자가 코드를 자주 합칠 때, 자동으로 빌드/테스트를 돌려 문제를 빨리 발견하는 방식입니다.

### CD (Continuous Delivery/Deployment)
검증된 코드를 배포 가능한 상태로 계속 유지하거나, 자동으로 배포까지 이어지게 하는 방식입니다.

### Git과 CI/CD의 최근 활용 흐름
요즘은 대부분 아래 패턴으로 운영합니다.

- GitHub에 `push` 또는 PR 생성
- 파이프라인이 자동 실행(테스트/검사)
- 통과하면 병합 혹은 배포 단계 진행

즉, Git 이력은 단순 기록이 아니라
**"코드 품질 검사와 배포 자동화의 트리거"** 역할까지 담당합니다.

> 이 문서에서는 CI/CD 구현 방법(툴 설정, YAML 작성, 배포 스크립트)은 다루지 않습니다.
> 여기서는 "왜 쓰는지"와 "어떤 흐름인지"의 개념 이해에 집중합니다.

---

## 한 줄 요약

- Git은 코드 이력 관리 도구, GitHub는 협업 플랫폼입니다.
- commit/push/pull/branch/PR 흐름만 익혀도 실무 소스 관리의 80%를 커버할 수 있습니다.
- CI/CD는 Git 변경을 기준으로 테스트/배포를 자동화하는 운영 방식입니다.
