# 실행 가이드

이 문서는 `dev-mall` 프로젝트를 개발 환경과 상용(Production) 환경에서 실행하는 방법과 각 명령어의 차이점을 설명합니다.

## 0. 실행 전 필수 조건

### 0.1 `ENCRYPTION_KEY`

`.env` 계열의 `DB_PASS` / `REDIS_PASSWORD` 는 AES-256-GCM 으로 암호화되어 `ENC:` 접두어로 저장됩니다. `config/env.js` 는 `ENCRYPTION_KEY` 가 없으면 복호화에 실패해 **즉시 종료(`process.exit(1)`)** 합니다.

- 키는 서버의 `/etc/environment` 에 있습니다.
- 비로그인 셸(CI/CD 의 SSH 액션 등)에서는 자동 로드되지 않으므로, 직접 실행할 때는 먼저 source 하세요.

```bash
set -a; . /etc/environment; set +a
```

`./dev-mall.sh` 는 이 과정을 스크립트 안에서 자동으로 처리합니다.

### 0.2 포트와 환경 파일

- **포트는 개발·상용 모두 3006** 입니다. (`.env` 의 `PORT=3000` 은 환경별 파일이 항상 덮어씀)
- `config/env.js` 가 `.env`(공통 기본값)를 먼저 읽고, `NODE_ENV` 에 따라 `.env.development` 또는 `.env.production` 을 override 로 덮어씁니다.
- 그 외 대부분의 설정(`SESSION_SECRET`, OAuth, SMTP, 결제 키 등)은 **DB `system_settings` 테이블**에서 로드되어 `process.env` 를 덮어씁니다.

## 1. 개발 환경 실행 (Development)

개발 중에는 **코드 수정 시 서버 자동 재시작**과 **CSS 실시간 빌드**가 핵심입니다.

### 1) 통합 개발 모드 (권장)

**명령어**: `npm run dev:all`

- **실행 내용**: 백엔드(`nodemon`)와 Tailwind 빌드(`tailwindcss --watch`)를 `concurrently` 로 동시에 실행합니다. (`NODE_ENV=development`)
- **특징**:
  - `app.js` 등 서버 코드 수정 시: 서버가 자동으로 재시작됩니다.
  - `views/*.ejs` 등 UI 코드 수정 시: CSS가 자동으로 다시 빌드되어 스타일이 반영됩니다.
- **사용 시점**: UI와 서버 로직을 함께 개발할 때 가장 편리합니다.

### 2) 백엔드 전용 모드

**명령어**: `npm run dev`

- **실행 내용**: `nodemon` 만 실행하여 서버 코드 변경만 감지합니다.
- **특징**: CSS 빌더가 돌지 않으므로 시스템 리소스를 적게 사용합니다.
- **사용 시점**: API 개발, DB 작업 등 UI 스타일 변경이 필요 없는 작업.

### 3) CSS 만 빌드

```bash
npm run build:css   # 1회 빌드
npm run watch:css   # 감시 모드
```

Tailwind 소스는 `public/css/input.css`, 결과물은 `public/css/style.css` 입니다.

## 2. 상용 환경 실행 (Production)

> ⚠️ **`git push origin main` = 즉시 운영 배포입니다.** `.github/workflows/deploy.yml` (GitHub Actions)이 운영 서버(`/data/yd-mall`)에서 `git reset --hard origin/main` 후 `./dev-mall.sh build && ./dev-mall.sh start` 를 실행합니다. 아래 명령들은 서버에서 직접 조작할 때 쓰는 것입니다.

### 1) `dev-mall.sh` (운영 서버 표준)

`ENCRYPTION_KEY` 로드 + nvm 으로 Node 22 선택 + PM2 기동까지 한 번에 처리합니다. **운영 서버에서는 이 스크립트를 사용하세요.**

```bash
cd /data/yd-mall
./dev-mall.sh build          # npm install + Tailwind CSS 빌드
./dev-mall.sh start          # 기동/갱신 (NODE_ENV=production) [기본]
./dev-mall.sh start dev      # 기동/갱신 (NODE_ENV=development)
./dev-mall.sh restart
./dev-mall.sh status
./dev-mall.sh logs           # pm2 logs dev-mall --lines 50
./dev-mall.sh stop
./dev-mall.sh delete         # PM2 목록에서 제거
```

내부적으로 `pm2 startOrRestart ecosystem.config.cjs --env <production|development> --update-env` 를 호출하고 `pm2 save` 로 프로세스 목록을 저장합니다.

### 2) PM2 직접 실행

```bash
npm run pm2:start          # pm2 start ecosystem.config.cjs           (NODE_ENV=production)
npm run pm2:start:dev      # pm2 start ecosystem.config.cjs --env development
npm run pm2:restart        # 상용 재시작 (--update-env)
npm run pm2:restart:dev    # 개발 재시작 (--update-env)
```

**`ecosystem.config.cjs` 실제 설정** — 앱명 `dev-mall`, **`exec_mode: 'fork'`, `instances: 1`** 입니다. 클러스터가 아닙니다.

- **단일 프로세스 fork 모드**: 세션이 프로세스 메모리에 있어도 유실되지 않으므로, Redis 없이 MemoryStore 폴백으로도 로그인이 유지됩니다.
- **자동 재시작**: 프로세스가 에러로 죽으면 PM2 가 다시 살립니다.
- **로그**: `logs/pm2-out.log`, `logs/pm2-error.log` 로 분리 기록. HTTP 접속 로그는 앱이 직접 `logs/access.log` 에 씁니다. (→ [`logs.md`](./logs.md))

> **인스턴스를 2개 이상(cluster)으로 늘리려면 Redis 세션 저장소가 필수입니다.** `.env` 에 `REDIS_HOST` 를 설정하면 `connect-redis` 를 사용하고, 없으면 MemoryStore 로 폴백하며 `app.js` 가 경고를 출력합니다.

### 3) 단순 Node 실행 (Docker/테스트용)

PM2 없이 프로세스 하나만 띄우는 방식입니다.

```bash
npm run start:prod          # cross-env NODE_ENV=production node app.js
npm start                   # node app.js  (NODE_ENV 미지정 → 환경변수로 직접 지정 필요)
NODE_ENV=production node app.js
```

- 프로세스 관리 기능(자동 재시작 등)이 **없습니다**.
- Docker/K8s 처럼 외부에서 프로세스를 관리해주는 환경, 또는 로컬에서 상용 설정을 잠깐 확인할 때 적합합니다.

## 3. 실행 방법 비교표

| 명령어 | NODE_ENV | 프로세스 관리 | CSS 감시 | 주요 용도 |
| --- | :---: | :---: | :---: | --- |
| **`npm run dev:all`** | development | Nodemon (자동) | O | **로컬 개발 (메인)** |
| `npm run dev` | development | Nodemon (자동) | X | 백엔드 개발 |
| **`./dev-mall.sh start`** | production | **PM2 (fork, 1개)** | X | **운영 서버 (표준)** |
| `npm run pm2:start` | production | PM2 (fork, 1개) | X | PM2 직접 기동 |
| `npm run start:prod` | production | Node (단일) | X | 컨테이너 배포 / 테스트 |

## 4. 기타 스크립트

```bash
npm run init:db     # DB 스키마 + 시드 초기화 (scripts/init_db.js)
```

> ⚠️ **개발 DB 와 운영 DB 가 같습니다**(`dev_mall` @ ydata.co.kr). 쓰기 작업은 신중히 하세요.

일회성 검증 스크립트를 돌릴 때는 `/etc/environment` 를 source 하고, 스크립트 안에서 `await require('./scripts/_bootstrap')()` 를 먼저 호출해야 합니다. 부트스트랩 없이 실행하면 `isShopifySyncEnabled()` 가 fail-open 으로 `true` 가 되어 **실제 Shopify API 를 호출**합니다.

```bash
set -a; . /etc/environment; set +a; node _tmp.js
```

## 5. 문제 해결

### Q: 앱이 기동하자마자 종료됩니다.

**A**: `ENCRYPTION_KEY` 가 없을 가능성이 큽니다. `config/env.js` 가 `ENC:` 값을 복호화하지 못하면 즉시 `process.exit(1)` 합니다. `/etc/environment` 를 source 하거나 `./dev-mall.sh` 를 사용하세요.

### Q: `.env` 를 고쳤는데 반영되지 않습니다.

**A**: 대부분의 설정은 DB `system_settings` 테이블이 `.env` 값을 **덮어씁니다**(`config/systemSettings.js`). 관리자 화면 `/admin/settings` · `/admin/sys-settings` 에서 값을 확인하세요. 반대로 서버·DB·Redis 접속 정보(`PORT`, `DB_*`, `REDIS_*`)는 `.env` 계열에만 있습니다.

### Q: 로그인 세션이 자꾸 풀립니다.

**A**: 현재 구성(fork, `instances: 1`)에서는 프로세스가 하나뿐이라 MemoryStore 로도 세션이 유지됩니다. 세션이 풀린다면 프로세스가 재시작되고 있다는 뜻이므로 `pm2 logs dev-mall` 로 크래시 원인을 확인하세요. 근본 대책으로는 `.env` 에 `REDIS_HOST` 를 설정해 Redis 세션 저장소를 쓰는 것이 좋습니다.

### Q: 인스턴스를 늘려 부하를 분산하고 싶습니다.

**A**: `ecosystem.config.cjs` 의 `exec_mode` 를 `cluster` 로, `instances` 를 늘리기 전에 **Redis 세션이 먼저 필요합니다.** 그렇지 않으면 요청이 다른 프로세스로 갈 때마다 로그인이 풀립니다.
